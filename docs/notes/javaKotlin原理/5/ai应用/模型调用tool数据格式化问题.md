这段 SpringAI 教程文字，本质上在解决一个**“流式输出 + Function Calling（Tool）” 的前后端协议问题**：

> 大模型一边在 **Flux 里流式吐文字**（token/片段），一边可能触发 **tool 调用**拿到结构化结果；
> 但前端要展示“卡片”需要 **JSON 结构化数据**，而不是模型的自然语言。
> 所以要把 **tool 的结构化结果**在流的最后**追加**给前端（在 STOP 之前），让前端渲染卡片。

------

## 1）它到底在解决什么“需求痛点”？

### 你的前端要的不是“文字”，而是“卡片数据”

比如课程卡片要 `id/name/price/...` 这种字段。
如果你只把模型的自然语言流给前端：

- 前端只能显示“字”
- 无法保证能解析出稳定字段（模型可能换措辞、漏字段、格式乱）

所以教程提出：**前端与后端约定一套事件协议**（eventType + eventData），例如：

- `eventType=1001`：流式文字片段（DATA）
- `eventType=1002`：结束标识（STOP）
- `eventType=1003`：卡片参数 JSON（PARAM）

前端只要看到 `1003`，就拿 `eventData` 渲染卡片。

------

## 2）为什么要“在 Flux 最后追加”？

因为 **tool 的执行时机不确定**：

- 模型可能在中途触发 tool
- tool 执行需要时间
- 而 Flux 正在持续往外吐 token

如果你想在“正确的时间点”把结构化数据插进去，有两种策略：

### 策略 A：实时插入（复杂）

tool 返回的瞬间就往 Flux 里插入一个 eventType=1003 的事件
→ 这需要把 tool 的结果“变成流事件”并和 token 流合并，处理顺序/并发很麻烦。

### 策略 B：**最后追加（教程方案）**

先让模型正常流式输出文字（1001）
等流结束时（准备发 STOP 1002 之前），**检查是否发生过 tool 调用**，如果有就把 tool 的结果作为 1003 插进去，再发 STOP。

这样前端体验是：

- 一边看文字生成
- 最后出现一张课程卡片（或多个卡片）

这就是教程说的“追加到输出流的结束标签之前”。

------

## 3）“Tool 执行结果已经给了大模型，我们在 Flux 输出时如何获取到？” 这句在问什么？

这是关键难点：SpringAI 默认流程是：

- 模型触发 tool call
- SpringAI 执行 tool
- tool 的返回结果 **作为工具消息**回喂给模型继续生成

**但**：这一套发生在框架内部，你的 Controller 那条 Flux 链路里拿到的通常只是模型最终输出的文本片段，并不会自动把 tool 的原始结构化结果“吐给你”。

所以教程在问：

> tool 的结构化返回，怎么让我们后端也拿到，然后发给前端？

------

## 4）为什么教程要引入“全局容器”？

因为它想用一种简单粗暴的方式实现：

- tool 执行时：把结果塞进一个容器（Map / ConcurrentHashMap）
- Flux 流结束时：去容器里取结果，取到了就追加 1003 事件

这等价于：**用“旁路存储”把 tool 结果从框架内部搬到你自己的 Controller 流里**。

------

## 5）为什么不能用 sessionId 关联？为什么要 requestId？

这是它在解决**并发正确性**问题。

如果你用 `sessionId -> toolResult` 存到容器：

- 同一个 sessionId 可能同时发起两次请求（比如用户连点两次、前端重连、并发提问）
- 两次 tool 调用结果会互相覆盖/串线
- 最终你在 Flux 结束时取到的可能是“另一条请求”的 tool 结果

所以教程说 sessionId 不行，要用 **requestId**（每次请求唯一）：

- 每次调用模型前生成 requestId
- requestId 随本次调用传递到 tool 执行上下文中
- tool 执行完：结果写入 `container[requestId]`
- Flux 收尾：读取 `container[requestId]`，只会拿到本次请求的结果

这就是它强调的“同 session 并发”问题。

------

## 6）它真正想实现的完整链路（用一句话串起来）

**把“工具调用产出的结构化数据”从模型内部流程中取出来，按前后端约定的事件协议，插入到流式输出的尾部，从而让前端既能看到实时输出文字，又能在最后稳定渲染卡片。**

------

## 7）你应该怎么在工程里实现得更干净（比“全局容器”更稳）

教程的“全局容器”是能跑，但工程上有两个隐患：

1. 内存泄漏：requestId 没清理就会堆积
2. 分布式：多实例部署时，全局 Map 不共享，取不到 tool 结果

更工程化的方式通常是：

- requestId -> toolResult 存 Redis / Caffeine(带 TTL) / Reactor Context
- 或者用 `Sinks.One<ToolResult>` / `MonoProcessor` 做“本次请求的信号”，tool 完成时 `tryEmitValue`

但你现在理解教程意图就够了：它是在讲**“如何让 tool 结果也进入你自己的 Flux 输出协议”**。
