把“一致性哈希”先当成一个**搬家问题**来看，就秒懂了。

---

### 1. 原始问题（普通哈希）
有 3 个仓库 0、1、2，货物编号 hash 后取模 3 决定放哪里：

| 货号 | hash | %3   | 仓库 |
| ---- | ---- | ---- | ---- |
| A    | 103  | 1    | 1号  |
| B    | 205  | 2    | 2号  |
| C    | 300  | 0    | 0号  |

👉 现在要**再建一个仓库 3**，规则变成 %4  
结果：A→3，B→1，C→0……**几乎全搬家！**  
缓存/数据库瞬间全失效，这就是**取模哈希的致命缺点**。

---

### 2. 一致性哈希的思路（搬家最少）
把仓库、货物都**扔到一个首尾相接的“环”**上（0→1→2→3→…→最大值→回到0）

步骤：  
1. 用同一个 hash 函数给  
   - 每个**仓库**算个角度（蓝色点）  
   - 每个**货物**也算个角度（红色点）  
2. 规定：货物**顺时针看第一个≥自己的仓库**就是它的家  
   👉 只与“角度”有关，与“仓库总数”无关！

| 操作     | 影响范围             | 搬家量 |
| -------- | -------------------- | ------ |
| 新增仓库 | 只切断环上一小段     | ≈1/N   |
| 删除仓库 | 同样只让一小段换邻居 | ≈1/N   |

---

### 3. 虚拟节点（防止扎堆）
如果仓库很少，角度会扎堆→负载不均。  
给每个真实仓库起**一堆小号**（虚拟节点）散在环上，**负载立刻均匀**。

---

### 4. 再回头看“一致性哈希是干啥的”
一句话：  
**“节点数量变化时，让数据搬家最少，又能保持均匀分布。”**

---

### 5. 生活比喻
把环想成**钟面**：
- 12 点、4 点、8 点各站一个快递员（仓库）  
- 包裹随机扔到钟面某刻度  
- 包裹**顺时针交给第一个遇到的快递员**

现在**再加一个快递员到 6 点**：
- 只有原来 4→8 这段包裹改交给 6 点  
- 12 点、8 点老包裹**原地不动**  
→ 搬家最少，这就是一致性哈希。
