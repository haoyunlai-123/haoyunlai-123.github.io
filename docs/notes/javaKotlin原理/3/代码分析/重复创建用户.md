我们来从数据库、事务、并发三个层面彻底分析一下。👇

---

## 🧩 一、问题点在哪里？

看这段代码：

```java
User user = query().eq("phone", phone).one();
if (user == null) {
    user = creatUserWithPhone(phone);
}
```

这里逻辑上是：

> “如果手机号不存在，就创建一个新用户。”

但是在**高并发**情况下（比如同一个手机号短时间内发起多次请求），
两个线程可能几乎同时执行到：

```
User user = query().eq("phone", phone).one(); // 都查不到
```

于是：

```
Thread A -> createUserWithPhone(phone)
Thread B -> createUserWithPhone(phone)
```

👉 两个线程都插入了同一个手机号的用户。
最终数据库中可能出现 **重复手机号记录**。

---

## ⚙️ 二、为什么会发生？（竞态条件）

两个线程几乎同时执行这三步：

| 时间线 | 线程A         | 线程B       |
| --- | ----------- | --------- |
| t1  | 查库（没查到）     | 查库（没查到）   |
| t2  | 执行 insert   | 执行 insert |
| t3  | 都 commit 成功 |           |

因为在 SQL 层面没有限制，它们互相都不知道对方已经准备插入相同的数据。

---

## 💣 三、可能造成的后果

* 同一手机号的用户表出现重复；
* 登录态混乱；
* Session 保存的 user 信息不唯一；
* 后续逻辑（比如手机号唯一约束、积分表、用户资料）全乱。

---

## ✅ 四、解决方案（推荐三种）

### ✅ 方案一：在数据库层面加唯一约束（**最推荐**）

从根源防止重复插入。

```sql
ALTER TABLE tb_user ADD UNIQUE KEY uk_phone (phone);
```

然后在 Java 层捕获异常：

```java
try {
    user = creatUserWithPhone(phone);
} catch (DuplicateKeyException e) {
    user = query().eq("phone", phone).one(); // 再查一次
}
```

💡 这样，即使两个线程同时插入，第二个会抛唯一键冲突异常，
再重新查一次即可拿到正确的用户。

---

### ✅ 方案二：加分布式锁（适合高并发登录）

用 Redis 分布式锁按手机号串行化创建：

```java
String lockKey = "lock:phone:" + phone;
boolean success = redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 5, TimeUnit.SECONDS);
if (!success) {
    return Result.fail("操作太频繁，请稍后再试");
}

try {
    User user = query().eq("phone", phone).one();
    if (user == null) {
        user = creatUserWithPhone(phone);
    }
    session.setAttribute("user", user);
} finally {
    redisTemplate.delete(lockKey);
}
```

> ⚙️ 优点：安全可靠、可扩展；
> 缺点：略微增加延迟。

---

### ✅ 方案三：使用数据库悲观锁或乐观锁（不太推荐登录场景）

例如：

```sql
SELECT * FROM tb_user WHERE phone = ? FOR UPDATE;
```

这样其他事务必须等待，但不适合登录这种高频、轻量操作。

---

## 🔐 五、最终推荐方案总结

| 方案             | 安全性 | 性能  | 实现复杂度 | 推荐度   |
| -------------- | --- | --- | ----- | ----- |
| 数据库唯一约束 + 捕获异常 | ✅✅✅ | ✅✅✅ | ⭐     | ⭐⭐⭐⭐⭐ |
| Redis 分布式锁     | ✅✅✅ | ✅✅  | ⭐⭐    | ⭐⭐⭐⭐  |
| 悲观锁/事务锁        | ✅✅  | ❌   | ⭐⭐⭐   | ⭐⭐    |

✅ **生产环境最佳实践**：

> * 在 `tb_user.phone` 字段上加唯一索引；
> * 插入时 try-catch 住重复键异常；
> * 不需要人工加锁。

public Result login(LoginFormDTO loginForm, HttpSession session) {
    String phone = loginForm.getPhone();
    if (RegexUtils.isPhoneInvalid(phone)) {

    }

    User user = query().eq("phone", phone).one();

        }
    }

    session.setAttribute("user", user);
    return Result.ok();
}
