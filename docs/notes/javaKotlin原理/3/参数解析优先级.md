如果我用post方式发送请求，但在资源路径中又带了参数，请求体里也有相同名称值不同的参数，后端requestMapping方法里会收到哪个参数？

非常棒的问题 ✅，
 这其实是很多 Web 框架底层的“参数解析优先级”机制问题。
 我们来精确地分析——当你发送一个 **POST 请求**，
 既在 URL 中带参数，又在请求体中带相同参数名时，Spring MVC（以及标准 Servlet）到底取哪个。

------

# 🧩 一、先举个例子

假设你发送请求：

```bash
POST /user?id=1 HTTP/1.1
Content-Type: application/x-www-form-urlencoded

id=2&name=小鸭哥
```

你会发现：

- URL 参数中：`id=1`
- 请求体参数中：`id=2`
- 同时出现！

------

# 🧠 二、底层：Servlet 规范的规则

Servlet 规范（`HttpServletRequest`）定义了一个方法：

```java
String getParameter(String name);
```

它的官方文档说明：

> Returns the value of a request parameter as a String, or null if the parameter does not exist.
>  *Request parameters are extra information sent with the request.*
>  *For HTTP POST requests, parameters can come from the query string or the posted form data.*

即：

> **无论是 GET 还是 POST，请求参数可以来自 URL 查询字符串，也可以来自请求体表单数据。**

### ✅ 关键结论：

- **Servlet 会自动把 URL 参数和请求体表单参数合并到一个 Map 中。**
- 如果键重复，**请求体中的值会覆盖 URL 中的值**（在多数实现中）。

------

# 🧩 三、在 Spring MVC 中的行为

Spring MVC 的 `@RequestParam`、`@RequestBody`、`@ModelAttribute`
 最终底层都依赖 `HttpServletRequest.getParameter()` 或 `ServletRequestParameterPropertyValues` 来获取数据。

因此，**默认行为与 Servlet 一致**：

| 情况                                 | Spring Controller 中取到的值 |
| ------------------------------------ | ---------------------------- |
| URL 参数：`?id=1`，body 参数：`id=2` | ✅ `id=2`（表单覆盖查询参数） |
| URL 参数：`?id=1`，body 无参数       | ✅ `id=1`                     |
| URL 参数无，body 参数：`id=2`        | ✅ `id=2`                     |

------

# 🔍 四、具体示例

```java
@PostMapping("/user")
@ResponseBody
public String test(@RequestParam("id") String id) {
    return "id=" + id;
}
```

如果你请求：

```bash
POST /user?id=1
Content-Type: application/x-www-form-urlencoded

id=2
```

返回：

```
id=2
```

👉 Spring 解析过程如下：

1. `DispatcherServlet` 收到请求；
2. 调用 `RequestParamMethodArgumentResolver`；
3. 解析参数时调用 `request.getParameter("id")`；
4. Servlet 层合并参数 Map，body 的表单参数覆盖 URL 的。

------

# ⚠️ 五、注意例外情况

### 1️⃣ Content-Type 不同

如果你发送：

```bash
Content-Type: application/json
```

请求体是：

```json
{ "id": 2 }
```

则：

- URL 参数仍然能通过 `@RequestParam("id")` 拿到；
- 请求体参数（JSON）不会被自动合并；
- JSON 只会在 `@RequestBody` 中出现。

**例如：**

```java
@PostMapping("/user")
public String test(@RequestParam String id, @RequestBody Map<String, Object> body) {
    System.out.println(id);           // 从URL来
    System.out.println(body.get("id")); // 从body来
}
```

此时：

- `/user?id=1` + body `{"id":2}`
   → `@RequestParam id=1`
   → `@RequestBody.id=2`

------

### 2️⃣ GET 请求（或非表单 POST）

GET 请求只有 URL 参数；
 只有在 **`application/x-www-form-urlencoded` 的 POST** 才会合并 body 参数。

------

# 🧩 六、总结对比表

| Content-Type                               | URL 参数 | Body 参数 | 最终 getParameter() | 备注                     |
| ------------------------------------------ | -------- | --------- | ------------------- | ------------------------ |
| GET 请求                                   | ✅ 有     | ❌ 无      | URL 参数            | —                        |
| POST + `application/x-www-form-urlencoded` | ✅ 有     | ✅ 有      | ⚡ body 覆盖 URL     | Servlet 自动合并         |
| POST + `multipart/form-data`               | ✅ 有     | ✅ 有      | ⚡ body 覆盖 URL     | 文件上传场景             |
| POST + `application/json`                  | ✅ 有     | ✅ 有      | ❌ 仅 URL 参数       | JSON 需用 `@RequestBody` |

------

✅ **结论一句话：**

> 在 Spring（基于 Servlet 规范）的表单请求中，
>  如果 URL 和请求体都带有同名参数，**请求体参数优先，会覆盖 URL 参数。**
>
> 但若是 `application/json` 类型，则 URL 与 Body 参数相互独立，
>  分别由 `@RequestParam` 与 `@RequestBody` 接收。
