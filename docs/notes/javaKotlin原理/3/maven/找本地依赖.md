我在当前maven项目pom.xml中依赖了另一个本地项目，首次执行时它是如何找到另一个项目位置的？而且他把这个项目放哪里了，我在本地仓库没有找到？

**已解决：确实需要先install另外一个项目到本地仓库，执行时再从本地仓库中找**

非常好的问题，这涉及到 **Maven 的依赖解析机制** 与 **多模块/本地项目引用** 的区别。我们分两种情况来分析：

------

## 🧩 一、你说的“依赖了另一个本地项目”有两种可能：

### ✅ 情况 1：两个项目都在同一个父项目（multi-module 项目）

例如：

```
parent/
 ├── pom.xml  （packaging = pom）
 ├── project-a/
 │   └── pom.xml
 └── project-b/
     └── pom.xml
```

在 `parent/pom.xml` 里：

```xml
<modules>
    <module>project-a</module>
    <module>project-b</module>
</modules>
```

`project-b` 依赖 `project-a`：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>project-a</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>
```

此时执行：

```bash
mvn clean install
```

Maven 会自动识别到这是一个多模块构建：

- 它**不会从本地仓库**找 `project-a`；
- 而是**在同一个 reactor（当前构建会话）中找到它**；
- Maven 在内部维护一个 **reactor 项目映射表**（Reactor Projects Map），构建时优先解析 reactor 内部的模块；
- 构建顺序：先构建 `project-a` → 安装（install）到 reactor 内存（而不是文件系统） → 再编译 `project-b`。

**结果：**
 不会在本地仓库（`~/.m2/repository`）看到 `project-a.jar`，因为它是通过内存引用（reactor resolution）传递的。

------

### ✅ 情况 2：完全独立的两个项目（不在一个 reactor 里）

比如：

```
/home/user/workspace/libA
/home/user/workspace/serviceB
```

`serviceB/pom.xml` 中：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>libA</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>
```

此时如果你没有先执行：

```bash
cd /home/user/workspace/libA
mvn install
```

Maven **第一次**构建 `serviceB` 时，会尝试：

1. 从本地仓库（`~/.m2/repository/com/example/libA/1.0-SNAPSHOT`）查找；
2. 如果没找到，再去远程仓库；
3. 都没有，就报错：`Could not resolve dependencies for project ...`

除非你在 `serviceB` 的 pom 中明确告诉 Maven：**依赖路径是本地文件系统**。

------

## 📁 二、那 Maven 到底“放在哪里”？

- 如果是 **多模块项目（Reactor 内解析）**：
   👉 根本不放到本地仓库，它是临时保存在 reactor 内存里。

- 如果是 **单独项目 + mvn install 构建过**：
   👉 会被安装到本地仓库路径：

  ```
  ~/.m2/repository/com/example/libA/1.0-SNAPSHOT/libA-1.0-SNAPSHOT.jar
  ```

- 如果只是执行了 `mvn package`：
   👉 仅在项目的 `target/` 下生成 jar，不会放进本地仓库。

------

## 🔍 三、你可以验证的几个命令

### 查看依赖解析来源

```bash
mvn dependency:tree -Dverbose
```

会显示：

```
[INFO] com.example:serviceB:jar:1.0-SNAPSHOT
[INFO] \- com.example:libA:jar:1.0-SNAPSHOT (project artifact)
```

其中 `(project artifact)` 就说明是 reactor 内解析的，不来自仓库。

### 查看 reactor 构建顺序

```bash
mvn validate -X | grep "Building project"
```

------

## 🧠 小结

| 情况                           | 来源                        | 是否写入本地仓库 |
| ------------------------------ | --------------------------- | ---------------- |
| 同一个多模块项目               | Maven reactor 内解析        | ❌ 否             |
| 独立项目且执行了 `mvn install` | 本地仓库 `~/.m2/repository` | ✅ 是             |
| 仅 `mvn package` 构建          | `target/` 目录              | ❌ 否             |

独立的两个项目，在一个idea界面中，没有父项目，在pom.xml引用时它还是找到了

非常好，你说的这种情况是 **很多人第一次遇到都会觉得神奇的**：

> 明明是两个独立项目、没有父子关系，也没 install 到本地仓库，IDEA（IntelliJ）却“自动识别”并能编译运行成功。

这其实并不是 Maven 本身的行为，而是 **IntelliJ IDEA 的增强解析机制（IDEA-level dependency resolution）** 在帮你做了“隐式项目关联”。
 下面我给你完整解释。

------

## 🧩 一、Maven 本身不会做到这点

如果你只用命令行执行：

```bash
mvn clean package
```

Maven **不会自动去找另一个项目的源码目录**。
 只有在那另一个项目先被 `mvn install` 过，它的 jar 存在于：

```
~/.m2/repository/com/example/libA/1.0-SNAPSHOT/
```

时，Maven 才能解析成功。

换句话说：
 👉 **原生 Maven 根本不具备 “自动找到另一个本地工程” 的能力。**

------

## 💡 二、但 IDEA 帮你“聪明地连起来了”

IntelliJ IDEA 的 Maven 插件在项目导入时，会扫描整个工作空间（workspace）里的 Maven 项目（即 `pom.xml` 文件）。

当它发现：

- 当前项目（`serviceB`）依赖了一个 groupId + artifactId + version；
- 同一个 IDEA 工程中 **存在另一个 pom.xml** 对应这个坐标；

它就会自动：

1. 在内存中建立一个 **IDEA 模块依赖（module dependency）**；
2. 将对 `libA-1.0-SNAPSHOT.jar` 的依赖替换为 **直接引用另一个项目的 target/classes 输出目录**。

也就是：

> 不是真正的 Maven 构建，而是 IDEA 的项目级联机制。

------

## 📂 三、IDEA 具体是怎么做的？

在 `.idea/modules.xml` 或 `.idea/workspace.xml` 中，
 IDEA 会记录这些模块关系。

举个例子：

```xml
<module name="libA" filepath="libA/pom.xml"/>
<module name="serviceB" filepath="serviceB/pom.xml"/>
```

然后当你在 `serviceB` 的 `pom.xml` 中写：

```xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>libA</artifactId>
    <version>1.0-SNAPSHOT</version>
</dependency>
```

IDEA 检测到 `libA/pom.xml` 存在且坐标匹配后，就自动建立引用：

> serviceB → module libA (输出路径 target/classes)

不需要 jar、不需要 install、不需要本地仓库。
 这就是为什么你“找不到 jar 文件”却能运行成功。

------

## 🧠 四、验证方法

你可以在 IDEA 中：

### ✅ 1. 打开 “File → Project Structure → Modules → Dependencies”

你会看到：

```
serviceB 依赖于 module libA
```

而不是：

```
serviceB 依赖于 libA.jar
```

### ✅ 2. 命令行验证

退出 IDEA，在命令行执行：

```bash
cd serviceB
mvn clean package
```

这时很可能会报：

```
Could not resolve dependencies for project com.example:serviceB:jar:1.0-SNAPSHOT
```

因为 Maven 找不到 libA 的 jar。

说明 IDEA 只是“在 IDE 层帮你解决了”，并不是 Maven 自身的行为。

------

## 🗂️ 五、那它到底“放在哪里”？

实际上 IDEA 没有把另一个项目“放到哪里”，
 而是通过 module 机制直接引用另一个项目的 **编译输出目录**：

```
libA/target/classes/
```

也就是说：

- 当你在 IDEA 中运行 `serviceB`；
- 它会编译 `libA`；
- 然后在 classpath 中直接引用 libA 的输出目录；
- 并不会产生 libA.jar。

------

## ✅ 六、总结对比表

| 场景                              | 解析来源          | 是否需要 install | 实际引用内容                | 是否写入本地仓库 |
| --------------------------------- | ----------------- | ---------------- | --------------------------- | ---------------- |
| 多模块项目（同 parent）           | Maven reactor     | 否               | 内存中的模块输出            | 否               |
| IDEA 中多个独立项目但同 workspace | IntelliJ 模块引用 | 否               | 另一个项目的 target/classes | 否               |
| 命令行独立构建                    | Maven 本地仓库    | ✅ 需要先 install | ~/.m2/repository 下的 jar   | ✅ 是             |

------

## 🔧 七、如果你希望命令行也能成功构建

执行：

```bash
cd libA
mvn install
```

之后再构建 serviceB：

```bash
cd ../serviceB
mvn package
```

Maven 就能在 `~/.m2/repository` 找到它。
