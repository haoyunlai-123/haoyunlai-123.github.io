```java
class Node<T> {
public T data;
public Node(T data) { this.data = data; }
public void setData(T data) {
System.out.println("Node.setData");
this.data = data;
}
}
class MyNode extends Node<Integer> {
public MyNode(Integer data) { super(data); }
// Node<T> 泛型擦除后为 setData(Object data)，而子类 MyNode 中并没有重写该方
法，所以编译器会加入该桥方法保证多态
public void setData(Object data) {
setData((Integer) data);
}
public void setData(Integer data) {
System.out.println("MyNode.setData");
super.setData(data);
}
}这段代码中最终方法区MyNode类的虚方法表中setData(Object data)是相当于重写了父类的setData(Object data)方法吗？
```
是的，**从虚拟机角度看，MyNode 的 `setData(Object)` 就是“重写”了父类 `Node.setData(Object)`**。

细节如下：

1. 泛型擦除后  
   父类 `Node` 只有  
   `setData(Object)` // 签名 `(Ljava/lang/Object;)V`

2. 桥方法  
   编译器为 MyNode 生成  
   `setData(Object)` // 签名同上，**synthetic + bridge**  
   内部强转后调用 `setData(Integer)`。

3. 虚方法表（vtable）  
   - 父类 vtable 里：  
     index N → `Node.setData(Object)`
   - 子类 vtable 里：  
     同一 index N → `MyNode.setData(Object)`（桥方法）  
   因此当通过父类引用调用 `setData(Object)` 时，实际执行的是 **MyNode 的桥方法**，完全符合“重写”语义。

结论：  
**桥方法在字节码层面就是普通虚方法，占据父类相同槽位，虚拟机把它当成真正的重写处理。**

```
没看懂桥方法有什么作用？举个例子
```
桥方法（bridge method）是**编译器偷偷塞进子类**的一个“翻译官”，没有它，泛型擦除后就会**调错方法**，破坏多态。  
下面用一步到底的“代码+字节码”例子演示——**去掉桥方法会发生什么**，你就明白它的作用了。

------------------------------------------------
1. 先写一段“正常”代码（编译器**会**生成桥方法）
------------------------------------------------
```java
abstract class Node<T> {
    public abstract void setData(T data);
}

class MyNode extends Node<Integer> {
    // 我们只想重写「Integer版本」
    public void setData(Integer data) {
        System.out.println("MyNode.setData(Integer)");
    }
}
```

客户端这样用：
```java
Node<Integer> node = new MyNode();
node.setData(42);          // 期望打印 MyNode.setData(Integer)
```

运行结果 ✅  
`MyNode.setData(Integer)`   // 符合预期

------------------------------------------------
2. 看编译器到底干了啥——javap 反汇编
------------------------------------------------
```bash
$ javap -v MyNode
...
public void setData(java.lang.Integer);
  ...
public synthetic bridge void setData(java.lang.Object);
  Code:
    0: aload_0
    1: aload_1
    2: checkcast     #7   // class java/lang/Integer
    5: invokevirtual #9   // Method setData(LInteger;)V
    8: return
```

桥方法签名正好是 `(Object)`，内部强转后调用我们自己写的 `(Integer)` 版本。

------------------------------------------------
3. 手工**删除**桥方法，模拟“没有桥方法”的世界
------------------------------------------------
我们用 ASM 把桥方法去掉，再跑一次：

```java
Node<Integer> node = new MyNode();
node.setData(42);
```

此时 `node` 的**静态类型**是 `Node<Integer>`，  
但泛型擦除后虚拟机里只认 `Node.setData(Object)`。  
由于子类**没有**签名为 `(Object)` 的方法，  
编译器会把它当成**对父类方法的调用**，于是：

- 父类 `Node` 是抽象方法，直接链接失败；  
- 即使父类有默认实现，也会执行父类版本，**永远进不到子类 `(Integer)` 方法**。

结果 ❌  
要么 `AbstractMethodError`，要么行为错误，**多态被打破**。

------------------------------------------------
4. 一句话总结桥方法的作用
------------------------------------------------
**泛型擦除后，父类只剩 `setData(Object)`；  
子类若只写了 `setData(Integer)`，签名对不上，**虚拟机认为你没重写**。  
桥方法就是用 `(Object)` 签名**占住槽位**，内部再转调 `(Integer)`，  
从而保证**多态正确**——这就是它存在的唯一理由。**

------
-----

```java
node不是赋的MyNode对象吗，node.setData(42)调用是不应该是调自己的setData(Integer data)方法吗？
```
因为**编译器生成调用指令时只看“静态类型”**，而静态类型 `Node<Integer>` 在**泛型擦除后**对应的方法签名是

```
setData(Ljava/lang/Object;)V
```

所以字节码里出现的是

```
invokevirtual Node.setData:(Ljava/lang/Object;)V
```

——它要找的**槽位**是 `(Object)`。

如果没有桥方法 `MyNode.setData(Object)`，子类虚方法表里这一槽仍是**父类版本**（或抽象空槽），`invokevirtual` 就**不会**落到 `setData(Integer)` 上；桥方法正是用**完全相同的 Object 签名**占住槽位，内部再强转去调 `(Integer)`，从而保证**运行时多态**正确发生。

-----
----

```java
也就是说子类虚方法表中有两个方法setData(Object data)和setData(Integer data)?
```
对，**子类虚方法表里确实会出现两个条目**：

1. `setData(Object)`  —— 编译器生成的 **桥方法**（synthetic + bridge）  
2. `setData(Integer)` —— 你自己写的方法

不过要注意：
- 这两个方法**字节码级别签名不同**，所以vtable里**占两个不同槽位**；
- 桥方法 `setData(Object)` 的槽位**正好覆盖**了父类擦除后留下的那个槽，保证多态；
- 你自己写的 `setData(Integer)` 是**额外**的一个普通虚方法，对外暴露给知道子类是 `MyNode` 的代码使用。

总结：  
**桥方法不“抢”原来那个(Integer)槽，而是再插一个(Object)槽，并把旧槽留给自己写的(Integer)方法。**

```java
你上面去掉桥方法后为啥Node<Integer> node = new MyNode();
node.setData(42);会报错？为什么不会调用setData(Integer data)？
```
因为 `node` 的**静态类型**是 `Node<Integer>`，而泛型在运行期被**全部擦除**成 `Node<Object>`，所以编译器在生成调用指令时，只认父类里这一根签名：

```
invokevirtual Node.setData:(Ljava/lang/Object;)V
```

——它要找的是**虚方法表中签名为 `(Object)` 的那个槽位**。

如果子类 **没有提供** `(Object)` 方法（桥方法被删掉），那么子类虚方法表里对应槽位仍是**父类留下的抽象空槽**，链接阶段就报错：

```
AbstractMethodError: Node.setData(Ljava/lang/Object;)V
```

换句话说：  
运行时**根本不会**去匹配 `setData(Integer)`，因为调用方给出的 descriptor 是 `(Object)`，而 `(Integer)` 在 JVM 眼里是**完全不同的方法**。  
桥方法的作用正是用 `(Object)` 签名**占住槽位**，内部再强转去调 `(Integer)`，从而把“擦除后的调用”转接到“程序员写的泛型方法”上。

```java
也就是说此时（去掉桥方法）编译后子类虚方法表中只有setData(Integer data),
但是node.setData(42);调用语句会被处理为调用setData(Object data)，
导致执行到这里时在子类虚方法表找不到这个方法而报错？
```
完全正确，一句话总结：

编译后调用方只认 `setData(Object)`，而子类虚方法表里**只有** `setData(Integer)`，**没有**对应签名的实现 → 链接阶段直接抛 `AbstractMethodError`（或 `NoSuchMethodError`）。  
桥方法就是用 `setData(Object)` 占住槽位，防止这种“签名对不上”的悲剧。
