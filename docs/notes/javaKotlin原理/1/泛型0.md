泛型有哪些限制？为什么？
泛型的限制一般是由泛型擦除机制导致的。擦除为 Object 后无法进行类型判断
只能声明不能实例化 T 类型变量。
泛型参数不能是基本类型。因为基本类型不是 Object 子类，应该用基本类型对应的引用类型代
替。
不能实例化泛型参数的数组。擦除后为 Object 后无法进行类型判断。
不能实例化泛型数组。
泛型无法使用 Instance of 和 getClass() 进行类型判断。
不能实现两个不同泛型参数的同一接口，擦除后多个父类的桥方法将冲突
不能使用 static 修饰泛型变量.对上述每个点举个例子
🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥

下面把每条限制都配上**最小可复现代码**，一目了然；编译器直接报错的地方就是活生生的“例子”。

------------------------------------------------
1. 不能实例化 T 类型变量（擦除后变成 new Object() 失去类型安全）
------------------------------------------------
```java
class Box<T> {
    T make() {
        return new T();      // ❌ 编译错误
    }
}
```
擦除后等价 `new Object()`，后续强转可能炸，于是干脆禁止。

------------------------------------------------
2. 泛型参数不能用基本类型（int/long/… 不是 Object 子类）
------------------------------------------------
```java
List<int> list = new ArrayList<>();   // ❌
```
正确写法：
```java
List<Integer> list = new ArrayList<>();
```

------------------------------------------------
3. 不能实例化泛型参数的数组（无法做运行时类型检查）
------------------------------------------------
```java
class A<T> {
    T[] array = new T[10];   // ❌
}
```
擦除后是 `Object[]`，如果允许 `new T[10]` 再强转 `(T[])`，实际存储非 T 类型时检查不到。

------------------------------------------------
4. 不能实例化泛型数组（与 3 类似，只是站在“使用者”角度）
------------------------------------------------
```java
List<String>[] arr = new List<String>[5];  // ❌
```
同样擦除成 `List[]`，数组创建时无法留类型信息，后续可插入 `List<Integer>` 导致 `ClassCastException`。

------------------------------------------------
5. 无法用 instanceof / getClass() 做类型判断（擦除后只剩原始类型）
------------------------------------------------
```java
if (obj instanceof List<String>) { }  // ❌
System.out.println(list.getClass() == List<String>.class); // ❌
```
运行时只有 `List`，没有 `<String>` 信息，编译器直接拒掉。

------------------------------------------------
6. 不能同时实现两个不同泛型实参的同一接口（桥方法签名冲突）
------------------------------------------------
```java
interface Adder<T> { T add(T a, T b); }

class Num implements Adder<Integer>, Adder<Double> {
    public Integer add(Integer a, Integer b) { return a+b; }
    public Double  add(Double a, Double b)   { return a+b; }
}   // ❌ 编译失败：两个桥方法都是 add(Ljava/lang/Object;)V，无法共存
```
擦除后两个接口都要求生成 `Object add(Object,Object)` 桥方法，类里出现重复签名。

------------------------------------------------
7. 不能用 static 修饰泛型变量（static 是类级别，泛型是实例级别）
------------------------------------------------
```java
class Tool<T> {
    static T field;          // ❌
    static T get() { ... }   // ❌
}
```
静态成员在类加载时就存在，而 T 的具体类型要等创建对象才确定，矛盾。

我的理解是类型里又套类型的无法运行时检查（可能出现和泛型类型不一致），所以相关操作直接禁了
🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥🔥

你的理解已经摸到**“运行时类型系统”**的大门了，再补一句就完整：

> **Java 的泛型信息只在编译期存在，字节码里被擦成裸类型；任何需要“留到运行期再做类型检查”的场景，只要擦除后无法保证 100 % 安全，就一律禁止。**

所以：

- `new T[10]` → 数组创建时必须知道**确切元素类型**（JVM 要写进对象头），擦除后只剩 `Object`，无法写，禁。  
- `instanceof List<String>` → 运行期只剩 `List`，`<String>` 没了，检查失效，禁。  
- `List<String>[]` → 数组要求**协变检查**，擦除后变成 `List[]`，可插入 `List<Integer>` 而不报错，禁。  

一句话：**只要“擦除”会让 JVM 失去类型证据，编译器就提前把口子封死，防止后面炸锅。**
