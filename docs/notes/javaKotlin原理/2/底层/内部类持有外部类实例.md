帮我分析下这个案例：发生了内存泄露，原因是TestActivity对象泄露了，但是匿名线程中不是使用的静态字段key吗？key指向的是Object对象，TestActivity又没在GC Root中，为什么TestActivity的实例会泄露？

这个案例里 **Activity 泄露的关键不在 `key` 是不是 static**，而在于：

> **匿名内部类 `new Thread(){...}` 会“偷偷”持有外部类 `TestActivity` 的引用（`this$0`）**，导致线程活着时，Activity 就一直可达、不能回收。

------

## 1）匿名内部类为什么会引用 TestActivity？

`new Thread(){ public void run(){...} }` 这不是普通的 `Thread`，而是一个**匿名内部类**（Anonymous Inner Class）。

Java 编译器会把它编译成类似：

```java
class TestActivity$1 extends Thread {
    final TestActivity this$0;  // 编译器自动加的

    TestActivity$1(TestActivity outer) {
        this.this$0 = outer;
    }

    public void run() { ... }
}
```

也就是说：**即使你的 run() 里完全没写 `TestActivity.this`，它也会自动带着 `this$0` 把外部 Activity 引用住。**

所以线程对象里实际藏着一条引用链：

```
Thread (TestActivity$1)` → `this$0` → `TestActivity
```

------

## 2）那 GC Root 从哪来？Activity 不是 Root 啊

对，`TestActivity` 本身不是 Root，但 **Thread 是 Root 的“近亲”**：

- JVM 里每个存活的线程都会在 JVM/ART 的线程列表里被强引用（可视为 GC Roots 可达）
- 线程的栈帧、Thread 对象、ThreadGroup 等都会被扫描

因此只要线程没结束，这条链就是：

**GC Roots**
→ `Thread`（活着的线程对象）
→ 匿名内部类字段 `this$0`
→ `TestActivity`

Activity 就变成可达了，GC 不会回收它。

------

## 3）`key` 是 static 还是不是 static，影响的是“锁对象生命周期”，不是泄露的核心

你说得对：`key` 是 `static final Object`，它只指向一个 Object。

但注意：

- `synchronized(key){ key.wait(); }` 让这个线程**无限期 WAITING**
- 因为没有任何地方 `notify/notifyAll`，线程不会结束
- 线程不结束 → Thread 一直是 Root 可达 → Activity 一直被 `this$0` 引用住

`key` 的 static 只是让锁对象永远存在（并且所有 Activity 实例共享），**它不是把 Activity 关联进去的那根线**。

把 `key` 换成非 static 也照样泄露（甚至更直观：会把 `key` 一起泄露），只要还是匿名内部线程 + 长时间阻塞。

------

## 4）为什么 run 里看不到 Activity 的使用，但仍然引用了它？

因为这是 **“隐式捕获外部 this”**：

- 非 static 内部类 / 匿名内部类：默认持有外部类引用
- 只有当你把内部类变成 `static`（或放到外部文件顶级类），才不会自动带 `this$0`

------

## 5）怎么改才不泄露？

### ✅ 改法 1：不用匿名内部类线程（改成静态内部类 + WeakReference）

```java
static class WaitThread extends Thread {
    private final WeakReference<Activity> ref;
    WaitThread(Activity a){ ref = new WeakReference<>(a); }

    public void run() {
        synchronized (key) {
            try { key.wait(); } catch (InterruptedException ignored) {}
        }
    }
}
```

### ✅ 改法 2：用 `Executor`/`HandlerThread` 管理生命周期，并在 `onDestroy()` 中中断

```java
Thread t = new Thread(() -> {
    synchronized (key) {
        try { key.wait(); } catch (InterruptedException ignored) {}
    }
});
t.start();

@Override protected void onDestroy() {
    t.interrupt(); // 或者 notify
    super.onDestroy();
}
```

### ✅ 改法 3：根本别用 `wait()` 挂死线程（Android 上更推荐 CountDownLatch / Condition / 阻塞队列等可控机制）

------

## 一句话总结

**泄露 TestActivity 的“引用线”不是 `key`，而是匿名内部类线程自动持有的 `this$0` 外部引用；线程又因为 `wait()` 永久阻塞而一直存活，导致 Activity 一直可达。**

匿名内部类，成员内部类，方法中的内部类是不是创建对象时都会持有外部类的实例？

是的✅——**只要它们是“非 static 的内部类”**，在**创建对象时都会自动持有外部类实例引用**（编译器偷偷塞进去的那根 `this$0`）。

下面分情况说清楚：

------

# 结论总览（最重要）

✅ **会持有外部类实例**（创建对象时就绑定）：

1. **成员内部类（非 static）**
2. **方法内部类（local class，非 static）**
3. **匿名内部类（anonymous class）**

❌ **不会持有外部类实例**：

1. **静态内部类（static nested class）**
2. **顶级类（top-level class）**
3. **Lambda（多数情况下不会额外生成 this$0，但可能捕获变量）**

------

# 1）成员内部类（非 static）✅一定持有外部类实例

```java
class Outer {
    class Inner {}
}
```

创建时必须依赖 Outer 实例：

```java
Outer o = new Outer();
Outer.Inner in = o.new Inner();
```

底层等价于：

```java
new Inner(o); // 编译器传入外部 this
```

------

# 2）方法内部类（局部内部类）✅一定持有外部类实例（若在实例方法中）

```java
class Outer {
    void foo() {
        class Local {}
        Local l = new Local();
    }
}
```

`Local` 是在 `foo()` 里定义的，但它仍然是 **非 static 内部类**，所以：

- 它也会有 `Outer this$0`
- 也会捕获它用到的局部变量（final/effectively final）

⚠️ 例外：如果它定义在 **static 方法里**，那就没有外部实例可抓：

```java
static void foo() {
    class Local {}
}
```

这种情况下不会有 `this$0`，因为没有外部 this。

------

# 3）匿名内部类 ✅一定持有外部类实例（若在实例上下文中创建）

```java
class Outer {
    void foo() {
        new Thread() { };
    }
}
```

同理，它会被编译成 `Outer$1` 并带上：

```java
final Outer this$0;
```

所以匿名内部类非常容易引发 Android 的 Activity 泄漏。

------

# 4）静态内部类（static nested class）❌不会持有外部类实例

```java
class Outer {
    static class Inner {}
}
```

创建时不需要外部对象：

```java
Outer.Inner in = new Outer.Inner();
```

它不会生成 `this$0`。

------

# 5）你要的“更严谨一句话”

> **任何非 static 的内部类对象，都隐式包含一个指向其外部类实例的引用（this$0），从而可以访问外部类成员。**
