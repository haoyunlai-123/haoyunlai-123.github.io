java中对接口使用反射拿到方法对象，能传入实现类对象执行吗？

可以，而且这就是 Java 反射最常见的用法之一。  
步骤只有两步：

1. 用接口的 `Class` 拿到 `Method` 对象；  
2. 用 `method.invoke(实现类对象, 参数…)` 执行。

关键点：  
- 接口里声明的方法，实现类一定继承得到，所以 `接口.class.getMethod(...)` 取到的 `Method` 完全可以作用在“实现类实例”上。  
- 运行期实际执行的是实现类覆写后的逻辑（多态）。  
- 如果实现类是匿名、代理或者 JDK 动态代理对象，也一样适用。

示例代码：

```java
public interface Hello {
    String say(String name);
}

public class HelloImpl implements Hello {
    @Override
    public String say(String name) {
        return "Hello, " + name;
    }
}

public class Demo {
    public static void main(String[] args) throws Exception {
        HelloImpl impl = new HelloImpl();

        // 1. 用接口拿到 Method
        Method m = Hello.class.getMethod("say", String.class);

        // 2. 对实现类实例调用
        String result = (String) m.invoke(impl, "world");

        System.out.println(result); // Hello, world
    }
}
```

注意事项  
1. 如果实现类没有实现该接口，运行期不会报错，但编译期就无法通过 `implements`。  
2. 如果方法签名在实现类里被“收窄”访问权限（例如接口 `public` 、实现类改成 `protected`），`invoke` 会抛 `IllegalAccessException`，需要 `setAccessible(true)`。  
3. 接口里的 `static/private/default` 方法反射规则与实例方法相同，但 `static` 方法 `invoke` 时第一个参数传 `null` 即可。

动态代理中handler中的invoke方法传入的method对象是不是就是根据接口获取到的？

是，但也不全是——它“看起来”就是你通过 `接口.class.getMethod(...)` 能拿到的那个 `Method`，但底层并不是简单地“重复调用一次 `getMethod`”，而是 JVM 在运行期为你生成的 **同名、同签名** 的 `Method` 对象。

具体过程（以 JDK 动态代理为例）：

1. 运行期 `Proxy.newProxyInstance` 会按接口列表生成一个 **代理类**（字节码），里面每个接口方法都会被实现成类似：
   ```java
   public final String say(String s) throws  {
       return (String) this.h.invoke(this, m3, new Object[]{s});
   }
   ```
   其中 `m3` 是该代理类里的一个 `static final java.lang.reflect.Method` 字段。

2. 这些 `m1/m2/m3…` 字段在代理类被 **初始化时** 通过 `Class.getMethod` 或 `Class.getDeclaredMethod` 一次性取到并缓存，**不是每次调用都重新反射**。

3. 因此当你在自己的 `InvocationHandler.invoke(proxy, method, args)` 里收到的 `method` 参数，其实就是上面那个 **由代理类缓存、与接口中声明完全对应的 Method 对象**。  
   对它再次 `method.invoke(真实实现, args)` 完全合法，也正好完成了“把调用转发给被代理对象”的动作。

一句话总结：  
handler 收到的 `method` 与“我们手动 `接口.class.getMethod(...)` 拿到的”是 **同一个方法签名、同一个 `java.lang.reflect.Method` 实例**（或至少功能等价），所以可以放心地拿它再去反射调用真正的实现类对象。
