早期绑定和晚期绑定是什么意思？和静态链接和动态链接有什么区别？

非常棒 💯，你这个问题其实是 **JVM 调用机制、编译原理、以及动态分派** 的交汇点。
 “早期绑定 / 晚期绑定” 和 “静态链接 / 动态链接” 看似相似，但其实是**不同层次**的两个概念：

- 前者关注的是 **调用时机（语义绑定）**；
- 后者关注的是 **符号解析方式（链接机制）**。

我来帮你系统理一下，一次性彻底讲清楚 👇

------

## 🧩 一、什么是“绑定”（Binding）？

> **绑定（Binding）**：确定一个符号（变量、方法、函数调用）和它所指向的实体（内存地址、函数体、对象方法）之间的对应关系。

------

## 🧱 二、早期绑定（Early Binding）与晚期绑定（Late Binding）

| 名称                     | 绑定时机 | 发生阶段 | JVM 中体现                         | 示例                 | 关键特征                     |
| ------------------------ | -------- | -------- | ---------------------------------- | -------------------- | ---------------------------- |
| **早期绑定（静态绑定）** | 编译期   | 编译阶段 | **符号在编译期就能确定目标**       | 方法重载（Overload） | 无需运行时判断，性能高       |
| **晚期绑定（动态绑定）** | 运行期   | 运行时   | **根据对象的实际类型确定调用目标** | 方法重写（Override） | 多态核心，需运行时查虚方法表 |

------

### ✅ 例子对比：

```java
class A {
    void say() { System.out.println("A"); }
}
class B extends A {
    void say() { System.out.println("B"); }
}

public class Main {
    public static void main(String[] args) {
        A a = new B();
        a.say(); // ?
    }
}
```

输出：

```
B
```

解释：

- `a` 的**静态类型**是 `A`，但**实际类型**是 `B`。
- 调用 `a.say()` 时，**编译期只知道它有 say() 方法（A类有定义）**；
- 但真正执行时 JVM 要查“实际类型是 B”的方法表，找到 `B.say()`；
   → **晚期绑定**。

------

### 🧩 早期绑定的例子（编译期就能确定）

```java
class C {
    void print(int x) {}
    void print(String s) {}
}
```

这里是**重载（Overload）**，
 编译器根据参数类型就能确定调用哪个版本。
 → 这是 **早期绑定**。

------

### 🧠 JVM 层面：

在 class 文件的“方法调用指令”中可以看出两种绑定方式：

| 指令              | 用途                                       | 绑定类型       |
| ----------------- | ------------------------------------------ | -------------- |
| `invokestatic`    | 调用静态方法                               | 早期绑定       |
| `invokespecial`   | 调用构造器、私有方法、父类方法             | 早期绑定       |
| `invokevirtual`   | 调用普通实例方法（虚方法）                 | 晚期绑定       |
| `invokeinterface` | 调用接口方法                               | 晚期绑定       |
| `invokedynamic`   | 运行时决定调用目标（Lambda, 动态语言支持） | **更晚的绑定** |

------

## ⚙️ 三、静态链接（Static Linking）与动态链接（Dynamic Linking）

“链接”是在类加载阶段进行的。
 主要发生在 **JVM 的“链接（Linking）”阶段**。

### JVM 加载阶段：

```
加载（Loading） → 链接（Linking） → 初始化（Initialization）
```

**链接阶段又分三步：**

1. 验证（Verify）
2. 准备（Prepare）
3. **解析（Resolve）**

------

### 🔹 静态链接（Static Linking）

> 在类加载时（解析阶段），就把符号引用解析为直接引用（内存地址或方法入口）。

📌 特点：

- 在**类加载时**就完成；
- 性能高；
- 通常用于**final、private、static 方法或字段**；
- 这种引用在类加载后就固定，不再变化。

```java
private void foo() {}
public static void bar() {}
```

这些方法在类加载时就能确定目标地址，
 JVM 会在“解析（resolve）阶段”完成绑定。
 → **静态链接**。

------

### 🔹 动态链接（Dynamic Linking）

> 直到方法调用时（运行时）才根据实际类型解析符号引用。

📌 特点：

- 在**运行时**才完成；
- 依赖对象的实际类型；
- 用于**虚方法调用（多态）**；
- JVM 会在**调用点**动态查找方法表。

例如：

```java
A a = new B();
a.say(); // 动态链接
```

编译期只知道 `A.say()` 的符号引用，
 运行时才在对象的虚方法表（vtable）中查到 `B.say()` 的入口地址。

------

## 🔄 四、两者的区别与对应关系

| 层次                     | 概念                 | 对应关系                                   |
| ------------------------ | -------------------- | ------------------------------------------ |
| **语义层（语言层）**     | 早期绑定 vs 晚期绑定 | 决定调用目标在何时确定（编译期 vs 运行期） |
| **JVM 执行层（链接层）** | 静态链接 vs 动态链接 | 决定符号引用在何时解析（加载时 vs 运行时） |

它们的区别在于关注点不同：

| 比较维度     | 早/晚绑定              | 静/动链接                        |
| ------------ | ---------------------- | -------------------------------- |
| 关注对象     | 方法调用行为           | 符号解析机制                     |
| 时机         | 编译期 vs 运行期       | 加载时 vs 调用时                 |
| JVM 对应阶段 | 执行阶段               | 类加载阶段                       |
| 实现机制     | 虚方法表、动态分派     | 解析 vs 运行时查找               |
| 示例         | 重载(早期)、重写(晚期) | private 方法(静态)、虚方法(动态) |

------

## 💡 五、举一个贯通所有概念的例子：

```java
class Parent {
    void hello() { System.out.println("Parent"); }
    static void hi() { System.out.println("Parent static"); }
}

class Child extends Parent {
    @Override
    void hello() { System.out.println("Child"); }
    static void hi() { System.out.println("Child static"); }
}

public class Test {
    public static void main(String[] args) {
        Parent p = new Child();
        p.hello(); // 动态绑定 + 动态链接
        p.hi();    // 早期绑定 + 静态链接
    }
}
```

**分析：**

- `p.hello()` 是**实例方法重写** → **晚期绑定 + 动态链接**
- `p.hi()` 是**静态方法隐藏** → **早期绑定 + 静态链接**

输出：

```
Child
Parent static
```

------

## ⚙️ 六、总结一句话

| 名称                     | 时机                 | 体现                                     |
| ------------------------ | -------------------- | ---------------------------------------- |
| **早期绑定（静态绑定）** | 编译期确定调用目标   | 静态方法、final 方法、private 方法、重载 |
| **晚期绑定（动态绑定）** | 运行时确定调用目标   | 实例方法、重写、多态                     |
| **静态链接**             | 类加载时解析符号引用 | 类加载时固定引用目标                     |
| **动态链接**             | 运行时才解析符号引用 | 多态 / invokevirtual 运行时解析          |

🔹 简单记法：

> 绑定讲“什么时候确定方法”；
>  链接讲“什么时候解析引用”。

非常好 👍，这是很多人初学 Java 时容易混淆的问题。
结论先给出：

> ✅ 但可以被 **隐藏（method hiding）**。

## 🧩 一、为什么不能重写？

因为：

* **静态方法属于类（Class），不属于实例（Object）**；
* **重写（Override）** 是一种 **运行时的动态绑定行为**，
  依赖对象的“实际类型”；
* 而静态方法在 **编译期就绑定到类上（早期绑定）**。

换句话说：

> 重写的前提是“方法调用依赖于实例对象”；
> 但静态方法根本不依赖对象。

---

## ⚙️ 二、例子验证

```java
class Parent {
    static void sayHello() {
        System.out.println("Hello from Parent");
    }
}

class Child extends Parent {
    static void sayHello() {
        System.out.println("Hello from Child");
    }
}

public class Test {
    public static void main(String[] args) {
        Parent p = new Child();
        p.sayHello();
    }
}
```

输出：

```
Hello from Parent
```

解释：

* 这里 `p` 的**静态类型**是 `Parent`；
* 调用 `p.sayHello()` 时，**编译期**就决定调用 `Parent.sayHello()`；
* 子类的同名静态方法只是“隐藏”了父类的那个方法，不构成重写。

---

## 🧠 三、Java 规范定义（JLS 8.4.8.2）

Java 语言规范明确规定：

> **A static method cannot be overridden, but can be hidden.**
> 静态方法不能被重写，但可以被隐藏。

---

## 🧩 四、“隐藏（Hiding）” 和 “重写（Override）” 的区别

| 特性         | 静态方法（Hiding）        | 实例方法（Override）   |
| ---------- | ------------------- | ---------------- |
| 是否依赖对象     | ❌ 仅依赖类              | ✅ 依赖实例           |
| 绑定时机       | 编译期（早期绑定）           | 运行期（晚期绑定）        |
| 调用方式       | `Parent.sayHello()` | `obj.sayHello()` |
| 能否通过多态改变行为 | ❌ 不能                | ✅ 可以             |
| 关键字修饰      | `static`            | 非 `static`       |

---

## 🧩 五、如果你想实现“多态行为”

要想在子类中让方法多态（运行时调用不同版本），
就必须使用**实例方法**：

```java
class Parent {
    void sayHello() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    @Override
    void sayHello() {
        System.out.println("Child");
    }
}

Parent p = new Child();
p.sayHello();  // 输出 Child
```

这才是典型的**重写（override） + 晚期绑定（动态分派）**。

---

## ✅ 六、总结一句话

| 对象             | 能否重写父类静态方法？     | 行为         |
| -------------- | --------------- | ---------- |
| 子类             | ❌ 不能            | 只是“隐藏”父类方法 |
| 父类引用调用子类静态方法   | ❌ 不可能（编译期绑定到父类） |            |
| 实例方法（非 static） | ✅ 可以重写，实现多态     |            |

> 🔹 静态方法属于类，编译期绑定；
> 🔹 实例方法属于对象，运行期绑定。

## 🧩 一、虚方法调用（invokevirtual / invokeinterface）

你的理解完全正确：

> ✅ **虚方法调用（Virtual Call）** 确实是在运行时根据对象的 `Klass*`（即类元信息指针）去动态查找目标方法入口地址的。

在 HotSpot 内部，对象布局大致如下：

```
+-----------------+
| mark word       |  // 锁状态、hash、GC age
+-----------------+
| Klass*          |  // 指向元空间(Metaspace)中的 Klass 对象
+-----------------+
| 实例字段 ...     |
```

当执行 `invokevirtual` 指令时，JVM 做的事是：

1. 从操作数栈弹出对象引用；
2. 通过对象的 `Klass*` 找到它的“类元数据”（在 Metaspace 中）；
3. 通过该类的 **虚方法表（vtable）** 查找到实际方法入口地址；
4. 跳转执行目标方法的机器码。

这就是 **晚期绑定（动态分派）** 的本质：

> “运行时根据对象的实际类型决定方法入口地址。”

虚方法表是类加载、链接、初始化完成时由 JVM 构建好的数组结构：

```
Klass (for class B)
  -> vtable[]
        [0] -> Object.toString
        [1] -> A.say
        [2] -> B.say   ← 这里覆盖了父类的入口地址
```

所以：

* 调用父类虚方法时仍然通过索引查表；
* 只不过子类的 vtable 对应位置被替换为子类的实现。

---

## ⚙️ 二、实方法调用（invokestatic / invokespecial）

你说的第二部分也对，只是可以更精确地区分：

> ✅ **实方法（非虚方法）** 的调用目标在“类加载 → 链接 → 解析”阶段已经静态确定。

也就是：

1. 当类文件被类加载器加载进 JVM（方法区/元空间）；
2. 在 **链接阶段的解析（resolve）子阶段**，
   所有能静态确定的符号引用（比如 `private`、`static`、`final` 方法）
   都会被转换为 **直接引用**（指向方法元信息或代码入口）；
3. 这样，执行 `invokestatic`、`invokespecial` 时，不需要再动态查表。

所以：

* `static`、`private`、`final` 方法：在解析阶段完成绑定 → 早期绑定；
* `invokevirtual`、`invokeinterface`：必须运行时查表 → 晚期绑定；
* `invokedynamic`：甚至连目标方法句柄都运行时再决定（最晚绑定）。

---

## 🔍 对比总结表

| 调用类型       | JVM 指令                          | 绑定时机      | 查找方式                | 示例方法                   | 是否查 vtable |
| ---------- | ------------------------------- | --------- | ------------------- | ---------------------- | ---------- |
| 实方法（静态绑定）  | `invokestatic`, `invokespecial` | 类加载“解析”阶段 | 直接引用（符号解析时固定）       | static / private / 构造器 | ❌ 否        |
| 虚方法（动态绑定）  | `invokevirtual`                 | 运行时       | 对象.Klass.vtable 查找  | 普通实例方法（可重写）            | ✅ 是        |
| 接口方法（动态绑定） | `invokeinterface`               | 运行时       | itable 查找           | 接口调用                   | ✅ 是        |
| 动态调用       | `invokedynamic`                 | 首次执行时     | Bootstrap Method 决定 | Lambda / 动态语言          | ✅ 运行时绑定    |

---

## 🧠 三、扩展一点：为什么虚方法查找也不慢？

HotSpot 做了很多优化：

* 虚方法表查找是**O(1)** 数组访问；
* JIT 会进行 **inline cache（内联缓存）**；
* 对象类型稳定时会直接内联目标方法；
* 因此虚调用性能几乎接近直接调用。

---

## ✅ 四、总结一句话

> * **虚方法调用**：运行时通过对象的 `Klass*` → `vtable` 查找入口地址（动态绑定，动态链接）
> * **实方法调用**：在类加载的“解析阶段”符号引用已经固定（早期绑定，静态链接）
