🔥 你这个思考非常专业了——这正是 Redis 在高并发场景中「从单机性能 → 集群扩展」的关键转折点。
 你说的完全没错：**点赞属于典型高频写操作**，哪怕每次只执行一条 Lua 脚本，在 Redis 单线程模型下依然可能成为瓶颈。
 但别急，我们可以系统性分析 **瓶颈点、真实负载量、优化策略、工程折中方案** 👇

------

## 🚦 一、你说的“Redis 压力大”——确实存在，但要看规模

我们先定量算一下。

Redis 单机的 QPS（纯内存操作）：

- 单机 **10–15 万 QPS** 是轻轻松松；
- Lua 脚本执行（`SADD + SISMEMBER + SREM`）这种操作链，大约 **1–2 万 QPS/核**。

👉 如果用户点赞量是这种级别：

| 业务场景                   | Redis 是否吃紧                   |
| -------------------------- | -------------------------------- |
| 小型博客系统（1–10万用户） | 完全没问题                       |
| 中型社区（百万用户）       | 单机够用，监控 CPU 即可          |
| 超大平台（上千万用户）     | ✅ 需要优化（分片、异步、批处理） |

所以你感觉「每个请求都串行执行脚本」确实对——但 Redis 的单线程是**事件驱动 + 内存级原子操作**，执行一次 Lua 只要几十微秒。
 真瓶颈出现在 **点赞量爆炸、持续写入时段集中** 的情况下。

------

## ⚙️ 二、Redis 压力的来源

核心瓶颈主要有三类：

| 类型     | 表现                   | 原因                        |
| -------- | ---------------------- | --------------------------- |
| 写压力   | Redis CPU 占用高       | Lua 脚本执行太频繁          |
| 网络开销 | RTT 高                 | 每次都要一次请求/响应       |
| 热点 Key | 某篇爆款博客被疯狂点赞 | 单 key 串行导致阻塞其他请求 |

## 🧩 三、三层优化策略（从轻到重）

### ✅ **1. 异步批量落库（最常见）**

你现在每次都更新 MySQL 一次：

```java
update().setSql("liked = liked + 1").eq("id", id).update();
```

→ 把这一步变为异步批量更新，可以立刻减 80% 的压力。

**优化方案：**

- 点赞操作仅更新 Redis；
- 异步线程 / MQ 每隔 3s 扫描增量数据批量写入 MySQL。

示例：

```java
// Redis: INCR 计数器或 Lua 返回状态
stringRedisTemplate.opsForHash().increment("blog:like:counter", blogId.toString(), delta);
```

后台定时任务批量写回：

```sql
UPDATE blog SET liked = liked + ? WHERE id = ?
```

✅ 优点：

- Redis 成为唯一“即时状态源”；
- MySQL 减压数十倍；
- 用户体验无感知。

------

### ⚙️ **2. 分片 key（解决热点问题）**

假设爆款博客 id=100 被 100 万人点赞，
 所有操作都打到 `"blog:liked:100"` 这个 key 上，会形成串行阻塞。

解决：**分片 key**。

```java
String key = "blog:liked:" + blogId + ":" + (userId.hashCode() & 15);
```

分成 16 个集合：

```
blog:liked:100:0
blog:liked:100:1
...
blog:liked:100:15
```

点赞时打到对应分片；
 统计总数时 `SUNION` 汇总或定期聚合。

✅ 优点：

- 热点分摊；
- Lua 仍原子但只锁局部分片；
- 扩展性 ×16。

------

### ⚡ **3. 多级缓存 + 异步队列 + 限流**

这是大厂级方案，三个核心组件：

#### (1) 本地缓存预防 Redis 压力：

Guava Cache / Caffeine 保存用户点赞状态短时间（5s–10s），
 避免短时间内重复请求 Redis。

```java
if (localCache.getIfPresent(userId_blogId) != null) return Result.ok();
```

#### (2) 异步队列批量执行 Lua：

多个点赞请求打入内存队列，合并执行一批 Lua 调用，减少网络往返。

#### (3) 高频用户限流：

用 Redis 的计数器限制「同一用户每秒点击次数」。

```lua
-- 限流Lua脚本
local key = KEYS[1]
local current = redis.call('INCR', key)
if current == 1 then
  redis.call('EXPIRE', key, 1)
end
if current > 5 then
  return 0
end
return 1
```

✅ 优点：

- 限制刷点击；
- 控制并发量；
- 平滑 Redis 压力。

## 🧠 四、现实取舍：为什么仍推荐 Lua

你可以这么看：

| 特性       | Lua 脚本方案 | 分布式锁方案 | 异步方案 |
| ---------- | ------------ | ------------ | -------- |
| 一致性     | ✅ 最强       | ✅ 强         | ⚠️ 弱     |
| 性能       | ⚡ 高         | ❌ 低         | ✅ 高     |
| 实现复杂度 | ⭐⭐           | ⭐⭐⭐          | ⭐⭐⭐⭐     |
| 延迟       | 1ms 内       | 高           | 可接受   |
| 容错性     | ✅ 自动恢复   | ⚠️ 需手动     | ✅ 自动   |

**结论：**

- 小中规模系统 → Lua 脚本最优；
- 超大系统 → “Lua + 异步批量 + 分片 key” 才是工业级组合拳。

## 🧩 一、你说的情况确实存在（且非常常见）

你现在的逻辑：

```java
if (res == 1L) {
    update().setSql("liked = liked + 1").eq("id", blogId).update();
} else if (res == -1L) {
    update().setSql("liked = liked - 1").gt("liked", 0).eq("id", blogId).update();
}
```

问题在于：
 **MySQL 计数不是原子的、不是幂等的**，而 Redis 状态是幂等的。

### 举个真实并发例子：

假设用户重复点击 3 次：

| 操作                    | Redis 状态               | MySQL 操作          | MySQL liked 实际值 |
| ----------------------- | ------------------------ | ------------------- | ------------------ |
| 第1次点赞               | 加入集合（OK）           | +1                  | 1                  |
| 第2次取消               | 移出集合（OK）           | -1                  | 0                  |
| 第3次点赞               | 加入集合（OK）           | +1                  | 1                  |
| 🚨 第4次取消（并发延迟） | Redis 已被改成「已取消」 | **可能重复执行 -1** | ❌ liked = -1       |

为什么？
 因为第 4 次取消的 Redis 操作在 MySQL 执行完第 3 次加 1 之前完成了，导致 MySQL 再次减 1。
 **Redis 和 MySQL 状态脱节**。

------

## 🧠 二、问题的本质：跨系统状态不同步

- Redis：状态型（0/1）
- MySQL：计数型（+1 / -1）
- Redis 是原子操作，但 MySQL 不是。
- Redis 状态变化频繁时，MySQL 无法保持强一致。

所以 ——
 无论你怎么加 `gt(0)`、怎么 CASE WHEN、怎么加锁，
 都无法在分布式系统中 100% 防止「丢赞 / 多减」。

------

## ⚙️ 三、根本解决思路（看规模选方案）

### ✅ 方案 1：**异步批量同步（推荐方案，主流网站用这个）**

**核心思想：**

- Redis 存储点赞状态；
- MySQL 不实时更新；
- 通过异步任务（定时 / 消息队列 / Stream）定期同步。

**做法：**

1. 点赞接口只改 Redis：

   ```lua
   if (redis.call('ZSCORE', key, userId)) then
       redis.call('ZREM', key, userId)
       return -1
   else
       redis.call('ZADD', key, timestamp, userId)
       return 1
   end
   ```

2. 不直接更新 MySQL；

3. 定期任务批量写回：

   ```sql
   UPDATE tb_blog SET liked = (SELECT COUNT(*) FROM redis_zset) WHERE id = ?
   ```

4. 或者：

   - 使用 `Redis Stream` 记录操作日志；
   - 后台异步消费更新 MySQL。

👉 优点：

- 不会丢赞；
- 可抗高并发；
- 最终一致性；
- 抖音 / 微博 / 知乎都是这么干的。

👉 缺点：

- 查询时 Redis 和 MySQL 可能暂时不一致；
- 增加一点延迟（但可控）。

------

### ✅ 方案 2：**纯 Redis 计数（MySQL 延后存）**

点赞数用 Redis 的计数器维护：

```lua
if redis.call('ZSCORE', key, userId) then
    redis.call('ZREM', key, userId)
    redis.call('DECR', countKey)
    return -1
else
    redis.call('ZADD', key, timestamp, userId)
    redis.call('INCR', countKey)
    return 1
end
```

- `countKey` 存储点赞总数；
- 查询直接读 Redis；
- 定时任务同步 Redis 计数回数据库。

👉 优点：

- 所有操作原子；
- MySQL 只是冗余存储；
- 实时查询无压力。

👉 缺点：

- Redis 挂了前若没刷盘，会丢数据；
- 需要定期 flush。

------

### ✅ 方案 3：**Redis 事务（MULTI + EXEC）+ 队列异步落库**

如果希望短时间内保持强一致：

1. Redis 执行事务（或 Lua）更新点赞状态；
2. 同时将操作写入一个 Redis List（或 Stream）；
3. 异步消费者从 Stream 读取并更新数据库。

即：

- 写 Redis：快；
- 写 MySQL：异步；
- 不阻塞主请求；
- 支持重试、补偿机制。

## 📘 四、总结对比

| 方案                       | 实时一致性 | 性能  | 数据安全     | 是否推荐       |
| -------------------------- | ---------- | ----- | ------------ | -------------- |
| 直接操作 MySQL             | ✅ 强一致   | ❌ 慢  | ✅            | ❌ 不适合高并发 |
| Redis + MySQL 同步更新     | ❌ 弱一致   | ✅ 快  | ⚠️ 有丢失风险 | ⚠️ 临时可用     |
| Redis 异步批量同步         | ⏱ 最终一致 | ✅✅ 快 | ✅ 安全       | ✅ 推荐         |
| Redis 计数器 + Stream 落库 | ✅ 原子     | ✅✅ 快 | ✅✅ 高可用    | ✅✅ 最推荐      |

## 🚀 五、如果你想保持代码简单（又不改架构）

你可以临时改成这种 **“防溢 + 校验回写”** 模式：

```java
if (res == 1L) {
    // 点赞 +1
    update().setSql("liked = liked + 1").eq("id", blogId).update();
} else if (res == -1L) {
    // 防止负数
    boolean success = update().setSql("liked = liked - 1")
                              .gt("liked", 0)
                              .eq("id", blogId)
                              .update();
    if (!success) {
        // 若没有更新成功，说明出现并发误差，重新同步
        Long count = stringRedisTemplate.opsForZSet().size(key);
        update().setSql("liked = " + count).eq("id", blogId).update();
    }
}
```

🔹 一旦检测到 MySQL 更新“没匹配上条件”（说明并发错乱），
 它立刻从 Redis 里**重新计算点赞数并矫正**。
 这样就实现了“自动自愈”。
