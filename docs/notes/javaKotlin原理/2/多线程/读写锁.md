读写锁为什么持有读锁再获取写锁会阻塞？

非常棒 👍 ——你问到了 **ReentrantReadWriteLock** 的核心机制之一：

> “为什么持有读锁再去获取写锁会阻塞？”

这个问题的关键点在于：**读锁是共享的，写锁是独占的**，而 AQS 的底层 `state` 状态控制了它们的互斥关系。
 我们来一点点拆开 👇

------

## 🧩 一、前置知识：读写锁的基本语义

在 `ReentrantReadWriteLock` 中：

| 锁类型            | 特征                       | 是否可共享 |
| ----------------- | -------------------------- | ---------- |
| 读锁（ReadLock）  | 多线程可同时持有（只读）   | ✅ 共享     |
| 写锁（WriteLock） | 只能一个线程持有（写操作） | ❌ 独占     |

核心规则：

> - 多个读锁可以并存；
> - 写锁与所有其他锁（包括读锁）互斥。

也就是说：

- 写锁没释放前，读锁不能进；
- 有读锁存在时，写锁也不能进。

------

## 🧠 二、AQS 中的 state 状态设计

在 `ReentrantReadWriteLock.Sync`（AQS 子类）中：

```java
// AQS state 的高16位表示读锁持有计数
// 低16位表示写锁持有计数
state = (readCount << 16) | writeCount;
```

- 当线程加读锁时 → `readCount++`
- 当线程加写锁时 → `writeCount++`

### 读写锁的加锁条件

| 操作     | 条件                                                         |
| -------- | ------------------------------------------------------------ |
| 获取读锁 | 当前没有其他线程持有写锁（`writeCount == 0` 或者 当前线程自己持有写锁可重入） |
| 获取写锁 | 必须没有任何线程持有读锁（`readCount == 0`），且没有其他线程持有写锁 |

------

## ⚙️ 三、那为什么“读锁 → 写锁”会阻塞？

假设场景如下：

```java
ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
lock.readLock().lock();  // 持有读锁
lock.writeLock().lock(); // 尝试升级为写锁
```

### 执行流程：

1. 当前线程成功加了 **读锁**；

   - state 变化：`readCount = 1, writeCount = 0`

2. 再次尝试加 **写锁**；

   - AQS 内部判断：

     ```java
     if (readCount != 0 && currentThread != exclusiveOwnerThread)
         return false; // 加锁失败
     ```

3. 由于当前有“读锁计数 ≠ 0”，且“还不是持有写锁的线程”，
    所以写锁获取失败，被 **阻塞**。

------

## 🧩 四、为什么不能“自动升级”？

从设计上讲，“读锁 → 写锁升级”是**危险的**：

### ⚠️ 1. 会引发死锁风险

设想有两个线程：

```
Thread A：持有读锁 → 想升级为写锁
Thread B：也持有读锁 → 想升级为写锁
```

此时：

- 两个线程都持有读锁；
- 都在等待对方释放读锁才能获得写锁；
- 于是互相卡住 → **死锁**。

所以 Java 的设计者**故意不允许读锁升级为写锁**。

------

## ✅ 五、反过来：写锁 → 读锁是可以的（降级）

```java
lock.writeLock().lock();
try {
    // 写操作
    lock.readLock().lock(); // ✅ 允许降级
} finally {
    lock.writeLock().unlock();
    lock.readLock().unlock();
}
```

因为：

- 当前线程已经独占写锁；
- 没有其他线程能同时访问；
- 持有写锁的线程再加读锁是安全的；
- state 在内部标记当前线程是“exclusiveOwnerThread”，允许重入。

所以**降级允许**，**升级不允许**。

------

## 🧩 六、总结对比表

| 操作                | 是否允许 | 原因                     |
| ------------------- | -------- | ------------------------ |
| 写锁 → 读锁（降级） | ✅ 允许   | 当前线程已独占锁，无冲突 |
| 读锁 → 写锁（升级） | ❌ 阻塞   | 存在潜在死锁风险         |
| 多个读锁并存        | ✅        | 共享模式，不互斥         |
| 写锁与读锁并存      | ❌        | 写锁需要独占             |

------

## 🧠 七、AQS 源码关键片段（读锁 → 写锁阻塞）

在 `ReentrantReadWriteLock.Sync.tryAcquire()` 中：

```java
if (readCount != 0 && 
    (exclusiveOwnerThread != currentThread)) {
    // 有读锁且不是当前线程的写锁 -> 获取失败
    return false;
}
```

所以当前线程即使持有读锁，只要不是“写锁持有者”，就不能再拿写锁。

------

## ✅ 八、一句话总结

> `ReentrantReadWriteLock` 不允许 **读锁升级为写锁**，
>  因为这会破坏“写锁独占”的原则，并可能导致死锁。
>
> 但允许 **写锁降级为读锁**，因为写锁线程已经独占整个资源，降级不影响安全性。
