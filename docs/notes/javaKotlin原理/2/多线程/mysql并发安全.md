多条sql语句发给mysql，mysql是串行的执行吗？

非常好的问题 👍。
 这涉及到 MySQL 的 **语句执行模型** 和 **事务执行机制**。我们分情况说明👇

------

## 🧩 一、同一个连接（Connection）内多条 SQL，是串行执行的 ✅

如果你在 **一个连接（同一个线程）** 里连续执行多条 SQL，比如：

```sql
UPDATE user SET age = age + 1 WHERE id = 1;
UPDATE user SET name = 'Tom' WHERE id = 2;
SELECT * FROM user;
```

> 那么这些 SQL 是 **严格串行执行的**。

MySQL Server 的执行模型是单线程处理单连接（除非你在一个语句内部使用并行执行优化，例如并行查询）。
 也就是说：

- 每条 SQL 语句都必须执行完成，MySQL 才会处理下一条；
- **同一个连接内部不会并发执行多条语句。**

------

## 🧱 二、不同连接（Connection）内的 SQL，可以并行执行 ✅

比如：

- 线程 A 执行：`UPDATE user SET balance = balance - 100 WHERE id = 1;`
- 线程 B 执行：`UPDATE user SET balance = balance + 100 WHERE id = 2;`

这是两个不同连接（两个 session），MySQL 的服务器线程模型是 **每个连接一个线程**，
 因此 A、B 两个线程可以在数据库层面**并发执行**。

------

## ⚙️ 三、事务语境下的执行顺序

在同一个事务中：

```sql
START TRANSACTION;
UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;
COMMIT;
```

MySQL 一定按顺序执行两条 `UPDATE`，只有执行完毕后才提交。
 这也是为什么在事务中可以保证**原子性与一致性（ACID）**。

------

## 🔄 四、批量执行 vs 串行执行区别

你可能见过这种：

```sql
INSERT INTO user (name, age) VALUES ('Tom', 20), ('Jerry', 21), ('Lucy', 19);
```

这也是一次执行，但实际上是 **一次语句执行多行写入**。
 MySQL 会在 **存储引擎层（InnoDB）** 批量插入，仍属于一次“语句执行”，
 不是多条 SQL 并发。

------

## ⚡ 五、特例：多语句查询（multi-statements）

JDBC 或 MySQL 命令行中允许你写：

```sql
UPDATE user SET age = age + 1 WHERE id = 1;
UPDATE user SET age = age + 2 WHERE id = 2;
```

如果客户端开启了 `allowMultiQueries=true`，
 MySQL 会**顺序执行每一条**，依然是串行，不会并发。

------

## 🧠 六、总结一览表

| 场景                           | 是否串行   | 说明                 |
| ------------------------------ | ---------- | -------------------- |
| 同一连接执行多条 SQL           | ✅ 串行     | MySQL 单线程顺序执行 |
| 同一事务中多条 SQL             | ✅ 串行     | 确保事务一致性       |
| 多个连接同时执行 SQL           | ❌ 可并行   | 每个连接独立线程     |
| 一条批量 SQL（多行 INSERT）    | ✅ 单语句   | 一次执行多个值       |
| `allowMultiQueries` 执行多语句 | ✅ 顺序执行 | 不会并发             |

------

## 📌 七、补充：为什么不并行？

MySQL Server 层对每个连接采用 **单线程模型**：

- 每个连接对应一个独立的 `THD`（线程处理对象）；
- SQL 执行计划的生成、优化、执行都在该线程内完成；
- 因此同一连接无法同时跑多条语句。

除非用 **多线程查询优化（parallel query）**，但那属于引擎内部优化（InnoDB 8.0+ 才部分支持）。

------

✅ **一句话总结：**

> MySQL 对同一个连接（Session）的多条 SQL 是串行执行的；
>  只有不同连接（Session）之间才能并行执行。

不同连接操作同一个表有并发安全吗？

非常好 👍，你这个问题问到了 MySQL 并发控制的核心 ——
 **“不同连接（不同线程）同时操作同一张表，是否安全？”**

答案是：

> ✅ **安全的，但性能与结果取决于事务隔离级别和锁机制。**

下面我详细讲清楚它是如何保证“安全”的，以及有哪些隐含问题需要注意👇

------

## 🧱 一、先给出结论总结

| 情况                   | 是否线程安全                   | 原因                                        |
| ---------------------- | ------------------------------ | ------------------------------------------- |
| 多连接 **查询** 同一表 | ✅ 安全                         | 查询不会修改数据                            |
| 多连接 **写入不同行**  | ✅ 安全                         | InnoDB 的行级锁隔离不同数据行               |
| 多连接 **写入同一行**  | ⚠️ 可能竞争                     | InnoDB 会加行锁，一个事务等待另一个事务释放 |
| 不同连接混合读写       | ✅ 一般安全，但结果依赖隔离级别 | 事务隔离决定读写可见性                      |

------

## 🧩 二、MySQL 为什么能做到“并发安全”

核心依靠两个机制：

### 1️⃣ **事务（Transaction）**

保证原子性、一致性、隔离性、持久性（ACID）
 —— 即便多个连接同时修改数据，也能保证结果“逻辑一致”。

### 2️⃣ **锁机制（Lock）**

InnoDB 在并发操作时通过锁粒度控制访问冲突：

- **行级锁**：锁定具体行，适用于大多数事务（并发度高）；
- **表级锁**：MyISAM 使用（并发差）；
- **意向锁**：用来加速行锁检测。

------

## ⚙️ 三、不同连接操作的几种典型情况

### ✅ 情况 1：并发读（SELECT）

多个连接执行：

```sql
SELECT * FROM user WHERE id = 1;
```

👉 完全并发执行，**不会加锁**。
 除非你显式使用：

```sql
SELECT ... FOR UPDATE;
```

这时就会加行锁。

------

### ✅ 情况 2：并发写不同的行（常见、安全）

```sql
-- 连接1
UPDATE user SET balance = balance - 100 WHERE id = 1;

-- 连接2
UPDATE user SET balance = balance - 200 WHERE id = 2;
```

👉 InnoDB 使用 **行锁**：

- 连接1 锁定 id=1；
- 连接2 锁定 id=2；
- 两者互不干扰，可并发执行。
   非常安全，也是高并发业务常见场景。

------

### ⚠️ 情况 3：并发写同一行（会阻塞）

```sql
-- 连接1
UPDATE user SET balance = balance - 100 WHERE id = 1;

-- 连接2
UPDATE user SET balance = balance - 200 WHERE id = 1;
```

👉 InnoDB 检测到同一行，会：

- 连接1 先加行锁；
- 连接2 被阻塞（等待锁释放）；
- 当连接1 提交/回滚后，连接2 才能执行。

💡 因此数据不会乱，**但性能会受影响**。

------

### ⚠️ 情况 4：并发读写同一行

```sql
-- 连接1
UPDATE user SET balance = balance - 100 WHERE id = 1;

-- 连接2
SELECT balance FROM user WHERE id = 1;
```

结果取决于事务隔离级别（InnoDB 默认为 `REPEATABLE READ`）：

| 隔离级别         | 读到的值                     |
| ---------------- | ---------------------------- |
| READ UNCOMMITTED | 能读到未提交的修改（脏读❌）  |
| READ COMMITTED   | 只能读到已提交的值           |
| REPEATABLE READ  | 保证事务期间读到的结果一致   |
| SERIALIZABLE     | 所有并发转为串行（性能最差） |

------

## 🧠 四、InnoDB 如何保证并发安全

### 1️⃣ **行锁**

- 精确锁定修改的那一行；
- 实现高并发写入。

### 2️⃣ **MVCC（多版本并发控制）**

- 用于读操作；
- 每个事务看到的“数据版本”由事务快照决定；
- 这样读不阻塞写、写不阻塞读（大多数情况）。

例如：

```sql
-- 连接1开始事务
BEGIN;
SELECT * FROM user WHERE id=1;  -- 读到 balance=100

-- 连接2更新并提交
UPDATE user SET balance=200 WHERE id=1; COMMIT;

-- 连接1再次读
SELECT * FROM user WHERE id=1;  -- 仍然读到 balance=100 (repeatable read)
```

👉 这就是 MVCC 实现的“可重复读”。

------

## 🧩 五、并发安全 ≠ 无锁竞争

要记住：

> MySQL 通过锁和 MVCC 保证并发安全，
>  但不能消除“性能竞争”——仍可能有等待、阻塞。

例如高并发秒杀系统下：

- 同一商品库存字段频繁更新；
- 行锁争抢严重；
- 性能急剧下降。

解决思路：

- 分库分表；
- 乐观锁（版本号 / CAS 更新）；
- 异步队列削峰；
- Redis 预扣库存。

------

## 🧾 六、一句话总结

| 场景                   | 是否并发安全 | MySQL 机制     |
| ---------------------- | ------------ | -------------- |
| 不同连接读同表         | ✅            | MVCC（快照读） |
| 不同连接写不同记录     | ✅            | 行级锁         |
| 不同连接写同一记录     | ✅（但阻塞）  | 行锁等待       |
| 并发读写同记录         | ✅            | MVCC + 行锁    |
| 不同隔离级别下的可见性 | ⚠️            | 依赖事务级别   |

------

✅ **一句话记忆：**

> MySQL 是并发安全的 —— 它通过「事务 + 锁 + MVCC」保证数据一致，
>  但不是“无阻塞”的，热点行更新仍可能引发性能瓶颈。
