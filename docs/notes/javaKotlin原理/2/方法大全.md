String.equalsIgnoreCase这是什么方法
`String.equalsIgnoreCase` 是 Java 提供的**忽略大小写**的字符串比较方法：

**签名**  
```java
public boolean equalsIgnoreCase(String anotherString)
```

**作用**  
- 仅比较字符内容，**不区分大小写**  
- 底层按字符逐个 **Unicode 大小写折叠**（Case Folding），对英文字母、部分西欧字符有效；对中文、符号无影响  
- 返回 `true` 表示两字符串在忽略大小写的前提下**内容相同**

**示例**  
```java
"Java".equalsIgnoreCase("java");   // true
"Hello".equalsIgnoreCase("HELLO"); // true
"abc".equalsIgnoreCase("AbC");     // true
```

**注意**  
- 与 `equals()` 一样，**null 会抛 `NullPointerException`**  
- 比 `equals()` 稍慢，因为需要额外的大小写转换

**一句话**：  
`equalsIgnoreCase` = **忽略大小写的字符串相等判断**。

computeIfAbsent这是什么方法
`computeIfAbsent` 是 Java 8 给 `java.util.Map` 接口新增的**“懒人初始化”**方法：

**签名**  
```java
V computeIfAbsent(K key, Function<? super K, ? extends V> mappingFunction)
```

**作用一句话**  
> “如果 `key` 当前没有映射（或映射为 `null`），就**自动调用函数**生成一个值并放入 Map，然后返回这个值；否则直接返回已存在的值。”

**典型用法：线程安全的“Map 版单例”**
```java
Map<String, List<String>> map = new HashMap<>();

// 第一次遇到 key 时自动 new ArrayList
List<String> list = map.computeIfAbsent("en", k -> new ArrayList<>());
list.add("Java");

// 第二次直接返回同一个 List
map.computeIfAbsent("en", k -> new ArrayList<>()).add("Spring");
System.out.println(map); // {en=[Java, Spring]}
```

**好处**  
- 省去 `if (!map.containsKey(...)) map.put(...)` 的模板代码  
- **原子语义**：对 `ConcurrentHashMap` 是线程安全的“检查再更新”

**注意**  
- 函数只在**缺失时执行一次**，返回值不能为 `null`（否则抛 `NullPointerException`）  
- 性能优于先 `get` 再 `put`，因为内部只一次哈希计算

**一句话**：  
`computeIfAbsent` = **“Map 懒人初始化器”**，不存在就计算并放入，存在直接返回，常用于缓存、分组收集器。

Objects.requireNonNull这什么方法

`Objects.requireNonNull(...)` 是 Java 7 开始提供的 **“快速空指针检查”** 语法糖，一句话作用：

> **如果传入的对象是 null，立即扔出 NullPointerException；否则把原对象原样返回。**

所在包  
`java.util.Objects`

常用重载  
```java
public static <T> T requireNonNull(T obj)
public static <T> T requireNonNull(T obj, String message)
public static <T> T requireNonNull(T obj, Supplier<String> messageSupplier)
```

典型用法  
1. 构造器 / setter **主动防御**  
   ```java
   public User(String name) {
       this.name = Objects.requireNonNull(name, "name must not be null");
   }
   ```

2. 方法入口 **快速失败**（Fail-fast）  
   ```java
   public void process(File f) {
       Objects.requireNonNull(f, "file");
       // 下面放心用 f，再不会出现莫名其妙的 NPE
   }
   ```

3. 结合 `assert` 式编程（返回对象可链式调用）  
   ```java
   return Objects.requireNonNull(map.get(key), () -> "missing key: " + key);
   ```

好处  
- 比手写 `if (obj == null) throw ...` 更简洁；  
- 异常栈顶就是 **业务代码位置**，调试时一眼定位；  
- 语义清晰：**“此处 null 是非法值，立即爆炸”**。

因此，`Objects.requireNonNull` 不是“放”什么，而是 **“空值守门员”**：**null 进，NPE 出；非 null 进，原样返回。**

classLoader.getResources这个方法是什么
`ClassLoader.getResources(String name)` 是 Java 标准库提供的一个**“枚举类路径下所有同名资源”**的底层方法。

一句话作用  
> **在当前 ClassLoader 的搜索范围内，找出 **所有** 叫 `name` 的资源文件（非类），并以 `Enumeration<URL>` 形式返回它们的物理位置。**

所在位置  
`java.lang.ClassLoader`

方法签名  
```java
public Enumeration<URL> getResources(String name) throws IOException
```

搜索范围  
1. 父加载器优先（双亲委派）  
2. 包含：  
   - classpath 目录  
   - jar 包根目录  
   - 模块路径（JDK 9+）  
   - 自定义 `URLClassPath` 等  

典型输出  
当资源出现多份时，不会合并，而是**逐个列出**。  
例：  
```
file:/C:/proj/conf/logback.xml  
jar:file:/C:/lib/shared.jar!/logback.xml  
```

与 `getResource(String)` 的区别  
- `getResource` 只返回**第一份**（找到即停）  
- `getResources` 返回**全部**（可用来检测冲突、加载同名配置多份）

代码示例  
```java
Enumeration<URL> urls = Thread.currentThread()
                               .getContextClassLoader()
                               .getResources("META-INF/spring.factories");
while (urls.hasMoreElements()) {
    URL url = urls.nextElement();
    System.out.println(url);
}
```

常见用途  
- Spring、Dubbo 等框架扫描 `META-INF/spring.*`、`META-INF/services/*`  
- 自己写插件系统，需要把同名配置文件全部读进来合并  
- 诊断 “classpath 上到底有几份同名 xml/properties”

注意  
- 返回的是 **只读 URL**，想读内容再用 `url.openStream()`  
- 如果没有任何匹配，返回**空 Enumeration**（不是 null）  
- 父加载器抛出的 `IOException` 会原样传递

一句话记：  
`getResource` 找第一份，`getResources` 把**整棵类加载树里所有同名资源**全部揪出来。
<div style="height:10px; background:linear-gradient(to right, #ff6a00, #ee0979, #00c6ff); border:none;"></div>
URLConnection urlConnection = url.openConnection();
                urlConnection.setUseCaches(false);
这两个代码是干啥的
两行代码干的事一句话就能说清：

1. `url.openConnection()`  
   把抽象的 `URL` 变成“真正准备传输数据”的**连接对象**（`HttpURLConnection`、`JarURLConnection`、`FileURLConnection` … 具体类型由协议决定）。

2. `setUseCaches(false)`  
   告诉协议处理器：**“别用任何本地缓存，每次都去源头重新拿数据。”**  
   对 `http/https` 来说就是**禁用客户端缓存**（忽略 `If-Modified-Since` / `If-None-Match` 等条件请求头）；对 `jar:file/!/` 就是**不解压到临时目录缓存**；对 `file://` 基本无影响。

典型用途  
- 强制拉取最新配置、快照版 Jar、动态脚本等  
- 调试时避免“老数据”干扰  
- 与 `setDefaultUseCaches(false)` 相比，这里只对**当前连接实例**生效。

一句话记：  
`openConnection()` 把“地址”变“通道”；`setUseCaches(false)` 给通道贴张条——**“不准偷懒，必须到源头重新取。”**

Class.isAssignableFrom这是什么方法
`Class.isAssignableFrom` 是 Java 反射库提供的 **“类型兼容判定”** 工具方法：

一句话作用  
> **判断“我（左边类/接口）能否安全接收你（右边类）的实例”**，即 **右边类型是不是左边类型的子类型（或本身）**。

方法签名  
```java
public boolean isAssignableFrom(Class<?> cls)
```

调用格式  
```java
boolean yes = 左边.class.isAssignableFrom(右边.class);
```

语义对照表  

| 调用示例 | 结果 | 等价代码 |
|----------|------|----------|
| `CharSequence.class.isAssignableFrom(String.class)` | true | `String instanceof CharSequence` |
| `Number.class.isAssignableFrom(Integer.class)` | true | `Integer instanceof Number` |
| `List.class.isAssignableFrom(ArrayList.class)` | true | `ArrayList instanceof List` |
| `ArrayList.class.isAssignableFrom(List.class)` | false | 反过来不行 |
| `int.class.isAssignableFrom(int.class)` | true | 基本类型只认自己 |
| `int.class.isAssignableFrom(Integer.class)` | false | 自动装箱不会算 |

与 `instanceof` 的区别  
- `instanceof` 运行时看**对象**；`isAssignableFrom` 编译期/反射看**Class 对象**。  
- 无须实例，可判接口、数组、基本类型。

典型用途  
1. 通用框架做**类型检查**  
   ```java
   if (MyPlugin.class.isAssignableFrom(clazz)) { … }
   ```
2. 避免强转异常  
   ```java
   if (expectedType.isAssignableFrom(value.getClass())) {
       return expectedType.cast(value);
   }
   ```
3. 替代 `instanceof` 写工具类时更灵活（没有实例也能判）。

口诀  
**“左边能否装右边，反射里面用它看。”**

用不同的类加载器加载的兼容的类也会返回false吧
是的，**不同类加载器加载的“同名全限定类”在 JVM 眼里就是两个完全不同的类型**，即使字节码一模一样，也满足不了赋值兼容，因此：

```java
Class<?>  a = loaderA.loadClass("com.foo.User");
Class<?>  b = loaderB.loadClass("com.foo.User");

a.isAssignableFrom(b)   // 一定返回 false
```

原因  
- `isAssignableFrom` 只比较 **运行时类型标识（Klass*）**，不比较字节码内容。  
- JVM 规范规定：**类全名 + 加载器 一起决定类型唯一性**。  
- 因此 `loaderA::User` 与 `loaderB::User` 互为**不相关类型**，互相 cast 会抛 `ClassCastException`，自然 `isAssignableFrom` 也返回 false。

结论  
用 `isAssignableFrom` 做类型兼容性判断时，**必须保证两个 Class 对象来自同一类加载器（或父子委派链上的同一节点）**，否则即使“代码一样”也会得到 false。
