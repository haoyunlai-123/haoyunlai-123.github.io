介绍下责任链模式并举几个例子

非常好 👍
 你现在已经掌握了装饰者、适配器、代理、观察者、享元等模式，
 接下来这个 —— **责任链模式（Chain of Responsibility Pattern）**
 属于行为型设计模式中非常重要、在框架中出现频率极高的一种。

------

## 🧩 一、核心思想

> **将多个处理对象串成一条链，请求在链上依次传递，直到某个处理器处理为止。**
>
> 每个处理器只关心自己能不能处理，不能处理就交给下一个。

这样可以让**请求的发送者与接收者解耦**，
 非常适合处理“有序过滤、审批流、拦截链”一类的场景。

------

## ⚙️ 二、角色组成

| 角色                              | 说明                                         |
| --------------------------------- | -------------------------------------------- |
| **Handler（抽象处理者）**         | 定义处理请求的接口，并持有下一个处理者的引用 |
| **ConcreteHandler（具体处理者）** | 实现具体的处理逻辑，决定是否处理或传递       |
| **Client（客户端）**              | 创建责任链并发送请求                         |

------

## 💡 三、示例 1：审批流程（经典例子）

假设公司请假流程如下：

- 普通员工请假 → 主管审批；
- 超过 3 天 → 经理审批；
- 超过 7 天 → 总经理审批。

```java
// 抽象处理者
abstract class Handler {
    protected Handler next;
    public void setNext(Handler next) { this.next = next; }
    public abstract void handleRequest(int days);
}

// 具体处理者
class Supervisor extends Handler {
    public void handleRequest(int days) {
        if (days <= 3)
            System.out.println("主管审批通过");
        else if (next != null)
            next.handleRequest(days);
    }
}

class Manager extends Handler {
    public void handleRequest(int days) {
        if (days <= 7)
            System.out.println("经理审批通过");
        else if (next != null)
            next.handleRequest(days);
    }
}

class Director extends Handler {
    public void handleRequest(int days) {
        if (days > 7)
            System.out.println("总经理审批通过");
    }
}

// 客户端
public class Main {
    public static void main(String[] args) {
        Handler supervisor = new Supervisor();
        Handler manager = new Manager();
        Handler director = new Director();

        supervisor.setNext(manager);
        manager.setNext(director);

        supervisor.handleRequest(2);
        supervisor.handleRequest(5);
        supervisor.handleRequest(10);
    }
}
```

**输出：**

```
主管审批通过
经理审批通过
总经理审批通过
```

👉 责任链让请求逐级传递，调用者只需发起一次请求。

## 💻 四、示例 2：Web 请求过滤链（现实框架原型）

这是在 **Servlet Filter / Spring Interceptor / Netty Pipeline** 中最常见的应用。

```java
interface Filter {
    void doFilter(Request req, Response res, FilterChain chain);
}

class Request { String msg; }
class Response { String msg; }

class FilterChain {
    private List<Filter> filters = new ArrayList<>();
    private int index = 0;

    public FilterChain add(Filter f) { filters.add(f); return this; }

    public void doFilter(Request req, Response res) {
        if (index == filters.size()) return;
        filters.get(index++).doFilter(req, res, this);
    }
}

// 具体过滤器
class AuthFilter implements Filter {
    public void doFilter(Request req, Response res, FilterChain chain) {
        System.out.println("鉴权通过");
        chain.doFilter(req, res);
    }
}

class LogFilter implements Filter {
    public void doFilter(Request req, Response res, FilterChain chain) {
        System.out.println("记录日志");
        chain.doFilter(req, res);
    }
}

public class Demo {
    public static void main(String[] args) {
        Request req = new Request();
        Response res = new Response();

        FilterChain chain = new FilterChain()
                .add(new AuthFilter())
                .add(new LogFilter());

        chain.doFilter(req, res);
    }
}
```

**输出：**

```
鉴权通过
记录日志
```

✅ 完全模拟了 Servlet Filter 的原理：
 请求从第一个过滤器进入，沿链传递，每个过滤器都能在调用前后处理逻辑。

## 🧠 五、示例 3：异常处理链

在系统中有时需要根据异常类型由不同模块处理：

```java
abstract class ErrorHandler {
    protected ErrorHandler next;
    public void setNext(ErrorHandler next) { this.next = next; }
    public abstract void handle(Exception e);
}

class NullPointerHandler extends ErrorHandler {
    public void handle(Exception e) {
        if (e instanceof NullPointerException)
            System.out.println("处理空指针异常");
        else if (next != null)
            next.handle(e);
    }
}

class IOHandler extends ErrorHandler {
    public void handle(Exception e) {
        if (e instanceof IOException)
            System.out.println("处理IO异常");
        else if (next != null)
            next.handle(e);
    }
}

public class App {
    public static void main(String[] args) {
        ErrorHandler npHandler = new NullPointerHandler();
        ErrorHandler ioHandler = new IOHandler();
        npHandler.setNext(ioHandler);

        npHandler.handle(new IOException("磁盘错误"));
    }
}
```

**输出：**

```
处理IO异常
```

## 🧠 七、优缺点总结

**✅ 优点**

- 解耦请求发送者与处理者；
- 动态组合责任链；
- 易于扩展新的处理节点。

**❌ 缺点**

- 请求可能在链中遍历多个节点，性能有损耗；
- 调试较难；
- 如果链过长或逻辑不当，可能导致循环调用。

------

## 🧾 八、一句话记忆

> **责任链模式：多个处理者依次接力，请求沿链传递直到被处理。**

## ✅ 十、总结

**结构要点：**

```text
Client → Handler1 → Handler2 → Handler3 ...
```

**关键思想：**

- 每个处理器只处理自己关心的部分；
- 不能处理就交给下一个；
- 可灵活增删节点，无需修改核心逻辑。

非常棒的问题 👍——你已经不只是会用责任链模式，而是在**从 JVM 调用栈层面思考它的运行代价**了，这正是高级工程师的思维。

你提到的：

> “`filters.get(index++).doFilter(req, res, this);` 每调一个过滤器都产生一个新的栈帧，不怕 StackOverflow 吗？”

这个问题非常关键。答案是：

------

## ✅ 一、是的，每经过一个过滤器，确实会多一个栈帧

你看这段调用：

```java
filters.get(index++).doFilter(req, res, this);
```

这是一个**递归链式调用**（虽然逻辑上是链，但实现上是递归结构）。

每进入一个过滤器方法（`doFilter`），JVM 就会为该方法分配一个新的**栈帧（stack frame）**，包括局部变量表、操作数栈、返回地址等。

比如：

```
main() 
 └─ chain.doFilter()
     ├─ filter1.doFilter()
     │   ├─ filter2.doFilter()
     │   │   ├─ filter3.doFilter()
     │   │   │   └─ ...
```

每个过滤器调用 `chain.doFilter()` 时，都会**压栈一次**，执行完毕后再弹栈。

------

## 🧠 二、为什么在实际应用中不会 StackOverflow

虽然每次都压栈，但通常：

- 过滤器数量是**有限且可控的**（几十个以内）；
- 每个方法调用栈帧消耗的内存也较小（几十字节~几百字节）；
- JVM 默认栈深可达几千层甚至上万层（`-Xss` 参数可调，默认 1M 左右栈空间）。

### 举例计算

假设每个 `doFilter()` 调用栈帧占用 512 字节：

- 1MB 栈空间可容纳 ≈ 2000 层；
- Web 应用中 Filter/Interceptor 数量一般 < 20；
- 所以完全安全，不会 StackOverflow。

------

## ⚙️ 三、如果真要防止栈过深怎么办？

你提到的担忧在“极深链式结构”（例如上千节点）时确实会出现。
 解决办法通常有两种：

### ✅ 方案 1：改为循环结构（显式索引）

在 Servlet Filter 源码中，确实就是用递归方式调用的，但你可以改为**迭代式实现**：

```java
class FilterChain {
    private List<Filter> filters = new ArrayList<>();

    public void doFilter(Request req, Response res) {
        for (Filter f : filters) {
            f.doFilter(req, res, this);
        }
    }
}
```

当然这样就失去了“前后调用嵌套”的灵活性（不能 easily 在调用前后分别处理）。

------

### ✅ 方案 2：使用显式栈模拟（避免 JVM 递归栈）

如果链极长，可以自己维护一个**手动栈结构**，将调用展开为循环执行：

```java
class SafeChain {
    private List<Filter> filters = new ArrayList<>();

    public void doFilter(Request req, Response res) {
        Deque<Integer> stack = new ArrayDeque<>();
        int index = 0;

        while (index < filters.size()) {
            filters.get(index).doFilter(req, res, this);
            stack.push(index);
            index++;
        }

        // 如果需要反向调用（如责任链回调）
        while (!stack.isEmpty()) {
            int i = stack.pop();
            // 后置逻辑 ...
        }
    }
}
```

这种写法能在极端情况下防止 StackOverflow，但一般框架不会这样做，因为性能和代码复杂度都更高。

------

## 💻 四、现实框架的处理方式

| 框架                      | 是否递归               | 是否担心 StackOverflow  |
| ------------------------- | ---------------------- | ----------------------- |
| **Servlet FilterChain**   | ✅ 是递归               | ❌ 不会，Filter 数量有限 |
| **Spring Interceptor**    | ✅ 是递归               | ❌ 不会                  |
| **Netty ChannelPipeline** | ✅ 是递归调用，但有优化 | ❌ Handler 数量少        |
| **MyBatis Plugin Chain**  | ✅ 递归                 | ❌ 插件层数有限          |
| **Dubbo Filter**          | ✅ 递归                 | ❌ 通常 < 10 个过滤器    |

🧩 这些框架都采用递归方式，因为：

1. 代码简单；
2. 层级有限；
3. 便于“前置逻辑 + 调用链 + 后置逻辑”嵌套执行（AOP 风格）。

------

## 🔍 五、为什么递归对责任链非常自然

责任链模式通常需要**“调用前逻辑 + 继续调用下一个 + 调用后逻辑”**的结构，例如：

```java
void doFilter(...) {
    before();
    chain.doFilter(...); // 调用下一个
    after();
}
```

要保持这种嵌套语义（类似 try/finally），递归是最自然的实现方式。
 如果改成循环，就很难优雅地实现这种“包裹式”逻辑。

------

## 🧩 六、总结

| 问题                         | 答案                                   |
| ---------------------------- | -------------------------------------- |
| 每个过滤器会不会产生新栈帧？ | ✅ 会，每个 `doFilter()` 调用都压栈一次 |
| 会不会 StackOverflow？       | ❌ 一般不会（调用层数几十级以内）       |
| 框架如何处理？               | 多采用递归结构（简单且逻辑清晰）       |
| 若链极长怎么办？             | 可改为显式循环或分段处理               |

------

## ✅ 一句话总结

> 责任链模式用递归结构确实会逐层压栈，但由于处理节点有限，
>  在 Web / 框架场景下远未到栈溢出的程度。
>  若节点成千上万，可用显式循环或栈结构优化。
