介绍下桥接模式并举几个例子

非常好 👍！你现在已经进入到 **结构型设计模式** 中的进阶部分了，
今天讲的 **桥接模式（Bridge Pattern）** 是这类模式里最抽象、但实际项目中**非常实用**的一个。

它经常用于：

* UI 组件层（如 Swing、Android View、跨平台渲染）
* JDBC 驱动、日志系统、消息发送等 **多维度扩展** 的场景。

---

## 🧩 一、概念简介

**桥接模式（Bridge Pattern）**
是一种用于“**分离抽象与实现，使它们可以独立变化**”的设计模式。

> 通俗讲：把两个容易变化的维度拆开，让它们通过“桥”连接。
> 以后无论哪一边变化，都不需要改另一边的代码。

---

### 🧠 换成人话举例：

假设你要实现不同形状在不同平台绘制：

| 形状        | 平台                  |
| ----------- | --------------------- |
| 圆形 / 矩形 | Windows / Mac / Linux |

如果直接继承，就要写：

```
WindowsCircle
MacCircle
WindowsRectangle
MacRectangle
...
```

→ 类爆炸。

桥接模式的思路是：

> 把“形状”抽象出来，
> 把“平台”单独做成实现接口，
> 两者通过“桥（Bridge）”关联。

于是：

```
Shape（抽象） → 有一个 DrawAPI（实现接口）
```

---

## 🧱 二、模式结构图

```
        ┌───────────────────┐
        │   Abstraction     │ ← 抽象类（如 Shape）
        └───────┬───────────┘
                │  依赖实现接口
        ┌───────▼───────────┐
        │ Implementor       │ ← 实现接口（如 DrawAPI）
        └───────┬───────────┘
                │
     ┌──────────▼──────────┐
     │ ConcreteImplementorA │ ← 实现类（WindowsDraw）
     └──────────────────────┘
     ┌──────────▼──────────┐
     │ ConcreteImplementorB │ ← 实现类（MacDraw）
     └──────────────────────┘
```

## ⚙️ 三、代码示例一：图形绘制（经典版本）

```java
// 实现接口
interface DrawAPI {
    void drawCircle(int x, int y, int radius);
}

// 两个不同平台的实现
class WindowsDrawAPI implements DrawAPI {
    public void drawCircle(int x, int y, int radius) {
        System.out.printf("Drawing Circle on Windows at (%d,%d) radius=%d%n", x, y, radius);
    }
}

class MacDrawAPI implements DrawAPI {
    public void drawCircle(int x, int y, int radius) {
        System.out.printf("Drawing Circle on Mac at (%d,%d) radius=%d%n", x, y, radius);
    }
}

// 抽象类（桥梁）—— 持有实现接口
abstract class Shape {
    protected DrawAPI drawAPI;
    protected Shape(DrawAPI drawAPI) {
        this.drawAPI = drawAPI;
    }
    abstract void draw(); // 抽象行为
}

// 具体抽象（扩展 Abstraction）
class Circle extends Shape {
    private int x, y, radius;
    public Circle(int x, int y, int radius, DrawAPI drawAPI) {
        super(drawAPI);
        this.x = x; this.y = y; this.radius = radius;
    }
    public void draw() {
        drawAPI.drawCircle(x, y, radius);
    }
}

// 客户端
public class BridgeDemo {
    public static void main(String[] args) {
        Shape winCircle = new Circle(10, 10, 5, new WindowsDrawAPI());
        Shape macCircle = new Circle(20, 20, 8, new MacDrawAPI());

        winCircle.draw();
        macCircle.draw();
    }
}
```

✅ 输出：

```
Drawing Circle on Windows at (10,10) radius=5
Drawing Circle on Mac at (20,20) radius=8
```

📘 **扩展性分析：**

* 想新增一个形状？ → 新建 `Rectangle` 类；
* 想新增一个平台？ → 新建 `LinuxDrawAPI`；
* 两边完全独立，无需修改原有代码。

## 💡 四、代码示例二：消息发送系统（多维变化）

场景：
你有不同类型的消息（普通 / 加急），
又有不同的发送方式（Email / SMS / Push）。

---

### 不使用桥接模式时：

你会写：

```
CommonEmailMessage
UrgentEmailMessage
CommonSMSMessage
UrgentSMSMessage
...
```

类爆炸。

---

### 使用桥接模式：

```java
// 实现接口（发送方式）
interface MessageSender {
    void send(String message);
}

// 具体实现类
class EmailSender implements MessageSender {
    public void send(String message) {
        System.out.println("Email sent: " + message);
    }
}

class SMSSender implements MessageSender {
    public void send(String message) {
        System.out.println("SMS sent: " + message);
    }
}

// 抽象类（消息类型）
abstract class Message {
    protected MessageSender sender;
    protected Message(MessageSender sender) {
        this.sender = sender;
    }
    abstract void sendMessage(String message);
}

// 具体抽象类
class CommonMessage extends Message {
    public CommonMessage(MessageSender sender) { super(sender); }
    public void sendMessage(String message) {
        sender.send("[普通] " + message);
    }
}

class UrgentMessage extends Message {
    public UrgentMessage(MessageSender sender) { super(sender); }
    public void sendMessage(String message) {
        sender.send("[加急] " + message.toUpperCase());
    }
}

// 客户端
public class Client {
    public static void main(String[] args) {
        Message msg1 = new CommonMessage(new EmailSender());
        msg1.sendMessage("会议在3点开始");

        Message msg2 = new UrgentMessage(new SMSSender());
        msg2.sendMessage("服务器宕机");
    }
}
```

✅ 输出：

```
Email sent: [普通] 会议在3点开始
SMS sent: [加急] 服务器宕机
```

## 🧠 五、桥接模式的本质

> 它并不是“建一座桥”，
> 而是通过组合（has-a）而不是继承（is-a）
> 来连接两个可独立变化的层次。

换句话说：

* 抽象部分（Shape / Message）定义“是什么”
* 实现部分（DrawAPI / MessageSender）定义“怎么做”

两边通过组合桥接，互不干扰。

> 🧩 **桥接模式**：
> 通过**组合（has-a）**关系连接“抽象部分”和“实现部分”，
> 使两者能独立扩展、不互相影响。
> 它是解决“多维扩展”问题的终极武器。

## 🧠 一、桥接模式的底层思想：就是“组合优于继承”

在 Java 里，很多模式其实是为了解决“继承带来的刚性问题”：

- 多维变化导致类爆炸；
- 修改父类容易影响子类；
- 无法在运行时动态替换实现。

桥接模式用一条关键策略解决了这些问题：

> 🔸 不通过继承（is-a），而是通过组合（has-a）
>  将抽象部分和实现部分分离。

也就是说：

```java
// ❌ 传统继承
class Circle extends WindowsShape { ... } // 一旦平台扩展就炸裂

// ✅ 桥接（组合）
class Circle {
    private DrawAPI drawAPI; // has-a
}
```

抽象类不再固定依赖于某个实现，而是持有一个“接口型字段”。
 这样——

- 抽象类可以独立扩展；
- 实现类也可以独立扩展；
- 组合关系可以在**运行时动态绑定**。

------

## 🧱 二、桥接模式其实是“结构化组合”的一种固定模板

很多人会误解“桥接 = 任意组合”，其实不对。
 桥接是**有明确分层**的组合。

结构上看：

| 层次   | 角色                         | 含义                             |
| ------ | ---------------------------- | -------------------------------- |
| 抽象层 | Abstraction                  | 定义高层业务接口（如 `Shape`）   |
| 实现层 | Implementor                  | 定义底层实现接口（如 `DrawAPI`） |
| 桥梁   | Abstraction 持有 Implementor | 通过组合“桥接”两层               |

🔹 这就让“组合”不只是松散拼装，而是一个有结构的双向解耦。

## 🧬 三、和 Go 语言对比（非常精彩的一点）

你提到：

> “不就是 Go 中的继承方式吗？”

没错。Go 没有传统意义上的继承（no subclass），
 它靠的是 **结构体嵌入（embedding） + 接口组合**。

这正好就是桥接模式的思想落地。

### 看一个对应例子：

#### Java（桥接模式）

```java
interface DrawAPI {
    void drawCircle(int x, int y, int r);
}

class Circle {
    private DrawAPI drawAPI;
    public Circle(DrawAPI drawAPI) { this.drawAPI = drawAPI; }
    void draw() { drawAPI.drawCircle(10, 10, 5); }
}
```

#### Go（组合 / 桥接）

```go
package main

import "fmt"

// 实现层
type DrawAPI interface {
    DrawCircle(x, y, r int)
}

type WindowsDraw struct{}
func (WindowsDraw) DrawCircle(x, y, r int) {
    fmt.Printf("Drawing Circle on Windows (%d,%d) radius=%d\n", x, y, r)
}

// 抽象层
type Circle struct {
    drawAPI DrawAPI // has-a
}

func (c *Circle) Draw() {
    c.drawAPI.DrawCircle(10, 10, 5)
}

func main() {
    c := &Circle{drawAPI: WindowsDraw{}}
    c.Draw()
}
```

✅ 输出：

```
Drawing Circle on Windows (10,10) radius=5
```

这段 Go 代码就是桥接模式的直接实现：

- `DrawAPI` 是实现接口；
- `Circle` 是抽象层；
- 两者通过组合解耦；
- 后续你可以换 `LinuxDraw`、`MacDraw`，`Circle` 一行代码不改。

------

## 🧩 四、Go 的组合和桥接的哲学共通点

| 概念     | Go                           | 桥接模式             |
| -------- | ---------------------------- | -------------------- |
| 继承机制 | 没有继承（embedding + 接口） | 拒绝继承，改用组合   |
| 扩展方式 | 嵌入新类型实现多态           | 抽象层 + 实现层分离  |
| 多维扩展 | 类型 × 行为自由组合          | 抽象 × 实现自由组合  |
| 优点     | 解耦、灵活、动态绑定         | 解耦、灵活、动态绑定 |

结论就是一句话：

> **Go 天然是“桥接风格”的语言。**

而 Java 需要通过 **桥接模式** 明确地去模拟这种组合思想。

------

## 🧩 五、桥接模式与组合优于继承（Composition over Inheritance）

这句经典原则出自 GoF《设计模式》前言部分，原文：

> “Favor object composition over class inheritance.”

意思是：

> 用对象之间的组合关系来实现复用，而不是依赖类层次继承。

桥接模式就是它最标准的“落地模式”之一。
 另外两个典型代表是：

- **策略模式**（Strategy Pattern）
- **装饰者模式**（Decorator Pattern）

这三个模式本质上都是“组合优于继承”的不同应用场景。

------

## ✅ 六、一句话总结

> **桥接模式 = 把“继承层次”拆成“组合维度”**，
>  是“组合优于继承”思想的典型实现。
>
> 而 **Go 的类型系统** 天生就支持这种组合式结构，
>  所以在 Go 里几乎不需要显式地写桥接模式 ——
>  它的类型系统就是桥接。
