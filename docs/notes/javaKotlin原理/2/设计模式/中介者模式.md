你现在进入到行为型模式中一个非常具有“**解耦精神**”的设计模式 ——
👉 **中介者模式（Mediator Pattern）**。

这个模式在实际开发中**非常实用**，尤其是在：

* GUI 界面组件交互、
* 聊天系统、
* 事件调度、

## 🧩 一、定义与核心思想

**中介者模式（Mediator Pattern）**
用一个中介对象来封装一系列对象之间的交互，
使这些对象不再显式地互相引用，
从而使它们之间的**耦合度降低**，交互关系更加灵活。

> 换句话说：
> 把“多对象之间的复杂交互”交给一个“中介对象”统一协调。

---

### 🧠 一句话理解

> “让对象不再互相聊天，而是都和中介沟通。”
> —— 像微信群：每个人只发消息给群，群来转发给别人。

## ⚙️ 三、示例一：聊天室（最直观例子）

每个用户不直接发消息给另一个用户，
而是通过聊天室（中介者）来转发。

```java
// 抽象中介者
interface ChatMediator {
    void sendMessage(String msg, User user);
    void addUser(User user);
}

// 具体中介者
class ChatRoom implements ChatMediator {
    private List<User> users = new ArrayList<>();

    public void addUser(User user) {
        users.add(user);
    }

    public void sendMessage(String msg, User sender) {
        for (User user : users) {
            if (user != sender) {
                user.receive(msg, sender);
            }
        }
    }
}

// 同事类
abstract class User {
    protected ChatMediator mediator;
    protected String name;

    public User(String name, ChatMediator mediator) {
        this.name = name;
        this.mediator = mediator;
    }

    public abstract void send(String msg);
    public abstract void receive(String msg, User sender);
}

// 具体同事
class ChatUser extends User {
    public ChatUser(String name, ChatMediator mediator) {
        super(name, mediator);
    }

    public void send(String msg) {
        System.out.println(name + " 发送消息: " + msg);
        mediator.sendMessage(msg, this);
    }

    public void receive(String msg, User sender) {
        System.out.println(name + " 收到来自 " + sender.name + " 的消息: " + msg);
    }
}

// 客户端
public class ChatDemo {
    public static void main(String[] args) {
        ChatMediator chatRoom = new ChatRoom();

        User alice = new ChatUser("Alice", chatRoom);
        User bob = new ChatUser("Bob", chatRoom);
        User carol = new ChatUser("Carol", chatRoom);

        chatRoom.addUser(alice);
        chatRoom.addUser(bob);
        chatRoom.addUser(carol);

        alice.send("大家好！");
        bob.send("嗨 Alice！");
    }
}
```

✅ 输出：

```
Alice 发送消息: 大家好！
Bob 收到来自 Alice 的消息: 大家好！
Carol 收到来自 Alice 的消息: 大家好！
Bob 发送消息: 嗨 Alice！
Alice 收到来自 Bob 的消息: 嗨 Alice！
Carol 收到来自 Bob 的消息: 嗨 Alice！
```

💡 这里的 `ChatRoom` 就是典型的“中介者”，
它让每个用户之间不再有直接依赖。

## 💡 四、示例二：GUI 组件交互（非常常见）

在一个登录界面中：

* 输入用户名、密码；
* 点击登录按钮；
* 如果输入为空，按钮应禁用；
* 全部填写后才可点击。

传统写法：各组件互相监听对方变化 → 高耦合。
使用中介者后，所有交互由 `DialogMediator` 统一协调。

```java
// 抽象中介者
interface Mediator {
    void componentChanged(Component c);
}

// 抽象同事类
abstract class Component {
    protected Mediator mediator;
    public void setMediator(Mediator m) { this.mediator = m; }
}

// 具体同事：输入框
class TextBox extends Component {
    private String text = "";
    public void setText(String text) {
        this.text = text;
        mediator.componentChanged(this);
    }
    public String getText() { return text; }
}

// 具体同事：按钮
class Button extends Component {
    private boolean enabled = false;
    public void setEnabled(boolean enabled) {
        this.enabled = enabled;
        System.out.println("按钮状态: " + (enabled ? "可点击" : "禁用"));
    }
}

// 具体中介者
class LoginDialog implements Mediator {
    private TextBox username, password;
    private Button loginButton;

    public LoginDialog(TextBox u, TextBox p, Button b) {
        this.username = u;
        this.password = p;
        this.loginButton = b;
        u.setMediator(this);
        p.setMediator(this);
        b.setMediator(this);
    }

    public void componentChanged(Component c) {
        boolean enable = !username.getText().isEmpty() && !password.getText().isEmpty();
        loginButton.setEnabled(enable);
    }
}

// 客户端
public class GuiDemo {
    public static void main(String[] args) {
        TextBox username = new TextBox();
        TextBox password = new TextBox();
        Button login = new Button();
        new LoginDialog(username, password, login);

        username.setText("Alice");  // 此时按钮仍禁用
        password.setText("123456"); // 输入密码后按钮启用
    }
}
```

✅ 输出：

```
按钮状态: 禁用
按钮状态: 可点击
```

💡 GUI 框架（如 Swing、Qt、Android View）中广泛使用这种模式。
中介者（Dialog、Layout）负责组件通信。

## ⚙️ 五、示例三：飞机塔台调度系统

飞机（同事）之间不直接通信，而由**塔台（中介者）**统一协调：

```java
interface AirTrafficControl {
    void notify(String message, Airplane sender);
}

class ControlTower implements AirTrafficControl {
    private List<Airplane> planes = new ArrayList<>();

    public void register(Airplane p) { planes.add(p); }

    public void notify(String message, Airplane sender) {
        for (Airplane p : planes) {
            if (p != sender) p.receive(message, sender);
        }
    }
}

class Airplane {
    private String name;
    private AirTrafficControl tower;

    public Airplane(String name, AirTrafficControl tower) {
        this.name = name;
        this.tower = tower;
        tower.register(this);
    }

    public void send(String msg) {
        System.out.println(name + " 发消息: " + msg);
        tower.notify(msg, this);
    }

    public void receive(String msg, Airplane sender) {
        System.out.println(name + " 收到 " + sender.name + " 的消息: " + msg);
    }
}

public class AirControlDemo {
    public static void main(String[] args) {
        ControlTower tower = new ControlTower();
        Airplane a1 = new Airplane("MU3101", tower);
        Airplane a2 = new Airplane("CA889", tower);
        Airplane a3 = new Airplane("CZ6001", tower);

        a1.send("准备降落");
        a2.send("跑道占用，请等待");
    }
}
```

✅ 输出：

```
MU3101 发消息: 准备降落
CA889 收到 MU3101 的消息: 准备降落
CZ6001 收到 MU3101 的消息: 准备降落
CA889 发消息: 跑道占用，请等待
MU3101 收到 CA889 的消息: 跑道占用，请等待
CZ6001 收到 CA889 的消息: 跑道占用，请等待
```

💡 `ControlTower`（塔台）是中介者，
所有飞机不再直接通信，从而避免复杂的多对多关系。

## 🧠 六、模式的本质

> **中介者模式的本质是“集中控制、解耦通信”。**
>
> 它把复杂的“网状关系”变成了“星状关系”：

```
Before:
A <-> B
A <-> C
B <-> C  （多对多）

After:
A -> Mediator <- B
            ↑
            C
```

---

## ⚙️ 七、优缺点总结

| 优点         | 说明                                       |
| ------------ | ------------------------------------------ |
| ✅ 降低耦合度 | 同事之间不再直接引用                       |
| ✅ 提高复用性 | 同事类可单独使用                           |
| ✅ 集中控制   | 交互逻辑集中到中介者                       |
| ⚠️ 缺点       | 中介者逻辑过多时可能变得臃肿（“上帝对象”） |

---

## 🧾 八、典型应用场景

| 场景       | 示例                                                         |
| ---------- | ------------------------------------------------------------ |
| GUI 系统   | 对话框组件（Dialog 控制 Button、TextBox）                    |
| 聊天室     | 聊天中介（ChatRoom）转发消息                                 |
| 游戏引擎   | 事件系统 / 场景协调器                                        |
| 航空调度   | 塔台统一指挥飞机                                             |
| 消息中间件 | 消息总线（Message Broker）解耦发布者和订阅者                 |
| Spring     | `ApplicationEventPublisher`、`DispatcherServlet`（都是中介者思想） |

---

## ✅ 九、一句话总结

> 🧩 **中介者模式（Mediator Pattern）**：
> 用一个中介对象封装一组对象的交互，使它们不必显式地相互引用。
> 它让系统的对象之间**从网状依赖变成星状依赖**，大大降低耦合。
>
> 🧠 “让对象都和中介打交道，而不是彼此纠缠。”
