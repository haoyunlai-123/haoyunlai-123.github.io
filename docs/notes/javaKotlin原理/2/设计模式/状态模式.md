介绍下状态模式并举几个例子

非常好 👍！
你现在进入行为型模式中非常经典、也极易与**策略模式**混淆的一个 ——
👉 **状态模式（State Pattern）**。

它和策略模式在结构上很像（都封装行为），
但本质上是解决**对象在不同状态下行为变化的问题**。
—— 换句话说，它是“带状态的策略模式”。

---

## 🧩 一、定义与核心思想

**状态模式（State Pattern）**
允许对象在其内部状态改变时，**改变它的行为**，
看起来就像是**修改了对象的类**一样。

> 换句话说：
> **将状态逻辑从主类中分离出来，每种状态封装成独立类，
> 主对象根据当前状态动态切换行为。**

---

### 🧠 一句话理解

> “对象的行为依赖于它当前所处的状态，
> 并且它可以在运行时改变自己的状态。”

---

## 🧱 二、结构图

```
┌────────────────────┐
│      Context        │ ← 上下文（环境类）
│────────────────────│
│ - state: State      │ 持有当前状态对象
│ + setState(State)   │ 切换状态
│ + handle()          │ 调用状态行为
└─────────┬──────────┘
          │
          ▼
    ┌────────────┐
    │   State    │ ← 状态接口
    └────┬───────┘
         │
 ┌───────▼───────────┐
 │ ConcreteStateA     │ ← 具体状态A
 └────────────────────┘
 ┌───────▼───────────┐
 │ ConcreteStateB     │ ← 具体状态B
 └────────────────────┘
```

## ⚙️ 三、示例一：电梯状态（最经典例子）

电梯的操作取决于当前状态：

| 状态   | 允许操作 |
| ------ | -------- |
| 运行中 | 不能开门 |
| 停止   | 可以开门 |
| 开门中 | 不能再开 |
| 关门中 | 不能再关 |

---

### 🧩 代码示例

```java
// 状态接口
interface ElevatorState {
    void openDoor(Elevator context);
    void closeDoor(Elevator context);
    void run(Elevator context);
    void stop(Elevator context);
}

// 环境类（上下文）
class Elevator {
    private ElevatorState state;

    public Elevator(ElevatorState state) { this.state = state; }
    public void setState(ElevatorState state) { this.state = state; }

    public void openDoor() { state.openDoor(this); }
    public void closeDoor() { state.closeDoor(this); }
    public void run() { state.run(this); }
    public void stop() { state.stop(this); }
}

// 具体状态：开门
class OpenState implements ElevatorState {
    public void openDoor(Elevator context) {
        System.out.println("🚪 门已开，无需重复操作");
    }
    public void closeDoor(Elevator context) {
        System.out.println("🚪 关门中...");
        context.setState(new ClosedState());
    }
    public void run(Elevator context) {
        System.out.println("❌ 开门状态无法运行电梯");
    }
    public void stop(Elevator context) {
        System.out.println("电梯已停止");
    }
}

// 具体状态：关门
class ClosedState implements ElevatorState {
    public void openDoor(Elevator context) {
        System.out.println("🚪 打开门");
        context.setState(new OpenState());
    }
    public void closeDoor(Elevator context) {
        System.out.println("🚪 门已关闭");
    }
    public void run(Elevator context) {
        System.out.println("🏃 电梯开始运行");
        context.setState(new RunningState());
    }
    public void stop(Elevator context) {
        System.out.println("电梯已停止");
    }
}

// 具体状态：运行
class RunningState implements ElevatorState {
    public void openDoor(Elevator context) {
        System.out.println("❌ 运行中禁止开门");
    }
    public void closeDoor(Elevator context) {
        System.out.println("门已关闭");
    }
    public void run(Elevator context) {
        System.out.println("电梯正在运行");
    }
    public void stop(Elevator context) {
        System.out.println("🏁 电梯停止运行");
        context.setState(new ClosedState());
    }
}

// 客户端
public class ElevatorDemo {
    public static void main(String[] args) {
        Elevator elevator = new Elevator(new ClosedState());

        elevator.openDoor();
        elevator.closeDoor();
        elevator.run();
        elevator.openDoor(); // 无效操作
        elevator.stop();
        elevator.openDoor(); // 成功开门
    }
}
```

✅ 输出：

```
🚪 打开门
🚪 关门中...
🏃 电梯开始运行
❌ 运行中禁止开门
🏁 电梯停止运行
🚪 打开门
```

💡 每种状态封装了自己的行为逻辑，
电梯对象无需用 `if (state == xxx)` 来判断状态。

## 💡 四、示例二：视频播放器状态

播放器的行为取决于当前状态：

* 播放中 → 可以暂停或停止
* 暂停 → 可以恢复播放
* 停止 → 可以播放

```java
interface PlayerState {
    void play(MediaPlayer player);
    void pause(MediaPlayer player);
    void stop(MediaPlayer player);
}

class MediaPlayer {
    private PlayerState state;
    public MediaPlayer(PlayerState state) { this.state = state; }
    public void setState(PlayerState s) { this.state = s; }

    public void play() { state.play(this); }
    public void pause() { state.pause(this); }
    public void stop() { state.stop(this); }
}

// 各种状态实现
class PlayingState implements PlayerState {
    public void play(MediaPlayer p) { System.out.println("正在播放中..."); }
    public void pause(MediaPlayer p) { System.out.println("暂停播放"); p.setState(new PausedState()); }
    public void stop(MediaPlayer p) { System.out.println("停止播放"); p.setState(new StoppedState()); }
}
class PausedState implements PlayerState {
    public void play(MediaPlayer p) { System.out.println("恢复播放"); p.setState(new PlayingState()); }
    public void pause(MediaPlayer p) { System.out.println("已经暂停"); }
    public void stop(MediaPlayer p) { System.out.println("停止播放"); p.setState(new StoppedState()); }
}
class StoppedState implements PlayerState {
    public void play(MediaPlayer p) { System.out.println("开始播放"); p.setState(new PlayingState()); }
    public void pause(MediaPlayer p) { System.out.println("无法暂停，当前已停止"); }
    public void stop(MediaPlayer p) { System.out.println("已经停止"); }
}

// 客户端
public class PlayerDemo {
    public static void main(String[] args) {
        MediaPlayer player = new MediaPlayer(new StoppedState());
        player.play();
        player.pause();
        player.play();
        player.stop();
    }
}
```

✅ 输出：

```
开始播放
暂停播放
恢复播放
停止播放
```

💡 每个状态只负责自己的一套逻辑，
避免 `switch(state)` 的庞大判断。

## ⚙️ 五、示例三：TCP 连接状态（实际工程常见）

在 JDK 的 `java.net` 源码中就有类似的状态模式思想：

| 状态        | 允许操作            |
| ----------- | ------------------- |
| CLOSED      | 可以 connect        |
| LISTEN      | 可以 accept         |
| ESTABLISHED | 可以 send / receive |
| FIN_WAIT    | 只能 close          |

不同状态下行为完全不同，状态切换时只要替换状态对象即可。

## 🧠 六、与策略模式的区别

| 维度           | 状态模式                         | 策略模式                       |
| -------------- | -------------------------------- | ------------------------------ |
| 意图           | 封装**对象状态切换**的行为       | 封装**可替换算法**             |
| 切换方式       | 状态对象**自行切换**（内部控制） | 策略由**外部指定**（外部选择） |
| 是否持有上下文 | ✅ 状态对象通常持有上下文引用     | ❌ 策略对象通常独立存在         |
| 是否有状态依赖 | ✅ 有（状态之间存在转换逻辑）     | ❌ 各策略互不关联               |
| 示例场景       | 电梯、播放器、TCP连接            | 支付方式、压缩算法、排序策略   |

> 🧠 一句话：
>
> * **策略模式**：外部替换算法（外界选择怎么做）
> * **状态模式**：内部随状态切换（对象自己知道怎么做）

---

## 🧩 七、优缺点总结

| 优点           | 说明                                   |
| -------------- | -------------------------------------- |
| ✅ 消除 if-else | 将状态逻辑分布到各状态类中             |
| ✅ 更易扩展     | 新增状态不影响原逻辑                   |
| ✅ 符合开闭原则 | 修改行为仅需添加新状态类               |
| ⚠️ 缺点         | 状态类增多；对象之间切换可能带来复杂性 |

## ⚙️ 八、在实际框架/系统中的应用

| 场景               | 应用                                         |
| ------------------ | -------------------------------------------- |
| JDK `Thread.State` | RUNNABLE, WAITING, TERMINATED 等             |
| 网络编程           | TCP 状态机                                   |
| 游戏开发           | 角色状态（Idle, Attack, Dead）               |
| GUI                | 按钮状态（Normal, Hovered, Disabled）        |
| 工作流引擎         | 任务流转状态（Created → Running → Finished） |

---

## ✅ 九、一句话总结

> 🧩 **状态模式（State Pattern）**：
> 把一个对象的行为封装到不同的状态类中，
> 让对象在内部状态变化时能自动切换行为。
>
> 🧠 “对象的行为会随状态的改变而改变，
> 就像它换了一个类一样。”
