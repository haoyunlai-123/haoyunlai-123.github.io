介绍下访问者模式并举几个例子

非常好 👍！
你现在进入到**行为型模式的最后一员大将** ——
👉 **访问者模式（Visitor Pattern）**。

这是一个稍显“抽象”的模式，但一旦理解它的动机，你会发现它非常优雅：

> 它解决的是：**如何在不改变数据结构的前提下，为其增加新的操作。**

这个模式在 **编译器（语法树遍历）**、**AST 分析器**、**代码生成器**、**对象序列化**、**报表系统** 等地方极其常见。

---

## 🧩 一、定义与核心思想

**访问者模式（Visitor Pattern）**
表示一个作用于某对象结构中的**各元素**的操作。
它使你可以在不改变这些元素类的前提下，定义作用于这些元素的新操作。

> 换句话说：
> “把**数据结构**和**操作逻辑**分离开，
> 新增操作时，不必修改原有类。”

---

### 🧠 一句话理解

> “在不动被访问者类的情况下，
> 给它们添加新功能 —— 把行为外包给访问者。”

---

## 🧱 二、结构图

```
     ┌────────────────────┐
     │  Visitor（访问者） │ ← 抽象访问者
     └────────┬──────────┘
              │
     ┌────────▼──────────┐
     │ ConcreteVisitorA  │ ← 具体访问者A
     │ ConcreteVisitorB  │ ← 具体访问者B
     └────────┬──────────┘
              │
 ┌────────────▼─────────────┐
 │ Element（被访问元素）     │ ← 抽象元素
 │ + accept(Visitor v)       │
 └────────────┬─────────────┘
              │
 ┌────────────▼─────────────┐
 │ ConcreteElementA/B...     │ ← 具体元素
 └───────────────────────────┘

      ElementCollection (对象结构)
```

## ⚙️ 三、示例一：公司报表系统（最常见例子）

假设公司中有两类员工：

* 普通员工（Engineer）
* 经理（Manager）

我们有两种操作：

* HR 查看绩效；
* CTO 查看代码质量。

我们希望**增加新操作时不修改员工类**。

---

### 🧩 代码示例

```java
// 访问者接口：定义对不同元素的访问操作
interface Visitor {
    void visit(Engineer engineer);
    void visit(Manager manager);
}

// 元素接口
interface Employee {
    void accept(Visitor visitor);
}

// 具体元素1：工程师
class Engineer implements Employee {
    private String name;
    private int codeQuality;

    public Engineer(String name, int codeQuality) {
        this.name = name;
        this.codeQuality = codeQuality;
    }

    public String getName() { return name; }
    public int getCodeQuality() { return codeQuality; }

    public void accept(Visitor visitor) {
        visitor.visit(this);
    }
}

// 具体元素2：经理
class Manager implements Employee {
    private String name;
    private int performance;

    public Manager(String name, int performance) {
        this.name = name;
        this.performance = performance;
    }

    public String getName() { return name; }
    public int getPerformance() { return performance; }

    public void accept(Visitor visitor) {
        visitor.visit(this);
    }
}

// 具体访问者A：HR 部门
class HRVisitor implements Visitor {
    public void visit(Engineer e) {
        System.out.println("HR 评估工程师 " + e.getName() + " 的绩效：代码质量 " + e.getCodeQuality());
    }
    public void visit(Manager m) {
        System.out.println("HR 评估经理 " + m.getName() + " 的绩效：团队绩效 " + m.getPerformance());
    }
}

// 具体访问者B：CTO
class CTOVisitor implements Visitor {
    public void visit(Engineer e) {
        System.out.println("CTO 检查工程师 " + e.getName() + " 的代码质量：" + e.getCodeQuality());
    }
    public void visit(Manager m) {
        System.out.println("CTO 评估经理 " + m.getName() + " 的项目绩效：" + m.getPerformance());
    }
}

// 对象结构
class Company {
    private List<Employee> employees = new ArrayList<>();

    public void add(Employee e) { employees.add(e); }

    public void accept(Visitor v) {
        for (Employee e : employees) e.accept(v);
    }
}

// 客户端
public class VisitorDemo {
    public static void main(String[] args) {
        Company company = new Company();
        company.add(new Engineer("Alice", 95));
        company.add(new Manager("Bob", 88));

        System.out.println("=== HR 审核 ===");
        company.accept(new HRVisitor());

        System.out.println("\n=== CTO 审核 ===");
        company.accept(new CTOVisitor());
    }
}
```

✅ 输出：

```
=== HR 审核 ===
HR 评估工程师 Alice 的绩效：代码质量 95
HR 评估经理 Bob 的绩效：团队绩效 88

=== CTO 审核 ===
CTO 检查工程师 Alice 的代码质量：95
CTO 评估经理 Bob 的项目绩效：88
```

💡 **优点：**

* 增加新的访问者（比如 CFO）很容易；
* 不用改动 Engineer、Manager 等类；
* 体现“开放扩展，封闭修改”。

## 💡 四、示例二：编译器的语法树（AST Visitor）

访问者模式最经典的实际应用就是在**编译器**或**解释器**中。

假设你有如下表达式：

```
(2 + 3) * 5
```

构建成抽象语法树（AST）：

```
       (*)
      /   \
    (+)    5
   /   \
  2     3
```

现在想实现两种操作：

* 求值；
* 打印表达式。

两种操作都属于“访问者”。

---

### 🧩 代码示例

```java
// 访问者接口
interface ExprVisitor {
    int visit(NumberExpr n);
    int visit(AddExpr add);
    int visit(MulExpr mul);
}

// 元素接口
interface Expr {
    int accept(ExprVisitor visitor);
}

// 具体元素
class NumberExpr implements Expr {
    int value;
    public NumberExpr(int value) { this.value = value; }
    public int accept(ExprVisitor v) { return v.visit(this); }
}

class AddExpr implements Expr {
    Expr left, right;
    public AddExpr(Expr l, Expr r) { this.left = l; this.right = r; }
    public int accept(ExprVisitor v) { return v.visit(this); }
}

class MulExpr implements Expr {
    Expr left, right;
    public MulExpr(Expr l, Expr r) { this.left = l; this.right = r; }
    public int accept(ExprVisitor v) { return v.visit(this); }
}

// 访问者1：求值
class EvalVisitor implements ExprVisitor {
    public int visit(NumberExpr n) { return n.value; }
    public int visit(AddExpr a) { return a.left.accept(this) + a.right.accept(this); }
    public int visit(MulExpr m) { return m.left.accept(this) * m.right.accept(this); }
}

// 访问者2：打印
class PrintVisitor implements ExprVisitor {
    public int visit(NumberExpr n) {
        System.out.print(n.value);
        return n.value;
    }
    public int visit(AddExpr a) {
        System.out.print("(");
        a.left.accept(this);
        System.out.print(" + ");
        a.right.accept(this);
        System.out.print(")");
        return 0;
    }
    public int visit(MulExpr m) {
        System.out.print("(");
        m.left.accept(this);
        System.out.print(" * ");
        m.right.accept(this);
        System.out.print(")");
        return 0;
    }
}

public class ASTVisitorDemo {
    public static void main(String[] args) {
        Expr expr = new MulExpr(
            new AddExpr(new NumberExpr(2), new NumberExpr(3)),
            new NumberExpr(5)
        );

        System.out.print("打印表达式：");
        expr.accept(new PrintVisitor()); // (2 + 3) * 5
        System.out.println();

        int result = expr.accept(new EvalVisitor());
        System.out.println("求值结果：" + result);
    }
}
```

✅ 输出：

```
打印表达式：(2 + 3) * 5
求值结果：25
```

💡 在编译器中，这种结构极其常见：

* 不同的 `Visitor` 代表不同操作（求值、类型检查、代码生成等）；
* 语法节点（Element）不需要修改。

## ⚙️ 五、示例三：文件系统统计

```java
interface FileVisitor {
    void visit(File file);
    void visit(Directory dir);
}

interface FileSystemItem {
    void accept(FileVisitor visitor);
}

class File implements FileSystemItem {
    String name;
    int size;
    public File(String name, int size) { this.name = name; this.size = size; }
    public void accept(FileVisitor v) { v.visit(this); }
}

class Directory implements FileSystemItem {
    String name;
    List<FileSystemItem> children = new ArrayList<>();
    public Directory(String name) { this.name = name; }
    public void add(FileSystemItem f) { children.add(f); }
    public void accept(FileVisitor v) {
        v.visit(this);
        for (FileSystemItem c : children) c.accept(v);
    }
}

// 访问者：统计文件大小
class SizeVisitor implements FileVisitor {
    private int total = 0;
    public void visit(File file) { total += file.size; }
    public void visit(Directory dir) { }
    public int getTotalSize() { return total; }
}

public class FileVisitorDemo {
    public static void main(String[] args) {
        Directory root = new Directory("root");
        root.add(new File("a.txt", 100));
        Directory sub = new Directory("docs");
        sub.add(new File("b.txt", 200));
        root.add(sub);

        SizeVisitor visitor = new SizeVisitor();
        root.accept(visitor);
        System.out.println("总大小: " + visitor.getTotalSize());
    }
}
```

✅ 输出：

```
总大小: 300
```

## 🧠 六、访问者模式的本质

> **把数据结构和操作解耦**。
> 元素类只提供 `accept(visitor)`，
> 所有逻辑都交给访问者去实现。

这样一来：

* 新增操作：✅ 只需新增一个 Visitor 类；
* 新增元素类型：⚠️ 必须修改所有 Visitor（代价较大）。

---

## ⚙️ 七、优缺点总结

| 优点           | 说明                             |
| -------------- | -------------------------------- |
| ✅ 符合开闭原则 | 新增操作（Visitor）不需改原类    |
| ✅ 职责清晰     | 数据结构与操作逻辑分离           |
| ✅ 易于扩展     | 访问逻辑统一集中在 Visitor 中    |
| ⚠️ 缺点         | 新增元素类型时要修改所有 Visitor |
| ⚠️ 结构复杂     | 类较多，入门代码量大             |

## ✅ 九、一句话总结

> 🧩 **访问者模式（Visitor Pattern）**：
> 在不改变数据结构的前提下，定义新的操作。
> 通过“双重分派”实现对不同类型元素的访问。
>
> 🧠 “数据稳定，操作多变——用访问者；
> 操作稳定，数据多变——用多态。”

## 🧠 来更深入一点看：为什么它“解耦”得这么彻底？

普通情况下：

```java
class Employee {
    void printReport() {...}
    void calculateSalary() {...}
    void exportJSON() {...}
}
```

👉 操作都混在类里，
每加一个新功能，就得改这个类（违反开闭原则）。
久而久之，这个类会变成“上帝类”（God Class）。

---

访问者模式改写为：

```java
interface Visitor {
    void visit(Employee e);
}

interface Element {
    void accept(Visitor v);
}
```

数据类只保留最小职责：

```java
class Employee implements Element {
    private String name;
    private int salary;
    public void accept(Visitor v) { v.visit(this); }
}
```

操作全部外包：

```java
class ReportVisitor implements Visitor {
    public void visit(Employee e) { /* 打印报表 */ }
}

class SalaryVisitor implements Visitor {
    public void visit(Employee e) { /* 计算工资 */ }
}

class ExportVisitor implements Visitor {
    public void visit(Employee e) { /* 导出 JSON */ }
}
```

✅ 想新增功能？
👉 新建一个 `Visitor`，完全不用动 `Employee`。
这就是**数据和行为的彻底解耦**。

## 🔄 对比：策略模式 vs 访问者模式

| 对比点      | 策略模式      | 访问者模式     |
| -------- | --------- | --------- |
| 关注点      | **算法可替换** | **操作可扩展** |
| 操作位置     | 注入策略类执行   | 访问者集中定义操作 |
| 是否依赖数据结构 | 不依赖       | 依赖数据结构    |
| 新增算法     | 增加策略类     | 增加访问者类    |
| 新增数据类型   | 不需改策略     | 需改所有访问者   |

👉 一句话：
**策略模式解决“算法替换”，访问者模式解决“操作扩展”。**

## 🧩 再来个真实世界的例子：AST 分析器

假设我们要做一个 Go / Java 代码语法分析工具。
代码经过解析器变成 AST（语法树）：

```text
Function
 ├── Parameters
 ├── VariableDeclarations
 └── ReturnStatement
```

如果你要做：

* ✅ 打印 AST
* ✅ 检查变量命名规范
* ✅ 静态复杂度分析
* ✅ 自动生成文档
* ✅ 安全扫描（SQL 注入检测）

👉 用访问者模式就可以：

```java
interface ASTVisitor {
    void visit(FunctionNode f);
    void visit(VarDeclNode v);
    void visit(ReturnNode r);
}

class PrintVisitor implements ASTVisitor {...}
class SecurityVisitor implements ASTVisitor {...}
class DocGeneratorVisitor implements ASTVisitor {...}
```

而 AST 本身完全不变。
所以 **编译器（如 javac, Go AST, clang）** 都是访问者模式的经典体现。

## 💡 再延伸一点：访问者模式 = 双重分派（Double Dispatch）

为什么访问者能做到“根据类型调用正确操作”？
靠的是“双重分派”：

1️⃣ 元素调用访问者：

```java
element.accept(visitor);
```

2️⃣ 元素反过来把自己传给访问者：

```java
visitor.visit(this);
```

这样 JVM 在运行时既知道“访问者是谁”，又知道“访问的对象是谁”，
实现了“根据对象类型+访问者类型”选择正确方法。

这比普通的多态更强 —— 普通多态只根据“一个对象”的类型动态分派。

| 场景            | 使用方式              |
| ------------- | ----------------- |

* “添加新统计逻辑” 而不动核心结构；
* “扩展输出格式” 而不改业务类；
* “插拔功能模块” 而不改主干代码。
