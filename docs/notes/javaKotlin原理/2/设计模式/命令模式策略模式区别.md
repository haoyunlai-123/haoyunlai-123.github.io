它和策略模式区别在哪

非常棒 👍，这个问题问得非常有深度 ——
 **命令模式（Command Pattern）** 和 **策略模式（Strategy Pattern）** 的结构长得非常像，
 很多人初学时都会觉得“这俩不是一个东西吗？”

确实，它们**表面上都封装了行为**，都使用了接口 + 实现类的结构，
 但它们的**设计目的、关注点和使用场景完全不同**。

下面我们来系统地对比它们 👇

------

## 🧩 一、核心定义对比

| 模式                    | 定义                                             | 关注点             |
| ----------------------- | ------------------------------------------------ | ------------------ |
| **策略模式 (Strategy)** | 封装**一系列算法**或行为，使它们可以互相替换     | “**选择不同算法**” |
| **命令模式 (Command)**  | 将**请求封装为对象**，以便存储、排队、撤销、组合 | “**封装执行动作**” |

------

## 🧠 二、核心思想差异

| 特点                 | 策略模式                       | 命令模式                                        |
| -------------------- | ------------------------------ | ----------------------------------------------- |
| 目标                 | 实现算法或行为的**可替换性**   | 实现请求的**可记录 / 可撤销 / 可排队 / 可组合** |
| 调用关系             | 客户端直接调用策略执行逻辑     | 客户端通过命令间接触发执行者                    |
| 是否包含“接收者”     | ❌ 通常没有（策略本身就是算法） | ✅ 有（命令内部会调用 Receiver）                 |
| 是否支持撤销（undo） | ❌ 一般不支持                   | ✅ 通常支持                                      |
| 是否可延迟执行       | ❌ 实时执行                     | ✅ 可延迟或排队执行                              |
| 用法比喻             | “选一个算法”                   | “发一个命令”                                    |

------

## 🧱 三、代码对比示例

------

### ✳️ 策略模式示例：支付策略

```java
// 策略接口
interface PaymentStrategy {
    void pay(double amount);
}

// 具体策略
class WeChatPay implements PaymentStrategy {
    public void pay(double amount) {
        System.out.println("微信支付 " + amount + " 元");
    }
}

class AliPay implements PaymentStrategy {
    public void pay(double amount) {
        System.out.println("支付宝支付 " + amount + " 元");
    }
}

// 上下文
class PaymentContext {
    private PaymentStrategy strategy;
    public PaymentContext(PaymentStrategy strategy) { this.strategy = strategy; }
    public void execute(double amount) { strategy.pay(amount); }
}

// 客户端
public class StrategyDemo {
    public static void main(String[] args) {
        PaymentContext ctx = new PaymentContext(new WeChatPay());
        ctx.execute(100.0); // 选择不同策略 = 使用不同算法
    }
}
```

🧠 **关键点**：

- 只有算法，不存在“命令”或“接收者”；
- 调用时立即执行；
- 主要解决 “不同实现如何互换” 的问题。

------

### ⚡ 命令模式示例：遥控器控制灯

```java
interface Command { void execute(); }

class Light {
    void on() { System.out.println("灯亮了"); }
    void off() { System.out.println("灯灭了"); }
}

class LightOnCommand implements Command {
    private Light light;
    public LightOnCommand(Light light) { this.light = light; }
    public void execute() { light.on(); }
}

class RemoteControl {
    private Command command;
    public void setCommand(Command command) { this.command = command; }
    public void press() { command.execute(); }
}

public class CommandDemo {
    public static void main(String[] args) {
        Light light = new Light();
        Command cmd = new LightOnCommand(light);
        RemoteControl remote = new RemoteControl();
        remote.setCommand(cmd);
        remote.press(); // 发出命令，执行者执行动作
    }
}
```

🧠 **关键点**：

- 命令封装了“动作”；
- 命令持有接收者；
- 可以被存储、撤销、组合；
- 调用者不关心命令如何执行。

------

## 🔍 四、核心区别（一句话总结）

| 对比项           | 策略模式                     | 命令模式                              |
| ---------------- | ---------------------------- | ------------------------------------- |
| 抽象对象代表什么 | 一个算法策略                 | 一次操作请求                          |
| 目标             | 可替换算法                   | 可控制请求                            |
| 调用关系         | 客户端直接执行策略           | 客户端发出命令，由接收者执行          |
| 是否有“接收者”   | 无                           | 有                                    |
| 是否支持撤销     | 否                           | 是                                    |
| 是否支持宏命令   | 否                           | 是                                    |
| 是否可延迟执行   | 否                           | 是                                    |
| 常见场景         | 算法选择、支付方式、排序策略 | 操作日志、撤销/重做、任务队列、宏指令 |

------

## 💡 五、打个比喻（非常直观）

| 模式         | 类比                                                 |
| ------------ | ---------------------------------------------------- |
| **策略模式** | 你自己开车上班 —— 每天可以选择不同路线（算法）       |
| **命令模式** | 你叫司机打车上班 —— 你发出“去公司”的命令，司机去执行 |

也就是说：

- 策略模式 = **自己执行，不同策略不同做法**
- 命令模式 = **发命令，别人执行，可以撤销/重放**

------

## ⚙️ 六、在框架中的典型应用对比

| 场景                   | 使用模式 | 案例                                 |
| ---------------------- | -------- | ------------------------------------ |
| Java `Comparator` 接口 | 策略模式 | `Collections.sort(list, comparator)` |
| Spring `KeyGenerator`  | 策略模式 | 不同缓存键生成算法                   |
| Swing `ActionListener` | 命令模式 | 按钮点击封装为命令                   |
| IDE 撤销功能           | 命令模式 | 每个编辑操作是命令，可回滚           |
| 任务调度器 / MQ        | 命令模式 | 命令可序列化执行                     |

------

## ✅ 七、一句话总结

> 🧩 **策略模式**：封装可互换的算法，实现行为选择。
>  🧩 **命令模式**：封装可执行的操作，实现行为请求、撤销与排队。
>
> 🧠 **策略关注“怎么做”，命令关注“谁去做”和“什么时候做”。**
