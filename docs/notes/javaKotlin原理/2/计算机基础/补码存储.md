帮我复习下补码的知识，以1字节-128-127的二进制存储为例

非常好 👍——补码是理解计算机底层整数存储、位运算、溢出等所有机制的基础。
 我们来系统复习一遍，用 **1 字节（8 位）范围 -128～127** 为例，从定义 → 原码 → 反码 → 补码 → 运算直观图，一步步讲清楚。

------

## 🧩 一、为什么需要补码？

计算机只能处理 **二进制（0/1）**，而我们要表示正数和负数。
 如果只用一个 bit 来区分正负（最高位为符号位），就有三个方案：

| 表示法                       | 缺点                                 |
| ---------------------------- | ------------------------------------ |
| **原码（sign-magnitude）**   | +0 和 -0 有两种表示，运算复杂        |
| **反码（ones' complement）** | 仍然存在 +0 / -0 问题                |
| **补码（two's complement）** | ✅ 唯一的 0、加减法统一，硬件实现简单 |

👉 所以所有现代 CPU（包括 Java、C、Go）都使用 **补码表示有符号整数**。

------

## 🧮 二、1 字节（8 位）补码的范围

| 位数 | 可表示个数 | 范围             |
| ---- | ---------- | ---------------- |
| 8 位 | 256        | **-128 ～ +127** |

最高位（bit7）为 **符号位**：

- `0` → 正数或 0
- `1` → 负数

------

## ⚙️ 三、正数的补码

> 正数的补码 = 原码 = 反码

举例：

| 十进制 | 原码（二进制） | 补码        |
| ------ | -------------- | ----------- |
| 0      | `0000 0000`    | `0000 0000` |
| +1     | `0000 0001`    | `0000 0001` |
| +2     | `0000 0010`    | `0000 0010` |
| +127   | `0111 1111`    | `0111 1111` |

------

## ⚙️ 四、负数的补码

> 负数的补码 = “反码 + 1”

步骤：

1. 先写出对应正数的二进制；
2. **逐位取反**；
3. **末位加 1**。

举例：

| 十进制   | 原码        | 反码        | 补码（反码+1）         |
| -------- | ----------- | ----------- | ---------------------- |
| -1       | `1000 0001` | `1111 1110` | ✅ `1111 1111`          |
| -2       | `1000 0010` | `1111 1101` | ✅ `1111 1110`          |
| -3       | `1000 0011` | `1111 1100` | ✅ `1111 1101`          |
| -127     | `1111 1111` | `1000 0000` | ✅ `1000 0001`          |
| **-128** | ❌ 超出范围  | ❌           | ✅ `1000 0000` （特殊） |

注意：

- `1000 0000` 是 **-128 的补码**；
- 没有对应的“+128”，所以补码表示区间不对称。

------

## ⚖️ 五、验证：补码加减法一体化

补码的最大优势：

> 加法器不用区分“减法”，因为 `A - B` 等价于 `A + (-B)`。

例如：

```text
0000 0011   (+3)
+
1111 1101   (-3)
------------
0000 0000   (结果 0)
```

CPU 不需要知道谁是负数，只要按位加，丢弃溢出即可。

------

## 🧠 六、补码的还原（求原数）

若已知补码，要反推出十进制值：

- 如果最高位 = 0 → 是正数 → 直接读；
- 如果最高位 = 1 → 是负数 → 先取反 +1 再加负号。

例：

```
1111 1011
→ 取反 0000 0100
→ +1 = 0000 0101
→ => -5
```

## 💡 八、几个关键性质（非常重要）

| 性质                 | 解释           |
| -------------------- | -------------- |
| 补码相加自动溢出环绕 | 127 + 1 → -128 |
| 零唯一               | 0000 0000      |
| 对称性破缺           | +128 不存在    |
| 位运算意义一致       | `~x + 1 = -x`  |
| 负数右移保留符号位   | 算术右移填 1   |

## 🔍 九、快速心算技巧

想从补码直接看出十进制（8 位）：

- 若最高位 0 → 正常二进制；
- 若最高位 1 → `(256 - 二进制值)` 再加负号。

例：

```
1111 1011 = 0xFB = 251
256 - 251 = 5  → -5
```

我在一些位运算的算法中使用移位运算时操作的是什么码？

💯 非常好的问题——你已经进入位运算真正的核心层面。
 答案一句话概括是：

> ✅ 在 Java（以及几乎所有现代 CPU）中，**位运算（包括移位）操作的是补码形式（Two’s Complement Representation）下的二进制位**。

但这句话背后有很多细节：包括**逻辑移位 vs 算术移位**、**符号位的处理**、**补码的右移行为** 等，我们一起来彻底搞懂它。

------

## 🧩 一、前提：Java 整型存储都用补码

Java 的 `byte`, `short`, `int`, `long` 全部采用 **补码存储有符号整数**。
 也就是说，在内存（寄存器）层面，CPU 实际看到的就是补码。

例：

```java
int x = -5;
System.out.println(Integer.toBinaryString(x));
```

输出：

```
11111111111111111111111111111011
```

这是 -5 的 **32 位补码**。

因此，当你写：

```java
x >> 1;
```

JVM 直接在补码的这 32 个二进制位上进行移位操作。

------

## ⚙️ 二、移位运算的三种类型（Java）

| 运算符 | 名称                                   | 是否保留符号位             | 左边补入               |
| ------ | -------------------------------------- | -------------------------- | ---------------------- |
| `<<`   | 左移（Left Shift）                     | ✅ 不影响符号位（除非溢出） | 右边补 0               |
| `>>`   | **算术右移（Arithmetic Right Shift）** | ✅ 保留符号位               | 左边补符号位（0 或 1） |
| `>>>`  | **逻辑右移（Logical Right Shift）**    | ❌ 不保留符号位             | 左边补 0               |

------

### 📘 举例分析

#### 1️⃣ 左移 `<<`

```java
int a = 5;   // 00000000 00000000 00000000 00000101
int b = a << 1;
```

结果：

```
00000000 00000000 00000000 00001010  // 10
```

负数也一样直接位移，因为只是往左挪位，末尾补 0：

```java
int a = -5;  // 11111111 11111111 11111111 11111011
int b = a << 1;
```

结果：

```
11111111 11111111 11111111 11110110  // -10
```

👉 注意：虽然符号位可能变化，但算法就是在“补码位串”上移位。

------

#### 2️⃣ 算术右移 `>>`（保留符号位）

```java
int a = -5;  // 11111111 11111111 11111111 11111011
int b = a >> 1;
```

右移 1 位并补上 **符号位 1**：

```
11111111 11111111 11111111 11111101  // -3
```

📌 补 1 的原因是为了保持符号不变（向下取整），等价于除以 2 向下取整。

------

#### 3️⃣ 逻辑右移 `>>>`（无符号右移）

```java
int a = -5;  // 11111111 11111111 11111111 11111011
int b = a >>> 1;
```

左侧补 **0**：

```
01111111 11111111 11111111 11111101  // 2147483645
```

👉 结果变成正数，因为符号位清 0 了。

------

## 🧮 三、所以操作的是什么“码”？

| 代码层面      | 实际操作对象                 | 移位规则                   |
| ------------- | ---------------------------- | -------------------------- |
| `int x = -5;` | 内存中是补码 `111111...1011` | 位运算直接在补码位串上执行 |
| `x >> 1`      | 对补码右移并补符号位         | 算术右移                   |
| `x >>> 1`     | 对补码右移并补 0             | 逻辑右移                   |

因此：

- 位移操作的**本质是对补码的二进制位操作**；
- CPU、JVM 都不会去管“原码”或“反码”；
- 对负数右移时是否补 1 完全由指令类型（算术/逻辑）决定。

------

## 🔍 四、结合示例看“补码视角”的重要性

```java
int a = -8;
System.out.println(Integer.toBinaryString(a));
System.out.println(Integer.toBinaryString(a >> 2));
System.out.println(Integer.toBinaryString(a >>> 2));
```

输出：

```
11111111111111111111111111111000   // -8 补码
11111111111111111111111111111110   // -2 (算术右移)
00111111111111111111111111111110   // 大正数 (逻辑右移)
```

- `>>` 保留符号位 1，所以仍是负数；
- `>>>` 直接补 0，符号丢失。

------

## 🧠 五、和 CPU 指令的关系（底层验证）

在汇编层面（例如 x86 指令集）：

| Java 操作 | 对应机器指令                                                |
| --------- | ----------------------------------------------------------- |
| `>>`      | `SAR` (Arithmetic Shift Right)                              |
| `>>>`     | `SHR` (Logical Shift Right)                                 |
| `<<`      | `SAL` or `SHL` (Shift Left Arithmetic/Logical — 实际无区别) |

这些都是在**补码寄存器值**上直接操作位。
 CPU 不知道什么“原码”或“反码”——它看到的永远是补码。

| 项目              | 说明                                   |
| ----------------- | -------------------------------------- |
| Java 整型存储形式 | 补码（Two’s Complement）               |
| 位运算操作对象    | 补码二进制位串                         |
| 左移 `<<`         | 低位补 0                               |
| 算术右移 `>>`     | 高位补符号位                           |
| 逻辑右移 `>>>`    | 高位补 0（无符号右移）                 |

| CPU 层对应        | `SAR` / `SHR` / `SHL` 指令             |
| 为什么这样设计    | 保证补码加减、移位、算术统一，硬件简单 |
