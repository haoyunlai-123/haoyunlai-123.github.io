如果弱引用的对象有finalize()方法并能再其中复活，那它会被直接清除还是加入finalizer队列？

💯 非常精彩的问题——这其实考察了 **JVM 对引用类型（特别是 WeakReference）与 Finalizer 队列的交互时序**。
 这个问题在 HotSpot 内部实现中非常细微，很多高级开发者都搞混。
 我们来逐层剖析👇

------

## ✅ 一、先说结论（结论清晰版）

> **弱引用（WeakReference）的对象如果定义了 `finalize()` 方法，依然会走 Finalizer 队列流程**。
>
> GC 不会直接清除它。
>
> 它在第一次被判定为不可达时：
>
> 1. 会被**加入 Finalizer 队列**（等待 `finalize()` 执行）；
> 2. 此时 WeakReference 的 referent 不会被清除；
> 3. 如果对象在 `finalize()` 中复活（重新建立强引用），则：
>    - 该对象从 Finalizer 队列移除；
>    - WeakReference 的 referent 会被重新指向强引用；
>    - 对象不被回收；
> 4. 如果未复活，下一次 GC 才会真正清除并将 WeakReference 入队。

👉 简单说：
 **Finalizer 优先于 WeakReference 的清除。**

------

## 🧩 二、GC 时的判断顺序（HotSpot 实现逻辑）

HotSpot（以及所有符合 JVM 规范的实现）在 GC 时会按照如下顺序处理引用类型和 finalize：

### 1️⃣ 普通可达性分析

GC 首先从 GC Roots 出发做 **可达性分析 (Reachability Analysis)**：
 标记出所有 **强可达** 对象。

------

### 2️⃣ 找到“不可达但有 finalize()”的对象

对那些“不可达对象”，如果其类覆写了 `finalize()` 且尚未执行过，
 JVM 会：

- 暂时**保留该对象**；
- 把它放入 **Finalizer 队列**；
- 标记为 “Finalizer pending”；
- **跳过所有 Reference（包括 WeakReference）清理阶段**。

📌 注意：

> 此时该对象的弱引用（`WeakReference.referent`）仍然指向它，
>  GC 不会把 referent 置空。

------

### 3️⃣ Finalizer 线程执行 finalize()

Finalizer 线程异步执行该对象的 `finalize()`。

有两种可能：

| 情况                                                | 结果                                |
| --------------------------------------------------- | ----------------------------------- |
| 对象在 finalize 中 **复活**（例如赋给某个静态变量） | 对象变为强可达，不再回收            |
| 对象未复活                                          | finalize 执行完后，下一次 GC 再处理 |

------

### 4️⃣ 下一轮 GC 时处理 WeakReference

在下一次 GC：

- 如果对象仍不可达（未复活），且 finalize 已执行过，
   它的 WeakReference 才会被清除（referent 置 null），
   并将该 WeakReference 对象放入其 ReferenceQueue。

------

## ⚙️ 三、时序图理解（文字版）

```
GC#1:
  检测对象 O (WeakReference → O)
  ↓
  O 不可达 且 有 finalize()
  ↓
  O 放入 Finalizer 队列（不清 WeakReference）
  ↓
  Finalizer 线程执行 O.finalize()

GC#2:
  若 finalize() 中未复活 → 清除 WeakReference
  若 finalize() 中复活 → 不清除 WeakReference（对象存活）
```

------

## 🧠 四、特殊补充：JDK 9+ 弃用 finalize 的影响

虽然 `finalize()` 在 JDK 9 起被标记为 deprecated，
 但 JVM 的 GC 内部逻辑仍保留上述顺序（Finalizer 队列优先于 Reference 清除）。
 只是如果一个类没有 `finalize()` 方法，这个阶段会被直接跳过。

------

## 🧱 五、为什么要这么设计？

关键原因是：

> `finalize()` 可能改变对象可达性（复活），
>  如果在执行 finalize 之前就清理 WeakReference，
>  那么 WeakReference 会提前失效，破坏一致性。

因此，HotSpot 强制要求：

- **Finalization 先于 Reference 清理阶段执行**；
- 保证在 finalize 完成、确定对象未复活之后，
   才可以把 WeakReference 的 referent 置空。

------

## ✅ 六、总结表格

| 阶段              | 是否执行 finalize()   | WeakReference referent | 是否清除     |
| ----------------- | --------------------- | ---------------------- | ------------ |
| 第一次不可达      | ✅ 加入 Finalizer 队列 | 保留                   | ❌ 不清除     |
| Finalizer 执行中  | ✅ 正在执行            | 保留                   | ❌ 不清除     |
| finalize 后未复活 | ❌ 等待下次 GC         | 将被清除               | ✅ 下一次清除 |
| finalize 后复活   | ❌                     | 保留                   | ❌ 不清除     |

------

## 🧩 七、一句话总结

> 弱引用的对象如果定义了 `finalize()` 方法，
>  在第一次被判定为不可达时不会立刻清除弱引用，
>  而是先进入 Finalizer 队列。
>
> 只有当 `finalize()` 执行完且对象未复活时，
>  JVM 才会在下一轮 GC 中清除它的 WeakReference。
