字符串常量池在eden区吗？

非常好的问题，这个细节很多人都容易搞混。我们来精确地看一下：

------

### 🧠 一句话回答

**不是的。字符串常量池（String Intern Pool）不在 Eden 区，而是在 Java 堆的永久代 / 元空间管理的堆区域中（取决于 JDK 版本）。**

------

### 🧩 不同版本下的分布情况

| JDK 版本         | 常量池存放位置                          | 说明                                                         |
| ---------------- | --------------------------------------- | ------------------------------------------------------------ |
| **JDK 6 及之前** | **PermGen（永久代）**                   | `String.intern()` 创建的字符串常量会被放进永久代；PermGen 是 HotSpot 专用概念。 |
| **JDK 7**        | **Java Heap（堆中单独区域）**           | Oracle 将字符串常量池从 PermGen 移到堆中，以缓解 PermGen OOM 问题。Eden 只是堆的**年轻代**，而常量池在**老年代或专用区**中。 |
| **JDK 8 及之后** | **Java Heap（堆）+ 元空间仅存类型信息** | PermGen 被废弃，类型元数据放入 Metaspace（本地内存），而字符串常量池依然在堆内。 |

------

### 📦 内存划分关系图（JDK 8+）

```
┌────────────────────────────┐
│       JVM 内存布局         │
├────────────────────────────┤
│  Metaspace (native)        │ ← 类元信息、方法元数据
├────────────────────────────┤
│  Java Heap                 │
│   ├── Young Gen            │
│   │    ├─ Eden             │
│   │    ├─ Survivor0/1      │
│   ├── Old Gen              │
│   │    └─ 对象晋升区       │
│   ├── String Intern Pool   │ ← 🔥 字符串常量池就在这里
│   └── 其他缓冲区           │
└────────────────────────────┘
```

> ✅ **注意**：字符串常量池在堆上，但不属于 Eden 或 Survivor，而是堆中一个独立的结构（`StringTable`），JVM 用哈希表管理。

------

### ⚙️ 实现原理补充

- HotSpot 源码关键类：`src/hotspot/classfile/stringTable.cpp`
- 它是一个全局哈希表结构，管理所有 `interned` 字符串。
- 当调用 `String.intern()` 时：
  1. 若池中存在该字符串，则直接返回池中引用；
  2. 否则将此字符串对象添加到堆中的 `StringTable`。

## 🔬 一、`new String("abc")` 实际创建了两个对象（或复用已有一个）

代码：

```java
String s = new String("abc");
```

### 对象分布过程：

1. `"abc"` 是**字符串字面量**，会在**编译期**放入 class 文件的常量池。
   - 加载类时，JVM 会将该常量加载到**运行时常量池**，并在**字符串常量池（StringTable）** 中创建/缓存一个 `"abc"` 对象。
   - 该对象**驻留在堆上**（自 JDK 7 起），是全局共享的，不会被回收，只要有类加载器引用在。
2. `new String("abc")`
   - 会在**堆的 Eden 区**中再创建一个新的 `String` 对象。
   - 其内部的 `value`（`byte[]` 或 `char[]`，取决于 JDK 版本）会指向 `"abc"` 的那个底层数组。
   - 它与字符串常量池的 `"abc"` 是**两个不同的对象**。

```
│   └── "abc" ──► char[] {'a','b','c'} │

┌──────────────────────────────────────┐

│   └── new String("abc")              │
│        value → 指向上面的 char[]     │
└──────────────────────────────────────┘
```

| 对象                | 所在位置           | GC 生存条件                    | 是否能被 Minor GC 回收                          |
| ------------------- | ------------------ | ------------------------------ | ----------------------------------------------- |
| `"abc"` 字面量对象  | 堆（字符串常量池） | 被全局引用（Class 常量池持有） | ❌ **不会被回收**                                |
| `new String("abc")` | 堆（Eden 区）      | 仅被 `s` 引用                  | ✅ 若 `s` 不再引用它，可以在 **Minor GC** 中回收 |

## 🧪 四、例子验证

```java

        String s = new String("abc");
        s = null;        // 去除引用
        System.gc();     // 触发 GC
    }

```

- 这里 `"abc"`（常量池中）仍被 Class 常量池引用，不会被回收。
- 但 `new String("abc")` 这个在 Eden 区的新对象**没有任何引用**，在下一次 **Minor GC** 中**会被回收**。

## ⚙️ 五、特殊情况

1. 若你写：

   ```java
   String s = new String("abc").intern();
   ```

   那么：

   - intern() 会返回池中 `"abc"` 对象。
   - 如果池中原本没有，会把此 `new` 出来的字符串加入池（JDK7+ 为堆内引用）；
   - 此时 `new` 出来的对象可能不再保留引用，被 GC 回收。

2. 若字符串量太多，常量池中缓存的 `"abc"` 等字符串**仍可能被 GC 回收**（JDK 8+ 的 StringTable 是弱引用实现，可以在 Full GC 时清理未被引用的条目）。

## 🧩 一、位置：字符串常量池在「堆」中，但不属于老年代/新生代

自 **JDK 7 开始（含 JDK 8、11、17、21 等）**：

```
┌───────────────────────────────┐
│        Java Heap（堆）         │
│ ┌──────────────┬────────────┐ │
│ │  新生代 (Eden+S0+S1)     │ │
│ │  老年代 (Tenured)         │ │
│ ├───────────────────────────┤ │
│ │  StringTable（字符串常量池）│ │  ← 🌟 在堆中独立哈希表
│ └───────────────────────────┘ │
└───────────────────────────────┘
```

🔹 **重点**：

* `StringTable` 是一个 **全局哈希表结构**（`stringTable.cpp` 实现），存放在**堆**中；
* 它 **不隶属于年轻代或老年代**，而是**独立于普通对象分配区之外的结构**；
* 但其内部的字符串对象（即 `String` 实例）**确实存放在堆的普通区域**（可能在老年代）。

---

## ⚙️ 二、回收行为：由「普通 GC 线程」负责，但触发时机不同

### 💡 背景：

* `StringTable` 是全局共享哈希表，键是 `Symbol*`（C++ 层 intern 的符号），值是 `oop` （Java 层 String 对象引用）。
* 哈希表中的 entry 是**弱引用（WeakHandle）**。
* 所以，当某个 intern 字符串没有被任何 Java 代码引用时，它的条目会被 GC 清理。

---

### 🧹 具体触发条件

| GC 类型              | 是否扫描 `StringTable` | 是否可能回收字符串             |
| ------------------ | ------------------ | --------------------- |
| **Minor GC**（年轻代）  | ❌ 否                | ❌ 不会清理常量池条目           |
| **Major GC**（仅老年代） | 🔸 通常不独立运行（取决于实现）  | ❌ 不会主动扫描 StringTable  |
| **Full GC**（整堆）    | ✅ 是                | ✅ 可以清理无引用的 intern 字符串 |

---

### 📘 举例（HotSpot 源码）

在 `StringTable::unlink_or_oops_do` 中可以看到：

```cpp
void StringTable::unlink_or_oops_do(BoolObjectClosure* is_alive, OopClosure* f, int* processed, int* removed) {
  ...
  if (!is_alive->do_object_b(o)) { // 若对象不再可达
      remove_entry(entry);
  }
}
```

这个方法仅在 **Full GC（或 Concurrent Marking / Remark 阶段）** 调用。
Minor GC 只扫描 Eden → Survivor 引用，不涉及 StringTable。

| 项目                     | 位置           | GC 是否参与            | 何时可被清理                             |
| ---------------------- | ------------ | ------------------ | ---------------------------------- |
| **StringTable（哈希表结构）** | 堆内特殊区        | GC 标记阶段扫描          | Full GC / CMS Remark / G1 Mixed GC |
| **其中的 String 对象**      | 普通堆（常驻老年代）   | 可标记为不可达            | 同上                                 |
| **Minor GC**           | 扫描年轻代对象      | 🚫 不影响 StringTable |                                    |
| **Major GC / Full GC** | 扫描整个堆 + 弱引用表 | ✅ 清理无引用的 intern 条目 |                                    |

## 🧠 四、在不同 GC 算法下的差异

| GC 算法                           | `StringTable` 清理时机        |
| ------------------------------- | ------------------------- |
| **Serial / Parallel GC**        | 在 Full GC 时统一扫描弱引用表       |
| **CMS (Concurrent Mark Sweep)** | 在 Remark 阶段处理 StringTable |
| **G1 GC**                       | 在 Mixed GC 或 Remark 阶段清理  |
| **ZGC / Shenandoah**            | 由于并发标记机制，周期性清理弱引用表        |

## 🧹 四、GC 时的实际逻辑（以 G1 为例）

1. GC Roots 遍历：扫描强引用（线程栈、本地句柄、静态字段等）；
2. 扫描弱引用表（包括 StringTable 的弱引用）；
3. 对于 WeakHandle 指向的对象，如果没有强引用存活：

   * 该 String 对象会被 GC；
   * StringTable 的条目在清理阶段被移除（`StringTable::unlink_or_oops_do`）。
