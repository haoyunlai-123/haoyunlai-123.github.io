当加载一个类时，它的class对象创建时会放在哪里？eden区还是直接放在老年代？

这个问题问得非常关键 👍，很多人学类加载机制时会把“类加载的元信息（metadata）”和“Class对象本身”混为一谈。你已经注意到这两个层次的区别：

- 一是「类的元信息」——Klass模板、方法表、常量池等；
- 二是「`java.lang.Class` 对象」——Java层的类对象。

我们就从 HotSpot 的实现层面把这两个部分都讲清楚👇

------

## 🧩 一、类加载后到底产生了两类结构

当一个类被加载时，JVM 会为它生成：

| 名称                           | 存放内容                                                 | 存放位置                  |
| ------------------------------ | -------------------------------------------------------- | ------------------------- |
| **Class 元信息（Klass 对象）** | 类名、字段表、方法表、常量池、继承关系、vtable/itable 等 | 🧠 **Metaspace（元空间）** |
| **`java.lang.Class` 对象**     | Java 层能访问到的类对象（反射时用）                      | 💾 **Java 堆（Heap）**     |

也就是说，一个类加载进来之后，其底层结构跨越了两个内存区域：

```
+-------------------+       +---------------------------+
|   Metaspace       |       |         Java Heap         |
|-------------------|       |---------------------------|
| InstanceKlass --> |-----> | java.lang.Class instance  |
| (方法表、常量池等) |       | (字段、反射、静态变量等)    |
+-------------------+       +---------------------------+
```

------

## ⚙️ 二、那 `java.lang.Class` 对象放在哪个分区？Eden 还是老年代？

答案分两步来理解：

### 1️⃣ 分配时机：初次加载时创建

当类加载完成、InstanceKlass 构建好后，JVM 会立刻创建一个对应的 Java 对象：

```cpp
oop java_mirror = oopFactory::new_instance(SystemDictionary::Class_klass());
```

这个对象就是 `java.lang.Class` 实例。

### 2️⃣ 分配位置：

- 它是**普通的 Java 对象**（只是类型特殊），
- 所以最初也会在堆的 **Eden 区** 中分配；
- 随后经历 Minor GC 时，像普通对象一样判断是否晋升；
- 若长时间存活，会晋升到 Survivor → Old Generation。

> ✅ **结论：**
>  `java.lang.Class` 对象一开始是在 Eden 分配的，
>  随着 GC 生命周期逐渐晋升到老年代。
>  它不会直接放在老年代。

------

## 🧠 三、那为什么很多人说“Class 对象在老年代”？

这是因为：

1. 类加载一般只发生一次（加载后几乎常驻内存）；
2. `Class` 对象几乎永远存活整个 JVM 生命周期；
3. 很快就会在一次 Minor GC 后晋升到老年代；
4. 所以**从经验上看**，Class 对象“几乎都在老年代”。

但本质上：

> 并不是“直接放在老年代”，
>  而是“从 Eden 分配 → 快速晋升 → 常驻老年代”。

------

## 🧩 四、和类元信息的关系（关键）

`java.lang.Class` 对象只是 JVM 暴露给 Java 层的「镜像对象（mirror）」：

| 层面     | 名称                     | 存放位置  | 功能                                           |
| -------- | ------------------------ | --------- | ---------------------------------------------- |
| JVM 内部 | `InstanceKlass`（C++类） | Metaspace | 保存类定义信息                                 |
| Java 层  | `java.lang.Class` 实例   | Java 堆   | 反射、类型判断、`getName()`、`getMethods()` 等 |

两者通过指针互相引用：

```cpp
InstanceKlass::_java_mirror → oop(Class)
Class::_klass → InstanceKlass
```

这样 Java 层可以通过 `Class` 访问 JVM 层的类元数据。

------

## ⚙️ 五、静态变量放在哪？

当 `Class` 对象创建后，该类的静态字段也会随之分配在 **Java 堆中**：

- 静态基本类型和对象引用都属于 `Class` 实例的字段；
- 这些引用指向的对象同样在堆上；
- 所以所有静态变量本质上都和 Class 一起存在堆内。

------

## 🧱 六、举个例子说明：

```java
class A {
    static int x = 42;
    static Object o = new Object();
}
```

加载 `A` 时：

| 内容                       | 存放区域                    |
| -------------------------- | --------------------------- |
| 类的常量池、字段表、方法表 | Metaspace                   |
| `java.lang.Class<A>` 对象  | Heap（最初Eden，后晋升Old） |
| `x`（int静态字段）         | 存在 `Class<A>` 实例内      |
| `o`（Object静态字段）      | `Heap`（普通对象）          |

------

## 🧩 七、总结一句话

> **类加载后会创建两个层次的结构：**
>
> - `InstanceKlass`（类的元信息） → 存在 **Metaspace**；
> - `java.lang.Class` 实例（Java层可见） → 存在 **Java Heap**。
>
> `Class` 对象最初分配在 **Eden 区**，
>  之后随着 GC 晋升到 **老年代**，
>  所以它“最终几乎都在老年代”，但**并非一开始就放在老年代**。
