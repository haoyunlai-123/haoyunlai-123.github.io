空间分配担保是什么？

非常棒 👍，你问到的“**空间分配担保（Allocation Guarantee）**”是理解 **Minor GC → Full GC** 之间关系的关键机制之一，它正是 JVM 为了避免**年轻代晋升失败（promotion failure）**而引入的安全策略。

我们分几步来讲透它👇

------

## 🧩 一、背景问题：Minor GC 可能失败？

Minor GC（年轻代 GC）采用复制算法：

- 把 Eden + From(Survivor) 中存活对象复制到 To(Survivor)；
- 如果 To 放不下，就晋升到 Old（老年代）。

问题是：

> 如果老年代空间不够放这些晋升对象怎么办？
>  就会出现 **Promotion Failure（晋升失败）**。

而这种失败会导致 Minor GC 结束不了，整个堆状态混乱。
 因此 JVM 必须在 Minor GC 之前做个“安全检查”，
 确认：👉 “老年代有足够空间承接最坏情况的晋升对象”。
 这一步检查就是 **空间分配担保（Allocation Guarantee）**。

------

## ⚙️ 二、空间分配担保机制的核心思想

当 Minor GC 即将发生前，HotSpot 会检查：

> 如果本次 Minor GC 后，所有年轻代存活对象都要晋升到老年代，
>  老年代是否有足够的空间来“担保”容纳它们？

如果不够，JVM 会直接提前触发 **Full GC**，
 以腾出更多空间，而不是冒险执行 Minor GC。

------

## 🔍 三、HotSpot 检查逻辑（简化伪代码）

```cpp
if (old_gen.free() < young_gen.survivor_to_be_promoted_estimate) {
    // 老年代空间不够放要晋升的对象
    trigger_full_gc();  // 提前执行 Full GC
}
```

执行完 Full GC 后再尝试 Minor GC。
 这样确保 Minor GC 时不会出现 “Promotion Failure”。

------

## 🧱 四、两个相关概念

### 1️⃣ **Handle Promotion Failure**（处理晋升失败的策略）

- 在早期 JVM（JDK1.4 以前），Promotion Failure 可能直接导致 `OutOfMemoryError`。
- 现在 JVM 有参数 `-XX:+HandlePromotionFailure`（已废弃，默认启用类似逻辑），
   代表 JVM 在 Minor GC 前做分配担保检查，避免失败。

### 2️⃣ **担保计算方式**

JVM 会根据 **上次 Minor GC 后的存活数据** 估算出下次可能的晋升量：

```
expected_promotion = 上次Survivor区 + Eden区存活对象大小
```

然后判断老年代是否够大：

```
if (old_free_space > expected_promotion)
    安全，执行Minor GC；
else
    触发Full GC。
```

------

## 🧠 五、举个例子直观说明

假设：

```
Eden  = 100 MB
S0    = 20 MB (from)
S1    = 20 MB (to)
Old   = 200 MB (free 50 MB)
```

上次 Minor GC 后发现存活对象 60MB（Eden+S0）。
 现在准备再次 Minor GC：

1️⃣ JVM 检查老年代剩余空间 50MB；
 2️⃣ 预估晋升量 60MB；
 3️⃣ 50 < 60，不够担保；
 4️⃣ 🚨 触发一次 Full GC；
 5️⃣ 若 Full GC 后 Old 空间变为 150MB；
 6️⃣ 再执行 Minor GC 就安全了。

------

## ⚙️ 六、相关 JVM 参数（了解）

| 参数                          | 作用                                                 |
| ----------------------------- | ---------------------------------------------------- |
| `-XX:+HandlePromotionFailure` | 是否允许在担保失败时继续执行 Minor GC（JDK7 后废弃） |
| `-XX:PretenureSizeThreshold`  | 控制大对象直接进入老年代                             |
| `-XX:MaxTenuringThreshold`    | 控制对象晋升年龄阈值                                 |
| `-XX:SurvivorRatio`           | 控制 Eden 与 Survivor 的比例                         |
| `-XX:TargetSurvivorRatio`     | 动态年龄判定触发阈值                                 |

## ✅ 七、总结一句话

> **空间分配担保（Allocation Guarantee）** 是 JVM 在执行 Minor GC 前的一次“安全检查”：
>
> - 用于保证老年代空间足够容纳最坏情况下要晋升的对象；
> - 若空间不够，会提前触发 Full GC；
> - 其核心目标是防止 **Promotion Failure（晋升失败）**。

## 🧩 一、如果没有空间分配担保，会发生什么？

假设场景如下：

| 区域     | 空间  | 状态       |
| -------- | ----- | ---------- |
| Eden     | 100MB | 已满       |
| From(S0) | 20MB  | 有少量存活 |
| To(S1)   | 20MB  | 空         |
| Old      | 200MB | 剩余 30MB  |

这时 Eden 满了，JVM 准备执行 Minor GC。

### 🧱 过程（无担保机制）

1️⃣ JVM 执行 **Minor GC**：

- 标记 Eden + From 中的存活对象；
- 尝试复制到 To 区；
- To 放不下的对象需要晋升到 Old。

2️⃣ 结果：

- Old 区只剩 30MB，而要晋升的对象有 60MB；
- 晋升失败（Promotion Failure）。

3️⃣ JVM 被迫触发一次 **Full GC**：

- 清理老年代；
- 尝试再次放入；
- 若还是放不下 → 抛出 `OutOfMemoryError`。

所以这次过程是：

```
Minor GC（失败） + Full GC（补救）
```

不仅浪费了一次 GC，还两次 Stop-The-World，延迟更大。

------

## ⚙️ 二、空间分配担保的优化思路

为了避免上面的“双 GC”，
 JVM 在执行 Minor GC 前，会**提前评估风险**：

```cpp
expected_promotion = 上次MinorGC存活对象大小
if (old_gen.free() < expected_promotion)
    // 担保失败，提前执行Full GC
    trigger_full_gc();
else
    // 担保成功，执行Minor GC
    trigger_minor_gc();
```

这样，如果老年代空间明显不够，就不浪费一次 Minor GC，
 直接执行 Full GC 清理出空间。

于是整个流程变成：

```
Full GC（一次） ✅
```

比上面的：

```
Minor GC + Full GC（两次） ❌
```

性能更好、停顿更短。

------

## 🧠 三、为什么叫“担保 Guarantee”

这其实是 GC 的一种**保险策略**：

> JVM 在执行 Minor GC 前，会“担保”——
>  即保证：无论此次 Minor GC 存活对象有多少，都能被老年代安全接收。

担保的本质是：

- **不是去预测精确的晋升大小**；
- 而是 **基于上次 GC 的统计信息做安全冗余估算**。

------

## 🧱 四、补充细节（你理解得更深可以看到这一层）

在 HotSpot 的实现中，
 这一步在 **DefNewGeneration::handle_promotion_failure()**
 和 **GenCollectedHeap::do_collection()** 之间执行。

简化逻辑大概是：

```cpp
if (old_gen.free() < young_gen.promotion_estimate) {
    // Full GC 前的担保
    collect_full();
}
collect_young();
```

这个机制在 `Parallel Scavenge`、`Serial GC`、`CMS` 等都存在，
 只是不同 GC 实现细节不同。

------

## ✅ 五、一句话总结（你说的就是这句话）

> JVM 的“空间分配担保”机制是为了**避免在 Minor GC 之后再触发一次 Full GC**。
>  它通过在 Minor GC 前提前评估老年代的可用空间，
>  如果预计不够容纳晋升对象，则**提前触发一次 Full GC**，
>  从而只执行一次 GC，而不是 **Minor + Full** 连环停顿。
