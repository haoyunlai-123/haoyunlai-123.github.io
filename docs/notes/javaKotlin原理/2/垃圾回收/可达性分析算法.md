------

## 🧩 一、GC Roots 的基本定义

> GC Roots 是一组“起始引用”，从它们出发通过引用链（Reference Chain）可达的对象都被认为是**存活**的。
>  任何**无法从 GC Roots 追踪到的对象**都被视为**可回收**。

------

## 🧠 二、HotSpot JVM 中的 GC Roots 分类（最权威划分）

| 类别                                     | 说明                                                         | 举例                                                         |
| ---------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| ① **虚拟机栈（栈帧局部变量表）中的引用** | 每个线程在执行方法时，栈帧里保存了本地变量（局部变量表），包括方法参数和临时变量。 | 普通对象引用，如 `User user = new User();`                   |
| ② **本地方法栈（Native Stack）中的引用** | JNI 方法调用中引用到的 Java 对象。                           | 通过 `JNIEnv*` 访问的对象                                    |
| ③ **方法区中类的静态字段引用**           | 存放在 `Class` 对象的静态变量（`static` 字段）中的对象引用。 | `MyClass.staticUser`                                         |
| ④ **方法区中运行时常量池的引用**         | `StringTable`、被 `static final` 常量引用的对象。            | `"abc"` 等字符串常量                                         |
| ⑤ **同步锁持有的对象**                   | 所有被 `synchronized` 持有的对象（Monitor 对象）。           | 进入 `synchronized(obj)` 时的 `obj`                          |
| ⑥ **各个线程对象本身及其引用链**         | 每个活线程（`java.lang.Thread`）本身也是 GC Root，线程栈中对象、锁、局部变量都从它出发。 | 所有仍在运行或可恢复的线程对象                               |
| ⑦ **JVM 内部系统对象**                   | JVM 自己维护的重要对象。                                     | 类加载器（`ClassLoader`）、`SystemClassLoader`、`Finalizer` 队列、`JNI Global Ref` 等 |
| ⑧ **GC 子系统特殊引用**                  | 分代收集中作为起点的 Card Table/Remembered Set 中记录的跨代引用。 | Young GC 时 Old 区引用的对象                                 |
| ⑨ **JVMTI / 调试器引用**                 | 当启用 JDWP 或 JVMTI 时，调试器中引用到的对象。              | 调试器“监视”的变量                                           |
| ⑩ **特殊系统结构**                       | 如系统类加载器（Bootstrap、Platform、AppClassLoader）及其加载的类。 | `sun.misc.Launcher$AppClassLoader`                           |

## 🔍 四、额外细节补充

- **非活动线程**：线程终止后，它的栈帧会被销毁，不再算 Roots。
- **软/弱/虚引用**：这些引用不作为 GC Roots，而是在可达性分析后单独处理。
- **局部收集（Partial GC）**：例如 Young GC 时，Old 区对象不会作为回收目标，但 Old → Young 的引用仍算 Roots。

 我们来一步步拆开：

------

## 🧩 一、首先澄清：

### “对象被标记为垃圾” ≠ “对象头中某个位打了标志”

在 HotSpot JVM 中，一个对象**是否是垃圾**（即是否可达）
 并**不会记录在对象头（Mark Word）里**。

因为：

- GC 的“标记”操作（marking）只是 **在 GC 的内存元数据中** 记录对象是否被访问过；
- 对象头（Mark Word）只是 **运行时对象元信息区**，保存锁状态、哈希码、年龄等，与 GC 可达性分析并无直接关系。

------

## 🧠 二、真正的 GC 标记在哪里做？

HotSpot 中的 GC（不论 Serial、Parallel、CMS、G1、ZGC、Shenandoah）都遵循这个流程：

### 1️⃣ **可达性分析（Root Tracing）**

从 GC Roots 出发（栈、本地变量、静态字段等），遍历所有能到达的对象。

### 2️⃣ **在 GC 自己的元数据中做标记**

每个 GC 实现都有专门的数据结构存放“对象是否被访问过”的信息：

- **Mark-Sweep 系列**：使用 **Mark Bitmap（位图）**；
- **G1、Parallel GC**：在 Region / Heap Bitmap 中记录；
- **CMS、ZGC**：维护独立的 **mark bitmap 或 colored pointer**；
- **Shenandoah**：在对象引用指针上附加标记位。

这时候 JVM 不会去修改对象头，而是在 GC 自己维护的结构里标记对象是否可达。

------

## 🧱 三、对象头的 Mark Word 里是什么？

Mark Word 是对象头（Header）的第 1 部分（通常 64 位 HotSpot 上是 8 字节），结构大致如下（64bit 模式）：

| 位段    | 内容                                  | 说明                                       |
| ------- | ------------------------------------- | ------------------------------------------ |
| 25 bits | 对象 hashCode（或为空）               | 对象计算过 hash 时存放这里                 |
| 31 bits | 分代年龄（Age） + 锁标志位            | 年龄（4bit）+ 锁状态                       |
| 2 bits  | 锁状态标志（01,10,11等）              | 无锁、偏向锁、轻量级锁、重量级锁、GC标记用 |
| 6 bits  | GC 用标志或 epoch（不同 GC 用法不同） | 如 G1、ZGC 可能复用部分位                  |

### 🚫 注意：

- **Mark Word 并不表示对象是垃圾还是存活。**
- “年龄”（Age）字段用于 **分代晋升判断**，不是可达性标志；
- 某些 GC（如 G1）在标记阶段会**暂时复用 mark word** 存放自己的标志信息，但只是临时用途，GC 结束后会恢复，不作为“对象头长期状态”。

------

## ⚙️ 四、举个实际例子：G1 的标记方式

G1 的标记阶段：

1. 扫描 Root → 标记可达对象；
2. 标记信息写在 **Heap Region 的 Mark Bitmap**；
3. 对象头中的 mark word 可能被复用来暂存转发指针（forward pointer）或 TAMS（Top-at-mark-start）边界信息；
4. GC 完成后，Mark Bitmap 中未标记的对象即被视为垃圾，等待回收。

对象头里的 mark word 只是一种临时空间复用，不是“永久标志垃圾”的地方。

------

## 🧩 五、总结区分两种“标记”

| 含义               | 存储位置                                | 用途                             |
| ------------------ | --------------------------------------- | -------------------------------- |
| GC 可达性标记      | GC 自身的 Mark Bitmap / Region Metadata | 判断对象是否存活                 |
| 对象头的 mark word | 对象元信息（锁状态、hash、年龄等）      | 运行时对象管理，与 GC 可达性无关 |

------

## ✅ 六、总结一句话

> 对象是否“被标记为垃圾”，
>  是由 **GC 自己的标记位图（Mark Bitmap）或 Region 元数据** 决定的，
>  并不会修改对象头的 mark word。
>
> mark word 主要存放锁状态、hashCode、年龄，
>  在极少数 GC（如 G1、ZGC）中可能被**临时复用**，但不会长期记录“是否垃圾”。
