## 🌳 一、G1（Garbage-First）设计初衷

G1（JEP 156）是为了解决传统 GC 的两个痛点：

1. **CMS 内存碎片严重、回收不可预测**；
2. **吞吐量回收器（ParallelGC）暂停时间太长**。

它的目标是：

> 「在可控暂停时间内尽可能地最大化回收效率」
>  并保持堆的长期整洁（低碎片）。

------

## 🧩 二、核心设计理念

你说的完全正确，G1 的本质是：

> **把整个堆切分为大量大小相等的 Region（通常 1~32 MB），
>  每个 Region 都可能属于不同“年龄层”或角色（Eden / Survivor / Old / Humongous）**。

![img](https://i.imgur.com/Fh0JtOv.png)

- **Region 是最小管理单位**，不再是连续的大区块；
- **逻辑上仍有年轻代/老年代划分**；
- **物理上分布不连续**；
- **Region 的角色可以动态变化**（例如，一个原本的老年代 Region 在下次 GC 后可变为空闲再分配为 Eden）。

------

## ⚙️ 三、主要阶段流程

G1 是一个 **分阶段 + 并发 + 增量压缩** 的回收器，
 整个周期可以分为以下几个阶段：

### ① 初始标记（Initial Mark）【STW】

- 标记出 GC Roots 直接可达的对象。
- 这一步很短，常常和一次 Young GC 一起执行。

### ② 并发标记（Concurrent Marking）【与应用并发】

- 类似 CMS 的标记阶段；
- 遍历对象图，标记出**整个堆中活对象**；
- 同时计算每个 Region 的**存活率（live ratio）**；
- **不会移动对象**。

👉 这一步的结果非常关键：
 G1 会建立一个「Region → 存活百分比」的统计表。

------

### ③ 最终标记（Remark）【STW】

- 修正并发标记期间对象变化造成的误差；
- 典型技术是 SATB（Snapshot-At-The-Beginning）。

------

### ④ 筛选回收集（Choose CSet）【并发 + STW 准备阶段】

- 根据统计结果计算每个 Region 的「回收价值」。

回收价值 =

> (Region 占用空间 − 存活对象大小) / 回收成本（预测的拷贝耗时）

然后把这些 Region 放入一个**优先队列**（MaxHeap），
 选择一批「回收收益高」的 Region 构成 **Collection Set（CSet）**。

------

### ⑤ Evacuation（转移/复制阶段）【部分 STW】

- 复制 CSet 中存活对象到新的空闲 Region；
- 并行执行（多个 GC 线程）；
- 年轻代 Region 总是被清空（复制算法）；
- 老年代 Region 也会被增量整理（部分压缩）；
- 复制完成后原 Region 变为空闲 Region；
- 不会产生碎片，因为所有存活对象都被整齐搬迁。

------

### ⑥ 并发清理（Concurrent Cleanup）

- 标记阶段发现的空 Region 加入空闲列表；
- GC 线程可在后台清理 Remembered Sets 等结构。

------

## 🔍 四、Region 内存模型与算法细节

### 🧠 Remembered Set（RSet）

因为 Region 不连续，为了支持精确 GC 和跨 Region 引用，
 每个 Region 维护一个 RSet：

> 记录「哪些其他 Region 引用了我」。

这让 G1 不需要扫描整个堆即可并行回收。

RSet 是 G1 最核心的数据结构之一，也是 CPU 开销的大户。

------

### 🧮 SATB （Snapshot-At-The-Beginning）

G1 在并发标记时使用 SATB 算法：

- 假设标记开始时堆中所有对象都是存活的；
- 如果在并发标记中对象被引用，仍保留；
- 这样可以避免反复遍历。

------

### ⚖️ Evacuation Pause 选择算法

在每次 GC 前， G1 会预测一批 Region 的回收时间：

```text
回收代价 = 拷贝对象数量 × 每对象拷贝时间 + RSet 维护成本
```

然后根据用户设置的 `-XX:MaxGCPauseMillis`（默认 200 ms）
 从优先队列里选出可在此时间预算内回收的 Region。
 这就是“Garbage First”名字的由来。

------

## 🚀 五、算法特性总结（与你的描述对应）

| 特性                    | 说明                                                |
| ----------------------- | --------------------------------------------------- |
| **Region化堆**          | 不连续小块，角色可动态切换                          |
| **全堆管理**            | 年轻代、老年代统一算法                              |
| **成本模型**            | 预测回收时间，优先回收收益高的 Region               |
| **混合回收 (Mixed GC)** | 既可回收年轻代，也可挑选部分老年代 Region           |
| **复制算法**            | 各 Region 内部采用复制（拷贝）算法                  |
| **避免碎片**            | 对象总被搬迁到整齐的目标 Region                     |
| **并行 + 并发**         | 标记阶段与用户线程并发；Evacuation STW 但并行多线程 |
| **可控暂停**            | 通过预测模型调节 CSet 大小来满足暂停目标            |

## 🧩 七、伪代码形式流程（理解整体逻辑）

```text
loop forever {
    if (heap usage > threshold) {
        initial_mark();               // STW
        concurrent_mark();            // with mutators
        remark();                     // STW
        cset = choose_regions();      // cost-benefit
        evacuate(cset);               // STW + parallel
        cleanup();                    // concurrent
    }
}
```

## ✅ 九、一句话总结

> G1 通过将堆切分为大量 Region，并基于预测模型选择收益最高的 Region 进行部分回收，
>  结合并发标记、并行复制算法，在保证高吞吐的同时，实现了**可控暂停**与**低碎片堆管理**。
