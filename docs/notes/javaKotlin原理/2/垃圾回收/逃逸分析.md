## 🧩 一、你看到的现象

你的程序中：

```java
public static void test() {
    User user = new User();
}
class User {
    byte[] bytes = new byte[1024];
}
```

在 VisualVM 里堆中几乎没有 `User` 对象，
但存在少量 `byte[]` 数组对象。

这正是因为：

> 🔹 HotSpot 通过 **逃逸分析** 发现 `User` 不会逃逸出 `test()` 方法；
> 🔹 JIT 编译器（C2）执行了 **标量替换** 优化；
> 🔹 于是 `User` 实例本身根本没有被分配在堆上。

---

## ⚙️ 二、但为什么还看到 `byte[]` 呢？

这就要看对象的组成结构了。

* `User` 对象里有一个字段：

  ```java
  byte[] bytes = new byte[1024];
  ```

HotSpot 的逃逸分析是**逐对象判断**的：

| 对象       | 是否逃逸                           | 说明 |
| -------- | ------------------------------ | -- |
| `User`   | ❌ 不逃逸（仅局部使用）                   |    |
| `byte[]` | ✅ 逃逸（被 `User` 的构造方法调用时 new 出来） |    |

换句话说：

* `User` 本身的分配被**标量替换**掉；
* 但 `byte[]` 是在 `new User()` 时真正创建出来的对象；
* 这个 `byte[]` 没有被标量化，它仍然被分配在 **堆中**；
* 因此 VisualVM 只能看到这些 `byte[]`，却看不到 `User`。

---

## 🧠 三、所以你提到的“对象内部引用也会分配到栈上”要具体区分

正确表述应是：

> 🔸 如果字段本身是值类型（如 `int`、`double`），它们会随对象一起被标量化或放在栈上；
> 🔸 如果字段是引用类型（如 `byte[]`、`Object`），JIT 会单独分析该引用是否逃逸：
>
> * 若也不逃逸 → 可能同样栈上分配或标量化；
> * 若逃逸 → 仍然会在堆上分配。

你的例子里，`byte[]` 明显逃逸，因此它被分配到了堆。

---

## 🧩 四、实际机器级优化的样子

JIT 编译器（C2）优化后逻辑大致变为：

```asm
# User 对象消失
# byte[] 数组正常分配在堆上

call runtime_newarray(byte[1024])  ; 分配数组
store arrayref 到临时寄存器       ; 存入一个本地寄存器
# test() 方法返回后寄存器释放，对象没有逃逸引用，所以 byte[] 会被 GC
```

这就是为什么：

* 堆上有少量短命的 `byte[]`；
* 但完全没有 `User` 对象。

---

## 🧱 五、总结成一句话

> ✅ `User` 对象本身由于未逃逸，被 JIT 优化为标量替换或栈上分配，因此在堆中不可见；
> ❌ 但其内部的 `byte[]` 字段仍会单独分配在堆中（因为是独立对象且逃逸）；
> ⚙️ 所以在 VisualVM 中看到的只是那些 `byte[]` 数组，而不是 `User`。

## 🧩 一、逃逸分析的粒度：以方法为单位

逃逸分析的根本问题是：

> “在这个方法执行完之后，这个对象是否还可能被别的代码访问到？”

分析范围是**方法级别**的（包括被 JIT 内联进来的方法）。
所以编译器会从每个方法入口开始构建一个“引用传播图”。

---

## 🧠 二、你的例子分析（逐步传播）

```java
class User {
    byte[] bytes = new byte[1024];
}
public static void test() {
    User user = new User();
}
```

### 步骤 1️⃣ ：`test()` 方法开始分析

JIT 看到：

```java
User user = new User();
```

于是内联进入 `User.<init>` 构造方法。

---

### 步骤 2️⃣ ：进入构造函数

在构造函数中：

```java
public User() {
    this.bytes = new byte[1024];
}
```

* `new byte[1024]` 分配了一个数组对象；
* 然后执行 `putfield bytes` —— 把数组引用写入了 `this` 的字段。

此时逃逸分析会这么判断：

| 对象       | 当前方法内被谁持有              | 是否逃逸                 |
| -------- | ---------------------- | -------------------- |
| `User`   | `test()` 的局部变量         | ❌ 暂未逃逸               |
| `byte[]` | `User` 的字段（this.bytes） | ✅ **逃逸**（被存入外部对象的字段） |

因为这个 `byte[]` 不再只在当前方法（构造器）可见，
它被赋值到了一个“可能逃逸”的对象（`this`）中。

于是 JIT 得出结论：

> * `User` 可能栈上分配 / 标量替换；
> * 但 `byte[]` **必须在堆上分配**。

---

## ⚙️ 三、关键点：逃逸传染性（transitive escape）

逃逸是“传染”的：

```
byte[] -> 存入 this.bytes -> this 逃逸 -> byte[] 也视为逃逸
```

所以哪怕 `User` 本身后来没被返回，
因为 `byte[]` 的引用被写入了它的字段，
从构造函数角度看，它已经「间接逃逸」。

这种传播关系 HotSpot 在 IR 图中称为：

> **Object Field Escape**

---

## 🧩 四、对比一个“不逃逸”的版本

```java
public static void test() {
    byte[] data = new byte[1024]; // 本地变量
    User user = new User(data);   // 传入但不保存引用
}
class User {
    public User(byte[] arr) {
        int len = arr.length; // 只读取，不保存引用
    }
}
```

分析结论：

| 对象       | 是否逃逸             | 说明 |
| -------- | ---------------- | -- |
| `User`   | ❌ 不逃逸            |    |
| `byte[]` | ❌ 不逃逸（构造器仅访问不存储） |    |

→ JIT 有机会把两者都标量化甚至消除。

---

## 🧱 五、总结一句话

> ✅ 逃逸分析确实是**以方法为单位**的；
> ✅ 在构造器中，`byte[]` 被存入了 `this.bytes` 字段；
> ✅ 因为 `this` 代表当前正在构造的对象，它**可能逃逸到调用者方法中**；
> ❗所以从构造器的角度看，`byte[]` 已经“逃逸”到了 `this` 对象中，
> 导致它不能栈上分配或标量替换，必须放到堆中。
