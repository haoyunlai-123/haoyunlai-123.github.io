java中所有类加载器包括自定义类加载器是不是在一个类加载器链上？终点是bootstrap？自定义类加载器是不是也是由systemclassloader加载的？

非常好的问题，这个问题其实是很多人对 **类加载器层次结构** 的“半对半错”理解点——
 你提到的几点都很关键，我们来逐步澄清：

------

## 🧩 一、先看总体结构图（标准 JDK）

JVM 启动时类加载器的层级关系是典型的「双亲委派」树状结构：

```
BootstrapClassLoader
       │
       ▼
PlatformClassLoader   （JDK 9+ 以前叫 ExtensionClassLoader）
       │
       ▼
AppClassLoader        （SystemClassLoader）
       │
       ▼
自定义类加载器（CustomClassLoader）
```

------

## 🧠 二、逐层解释

### 1️⃣ **BootstrapClassLoader**

- 是 **JVM 内部（C++ 层）** 实现的类加载器；
- 负责加载核心类（`java.base` 模块下的类，例如 `java.lang.*`、`java.util.*` 等）；
- 不是 Java 对象，用 `null` 表示；
- 也叫「根加载器」。

------

### 2️⃣ **PlatformClassLoader**

- 位于 Java 层，是 `jdk.internal.loader.ClassLoaders$PlatformClassLoader`；
- 加载平台相关的模块（如 `java.sql`, `java.logging`）；
- 它的父加载器是 `BootstrapClassLoader`；
- 在 JDK 8 中它叫 `ExtClassLoader`，加载 `jre/lib/ext` 下的 jar 包；
- 从 JDK 9 起它加载模块化后的平台包。

------

### 3️⃣ **AppClassLoader（SystemClassLoader）**

- 是大部分应用的默认加载器；
- 负责加载 **classpath** 或 **modulepath** 下的类；
- 通过 `ClassLoader.getSystemClassLoader()` 获取；
- 父加载器是 `PlatformClassLoader`。

------

### 4️⃣ **自定义类加载器（CustomClassLoader）**

- 你自己继承 `ClassLoader` 或 `URLClassLoader` 实现；
- 默认的父加载器是 `AppClassLoader`；
- 但可以手动指定别的父加载器。

例如：

```java
ClassLoader myLoader = new MyClassLoader();
System.out.println(myLoader.getParent()); 
// 默认输出 AppClassLoader
```

------

## 🧱 三、关键问题回答：

### ❓1. “所有类加载器是否都在一个‘类加载器链’上？”

✅ **不完全是。**

更准确地说：

> 它们是 **一棵树（或森林）**，而不是严格的一条“链”。

解释：

- 每个类加载器都有一个“父加载器”；
- 但 JVM 允许你创建多个不相关的类加载器；
- 这些加载器可以形成 **多条分支（子树）**。

例如：

```java
ClassLoader a = new MyLoaderA();
ClassLoader b = new MyLoaderB();
```

这两个加载器 **都以 AppClassLoader 为父**，但它们互相独立。
 各自加载的类在 JVM 看来是「不同的类型空间」。

```java
a.loadClass("test.Hello") != b.loadClass("test.Hello")
```

✅ 所以——它们都**最终能追溯到 BootstrapClassLoader**，
 但结构上是“树”，不是“线”。

------

### ❓2. “终点是不是 BootstrapClassLoader？”

✅ 是的。

无论多少层继承：

```text
Custom → App → Platform → Bootstrap
```

**根都是 BootstrapClassLoader。**

Bootstrap 是整个委派链的“终极父类加载器”。

------

### ❓3. “自定义类加载器是不是由 SystemClassLoader 加载的？”

✅ 一般情况下是的（但要看你怎么创建）。

- 你写的类（定义自定义加载器的类）是由 SystemClassLoader 加载的；
- 当你 `new MyClassLoader()` 时，这个类加载器实例是由 Java 代码创建的对象；
- 它的父加载器默认是 **SystemClassLoader**。

但要注意两种特殊情况：

| 场景                                                 | 父加载器                              |
| ---------------------------------------------------- | ------------------------------------- |
| `new MyLoader()`                                     | 默认是 SystemClassLoader              |
| `new MyLoader(ClassLoader.getPlatformClassLoader())` | 手动指定为 Platform                   |
| `new MyLoader(null)`                                 | 父加载器为空（直接与 Bootstrap 并列） |

所以你可以创建「脱离系统类加载器链」的加载器。

------

> - 所有类加载器最终的祖先都是 **BootstrapClassLoader**；
> - 但它们形成的是一棵“树”，不是单一链表；
> - 自定义加载器默认的父加载器是 **SystemClassLoader**；
> - 也可以手动指定父加载器（甚至为 null）脱离主链。

a.loadClass("test.Hello") != b.loadClass("test.Hello")这样用自定义的类加载器加载类走双亲委派机制吗？如果在别的类加载器树上走不走双亲委派？ 

非常好 👍，你这问题问到了**类加载机制的核心底层逻辑**，比一般的“会不会委派”更深入一层。
 我们来一步步把它讲清楚：

------

## 🧩 一、首先明确：**双亲委派是逻辑行为，不是语言强制规则**

> “双亲委派机制”只是 `ClassLoader.loadClass()` 的 **默认实现策略**，
>  不是 JVM 强制执行的机制。

也就是说：

- 是否委派，完全取决于**你实现的类加载器逻辑**；
- JVM 并不会自动“帮你委派”；
- 所以「走不走双亲委派」，要看你是否调用了 `super.loadClass(name, resolve)`。

------

## 🧱 二、看源码：`ClassLoader.loadClass()`

JDK 的默认实现（在 `java.lang.ClassLoader` 中）：

```java
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    synchronized (getClassLoadingLock(name)) {
        // 1. 先检查缓存（已加载）
        Class<?> c = findLoadedClass(name);
        if (c == null) {
            try {
                // 2. 委派给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                } else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器没找到则自己加载
            }

            // 3. 自己加载
            if (c == null) {
                c = findClass(name);
            }
        }

        if (resolve) {
            resolveClass(c);
        }
        return c;
    }
}
```

👉 也就是说：

- **只要你没有改写这个逻辑**，就会先让“父加载器”去加载；
- “父加载器”没找到，才自己加载；
- 这就是「双亲委派机制」。

------

## 🧩 三、那么看你的例子：

```java
ClassLoader a = new MyLoaderA();
ClassLoader b = new MyLoaderB();

Class<?> c1 = a.loadClass("test.Hello");
Class<?> c2 = b.loadClass("test.Hello");

System.out.println(c1 == c2);
```

### 默认情况下（你没改 `loadClass`）

✅ 依然会走双亲委派机制。
 也就是说：

1. `a.loadClass("test.Hello")`
    → 先委派给 `AppClassLoader` → 再委派到 `Platform` → 再到 `Bootstrap`
2. 如果父链中都没有加载过 `test.Hello`，
    → 最后才调用 `a.findClass("test.Hello")` 自己加载。

同理 `b` 也一样。

------

### 那为什么最终 `c1 != c2` 呢？

因为：

- 虽然都委派到了同一个父类加载器链；
- 但最后父加载器找不到；
- 各自的 `findClass()` 加载出了 **不同的类定义副本**。

> **JVM 把“类 + 定义它的加载器”视为唯一标识。**
>  即：
>
> ```text
> (class name + defining loader)
> ```
>
> 相同的 class name，不同的 loader ≠ 同一个类。

------

## 🧠 四、“在不同类加载器树上”会不会走双亲委派？

这就分两种情况：

| 情况                       | 是否走双亲委派 | 说明                         |
| -------------------------- | -------------- | ---------------------------- |
| ① 有父加载器关系           | ✅ 会           | 只要父不为 null 且没重写逻辑 |
| ② 无父加载器关系（独立树） | ❌ 不会         | 根本没“父”，自然不会委派     |

------

### 🔸 示例 1：默认构造（有父加载器）

```java
class MyLoaderA extends ClassLoader {}
MyLoaderA a = new MyLoaderA(); // 默认 parent = SystemClassLoader
```

✅ 会走双亲委派链：

```
MyLoaderA → System → Platform → Bootstrap
```

------

### 🔸 示例 2：指定 null 父加载器

```java
class MyLoaderB extends ClassLoader {
    public MyLoaderB() {
        super(null); // 父加载器为空
    }
}
```

🔸 委派链断开：

```
MyLoaderB
↑ (no parent)
Bootstrap(独立存在)
```

此时 `MyLoaderB.loadClass()` 执行时：

```java
if (parent != null) {
    parent.loadClass(...)
} else {
    c = findBootstrapClassOrNull(name);
}
```

⚠️ 注意：即使父为 null，它仍然会查 **BootstrapClassLoader**（即查核心类）。

所以：

- 它不会委派到 SystemClassLoader；
- 但仍然可以加载核心类（如 `java.lang.String`）。

------

### 🔸 示例 3：完全自定义，重写 `loadClass()`

你可以彻底禁用委派：

```java
@Override
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    return findClass(name); // 不再委派父类
}
```

这样你创建的加载器将完全独立，不走任何父链。

------

## 🧱 五、可视化总结

```
              Bootstrap
                 │
                 ▼
           PlatformClassLoader
                 │
                 ▼
           AppClassLoader
              /     \
             ▼       ▼
      MyLoaderA   MyLoaderB
```

| 调用者                  | 是否委派 | 说明                  |
| ----------------------- | -------- | --------------------- |
| `MyLoaderA.loadClass()` | ✅        | 有父加载器（App）     |
| `MyLoaderB.loadClass()` | ✅        | 有父加载器（App）     |
| `new MyLoader(null)`    | ❌        | 没有父，只有Bootstrap |
| 重写 `loadClass()`      | ❌        | 逻辑上禁用了委派      |

## 🧠 六、结论总结

| 问题                                 | 答案                                  |
| ------------------------------------ | ------------------------------------- |
| 自定义类加载器加载类是否走双亲委派？ | 默认会（除非你重写 `loadClass`）      |
| 不同加载器树之间是否委派？           | 不会，它们相互独立                    |
| 父为 null 会怎样？                   | 仍查 Bootstrap 类加载器（加载核心类） |
| 委派机制是 JVM 强制的吗？            | 否，是 ClassLoader 的默认实现策略     |
| 两个加载器加载相同类名，类是否相同？ | 否，因为 loader 不同（定义空间隔离）  |

---

## 🧩 一、JDK 8 以前的类加载器结构

在 **JDK 8 及之前**（非模块化时代）：

```
BootstrapClassLoader (C++)
        │
        ▼
ExtClassLoader    (sun.misc.Launcher$ExtClassLoader extends URLClassLoader)
        │
        ▼
AppClassLoader    (sun.misc.Launcher$AppClassLoader extends URLClassLoader)
        │
        ▼
自定义类加载器
```

这时：

* 系统类加载器（`AppClassLoader`）继承自 `URLClassLoader`；
* 扩展类加载器（`ExtClassLoader`）也继承自 `URLClassLoader`；
* 所以 “系统类加载器” 实际上是 “URLClassLoader” 的实例。

你可以在 JDK 8 看到源码（`sun.misc.Launcher`）：

```java
public static class AppClassLoader extends URLClassLoader {
    ...
}
```

---

## 🧠 二、JDK 9 模块化后（Project Jigsaw）

从 **JDK 9 开始**，整个类加载体系彻底重构了。

新结构如下：

```
BootstrapClassLoader (native, C++)
        │
        ▼
PlatformClassLoader  (jdk.internal.loader.ClassLoaders$PlatformClassLoader)
        │
        ▼
AppClassLoader       (jdk.internal.loader.ClassLoaders$AppClassLoader)
```

它们都继承自：

```java
jdk.internal.loader.BuiltinClassLoader
```

而不是 `URLClassLoader`！

---

## ⚙️ 三、看看源码（JDK 24）

`jdk/internal/loader/ClassLoaders.java`：

```java
public static class AppClassLoader extends BuiltinClassLoader {
    ...
}
```

`BuiltinClassLoader` 定义在 `jdk.internal.loader` 包下：

```java
public abstract class BuiltinClassLoader extends SecureClassLoader {
    ...
}
```

可以看到：

* 它是 JDK 9 引入的新父类；
* 替代了 `URLClassLoader` 的系统级加载逻辑；
* 专门服务于模块化加载。

---

## 🧩 四、URLClassLoader 现在的位置

`java.net.URLClassLoader` 仍然存在，源码大致如下（JDK 24）：

```java
public class URLClassLoader extends SecureClassLoader implements Closeable {
    ...
}
```

但它现在主要用于：

* **手动加载外部 JAR、插件、动态模块**；
* 比如 Tomcat、IDEA、Spring Boot 等；
* 不再是系统类加载器的实现基础。

👉 也就是说：

> 它仍然是一个通用的类加载工具类，
> 但 **JDK 自己已经不再用它** 来加载模块或系统类。

## 📘 五、关键区别总结表

| 项目                  | JDK 8 及之前                                        | JDK 9 及之后（含 24）                                          |
| ------------------- | ------------------------------------------------ | -------------------------------------------------------- |
| 系统类加载器实现类           | `Launcher$AppClassLoader extends URLClassLoader` | `ClassLoaders$AppClassLoader extends BuiltinClassLoader` |
| 扩展类加载器              | `Launcher$ExtClassLoader`                        | 被 `PlatformClassLoader` 取代                               |
| 公共父类                | `URLClassLoader`                                 | `BuiltinClassLoader`                                     |
| 模块支持                | ❌ 无模块系统                                          | ✅ 模块化加载（ModuleLayer）                                     |
| URLClassLoader 是否废弃 | ❌ 否，但被弱化                                         | ❌ 否，仍可用                                                  |

## 🧱 六、`BuiltinClassLoader` 的核心作用

`BuiltinClassLoader` 是 JDK 9 引入的一个 **新抽象父类**，用于：

* 管理模块（Module）的加载；
* 维护模块→加载器的映射；
* 实现了受控的包可见性与导出规则；
* 内部使用 `BuiltinClassLoader.loadClassOrNull()` 代替 `findClass()`。

简化逻辑（伪代码）：

```java
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {
    // 检查模块系统
    Module m = findModule(name);
    if (m != null) return defineClass(m, name);
    // 委派父加载器
    return parent.loadClass(name, resolve);
}
```
