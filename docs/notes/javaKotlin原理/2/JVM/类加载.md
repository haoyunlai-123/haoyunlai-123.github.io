同一类加载器链上的不同类加载器加载的同名类会认为相同吗？比如一个类加载器和它的父类加载器加载同一个类会不会冲突？是不是双亲委派机制已经避免了这种情况发生？

双亲委派机制 -->责任链模式

非常棒的问题，这个触到了 **Java 类加载机制的核心逻辑**。
 我们分三步来讲清楚：

------

## 🧩 一、类的“相同性”判定标准

在 JVM 中，**类是否相同**，不仅由类的名字（全限定名）决定，还由 **加载它的类加载器（ClassLoader）实例** 一起决定。

> JVM 认为两个类相同，当且仅当：
>
> - 它们的 **全限定名** 一样；
> - 并且是由 **同一个 ClassLoader 实例** 加载的。

否则，即使类名完全一样，也被认为是**两个不同的类**。

```java
class Demo {
    public static void main(String[] args) throws Exception {
        ClassLoader loader1 = Demo.class.getClassLoader();
        ClassLoader loader2 = new URLClassLoader(((URLClassLoader)loader1).getURLs(), null);

        Class<?> c1 = loader1.loadClass("com.example.Test");
        Class<?> c2 = loader2.loadClass("com.example.Test");

        System.out.println(c1 == c2); // false
    }
}
```

👉 输出 `false`，说明是不同类，即使字节码内容完全一样。

------

## 🧠 二、同一“类加载器链”中的父子关系怎么影响？

假设存在一条**父子链**：

```
Bootstrap → Ext → App → Custom // 并不是父子关系，也不是加载关系，而是只是在一棵类加载器树上
```

当一个类加载器（例如 CustomClassLoader）要加载某个类时，双亲委派机制规定：

1. 先让 **父加载器** 去尝试加载；
2. 如果父加载器加载成功，就直接返回；
3. 只有父加载器找不到该类时，才由子加载器自己加载。

也就是说：

> **在同一条父子链中，父加载器和子加载器不会重复加载同一个类。**

因为：

- 父先尝试；
- 父成功 → 子不会再加载；
- 父失败 → 子才会加载；
- 这样避免了“同名类由不同加载器加载”的冲突。

------

## 🧱 三、那如果强行绕过双亲委派呢？

如果你**自己写一个类加载器**，并且在 `loadClass()` 中**不调用 `super.loadClass(name)`**（即不执行父类的加载逻辑），而是直接 `defineClass()`：

<div style="height:8px; border:none; border-radius:4px;
background:linear-gradient(to right, #ff00cc, #333399); box-shadow:0 0 10px #ff00cc;">
</div>

自己定义的类加载器没有在Bootstrap → Ext → App → Custom 类加载器树上

<div style="height:8px; border:none; border-radius:4px;
background:linear-gradient(to right, #ff00cc, #333399); box-shadow:0 0 10px #ff00cc;">
</div>

```java
@Override
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    // 不使用父加载器，自己加载
    byte[] bytes = loadClassBytes(name);
    return defineClass(name, bytes, 0, bytes.length); // 此方法根据class字节数组创建KlassInstance对象
}
```

那么：

- 你可能加载出**与父加载器已加载类重名但不相同**的类；
- 这两个类在 JVM 看来是不同的；
- 即使它们的字节码一模一样。

这种情况在：

- **Tomcat / OSGi / SPI 插件系统** 中非常常见；
- 每个模块或插件用独立的类加载器隔离；
- 避免互相污染（类名相同但逻辑不同）。

------

## ⚙️ 四、总结表

| 场景                         | 是否同类？             | 说明                   |
| ---------------------------- | ---------------------- | ---------------------- |
| 同一个类加载器加载的同名类   | ✅ 相同                 | 同一 ClassLoader       |
| 不同类加载器加载的同名类     | ❌ 不同                 | 无论父子关系           |
| 子加载器重复加载父已加载的类 | 🚫 不会发生             | 双亲委派避免           |
| 子加载器绕过父机制强行加载   | ✅ 能加载，但结果不同类 | 常用于隔离（插件系统） |

------

## 🔐 五、结论

> ✅ 在**正常双亲委派机制下**，
>  父加载器和子加载器不会重复加载同一个类；
>  因此不会发生“同名类冲突”。
>
> ❌ 但如果**子加载器故意绕过父加载器**（比如自定义类加载器不委派），
>  那么可以加载同名类，但它们在 JVM 里是两个独立的类。

| 是否同一个人 | 取决于“人名 + 户口本” | 类名 + ClassLoader              |

即使两个人都叫“张三”，只要他们不在同一个户口本里，就是两个不同的人。

## 🧩 一、从 JVM 的角度看

在这一段代码中：

```java
@Override
protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
    byte[] bytes = loadClassBytes(name);
    return defineClass(name, bytes, 0, bytes.length);
}
```

执行顺序是：

1. **你** 调用了 `loadClass(name, resolve)`；
2. JVM 调用你自己重写的逻辑；
3. 你调用了自定义方法 `loadClassBytes(name)`；
4. 返回的字节数组由你决定怎么读；
5. 然后 `defineClass(name, bytes, 0, bytes.length)`
    把这些原始字节变成一个真正的 `Class<?>` 对象。

整个过程**完全不会自动使用父加载器或 AppClassLoader**。

## 🧠 二、那 `loadClassBytes(name)` 通常怎么写？

最常见写法是——自己去读文件系统、网络、jar 包里的 `.class` 文件：

```java
private byte[] loadClassBytes(String name) throws IOException {
    String path = name.replace('.', '/').concat(".class");
    try (InputStream is = new FileInputStream(new File(classPath, path))) {
        return is.readAllBytes();
    }
}
```

或者（如果想从当前类路径加载）：

```java
private byte[] loadClassBytes(String name) throws IOException {
    String path = name.replace('.', '/').concat(".class");
    try (InputStream is = getClass().getClassLoader().getResourceAsStream(path)) {
        return is.readAllBytes();
    }
}
```

第二种情况里用的是：

```java
getClass().getClassLoader()
```

也就是 **加载你这个自定义类加载器自身类的加载器**。
 通常它是 **AppClassLoader**。
 因此，如果你这样写，就“间接地”用了 AppClassLoader 的资源查找功能。

------

## 🧩 三、总结：三种常见情况

| 场景                                                     | `loadClassBytes` 实现 | 实际谁来找 class 文件                 | 是否用 AppClassLoader |
| -------------------------------------------------------- | --------------------- | ------------------------------------- | --------------------- |
| 自己读文件（FileInputStream、网络流）                    | 手动读取磁盘或网络    | 你自己实现的 IO                       | ❌ 否                  |
| 使用 `getClass().getClassLoader().getResourceAsStream()` | 从当前类路径读取      | 当前类加载器（通常是 AppClassLoader） | ✅ 是                  |
| 使用父加载器的 `getResourceAsStream()`                   | 显式委派给父加载器    | 父加载器                              | ✅ 是（显式委托）      |

## ⚠️ 四、重点区别

**`defineClass()` 只是“定义类”**，不关心你怎么拿到字节。
 **`loadClassBytes()` 决定了“类字节的来源”**。

你可以：

- 从文件系统读；
- 从 jar 包读；
- 从网络下载；
- 从数据库里取；
- 从父加载器拿资源；
- 从完全不同的 classpath。

这些都会影响“谁在帮你加载”。

> 在你给出的代码中：
>
> ```java

> ```
>
> 并没有自动使用 AppClassLoader。
>
> - 如果你自己在方法中使用了 `getClass().getClassLoader()` → 是的，那是 AppClassLoader。
> - 如果你自己从文件或网络读取 → 就不是 AppClassLoader。
> - JVM 不会帮你替你调用 AppClassLoader 的逻辑。

## 🧩 一、先明确三层角色

在 JVM 里，“类的加载”有三步：

| 层次                     | Java 层 API                        | HotSpot 内部动作         |
| ------------------------ | ---------------------------------- | ------------------------ |
| ① 获取字节流             | `loadClassBytes(name)`             | 纯用户逻辑，JVM 不介入   |
| ② 定义类                 | `defineClass(name, bytes, 0, len)` | native 方法：解析 + 绑定 |
| ③ 链接（验证/准备/解析） | JVM 自动完成                       | 连接符号引用、分配静态区 |

👉 也就是说：

- 你读字节的阶段只是 I/O；
- 真正“让 JVM 认识这个类”的时刻在 **`defineClass()`**；
- 绑定和注册也是在那里完成的。

------

## 🧠 二、`defineClass()` 在 JVM 层做了什么？

我们看 HotSpot 的源码（`src/hotspot/share/classfile/classLoader.cpp`）：

```cpp
// Java_java_lang_ClassLoader_defineClass0 (简化)
JNIEXPORT jclass JNICALL Java_java_lang_ClassLoader_defineClass0(
  JNIEnv *env, jobject loader, jstring name, jbyteArray data, jint offset, jint length, jobject pd, jstring source)
{
    // 1. 把 byte[] 拷贝成 native 字节数组
    // 2. 调用 ClassFileParser 解析 class 文件结构
    Klass* k = ClassFileParser(...).parseClassFile(...);

    // 3. 调用 SystemDictionary::resolve_from_stream
    return SystemDictionary::resolve_from_stream(
              name, loader, k, protection_domain, CHECK_NULL);
}
```

重点在：

```cpp
SystemDictionary::resolve_from_stream(...)
```

里面执行了两个关键动作：

### 🧩 ① 建立“类与加载器”的绑定（ClassLoaderData）

```cpp
ClassLoaderData* loader_data = ClassLoaderData::class_loader_data(loader);
k->set_class_loader_data(loader_data);
loader_data->add_class(k);
```

📌 意思是：

- JVM 为每个 `ClassLoader` 创建一个 `ClassLoaderData`；

- 当你 `defineClass()` 时：

  - 把当前 `ClassLoader` 的 `ClassLoaderData` 取出来；
  - 将新定义的类 (`InstanceKlass*`) 挂到它下面；

- 这样形成了：

  ```
  ClassLoaderData ───► Klass ───► java.lang.Class 对象
  ```

### 🧩 ② 注册到系统字典（SystemDictionary）

```cpp
SystemDictionary::add_to_hierarchy(k, loader_data);
```

- JVM 内部维护一张全局哈希表（`SystemDictionary`），
   用于快速查找 `(loader, name)` → `Class`；

- 所以同名类但不同加载器可以共存：

  ```
  ("AppClassLoader", "com.Foo")  → Class@123
  ("CustomLoader", "com.Foo")    → Class@456
  ```

------

## ⚙️ 三、Java 层感知到的绑定

在 Java 层，最终效果是：

```java
Class<?> cls = defineClass(name, bytes, 0, bytes.length);
System.out.println(cls.getClassLoader() == this);  // true
```

也就是说：

- 该类对象的 `ClassLoader` 字段（C++层是 `oopDesc->_class_loader`）被设为了当前的 `this`；
- 这个绑定在定义时确定，不可更改。

------

## 🧩 四、所以总结一下：

| 阶段               | 动作                                                         | 是否绑定             |
| ------------------ | ------------------------------------------------------------ | -------------------- |
| `loadClassBytes()` | 只是你读字节的逻辑                                           | ❌ 否                 |
| `defineClass()`    | JVM 解析字节流、创建 Klass、挂到 ClassLoaderData、注册进 SystemDictionary | ✅ 是                 |
| `loadClass()`      | 上层封装（通常先查缓存→委派→defineClass）                    | 间接触发 defineClass |

## 🧠 五、进一步：ClassLoaderData 的作用

每个 `ClassLoader` 在 HotSpot 内都有对应的：

```cpp
class ClassLoaderData {
    ClassLoaderWeakHandle _class_loader;
    GrowableArray<Klass*> _classes; // 所有定义的类
}
```

它主要负责：

- 记录当前加载器加载的所有类；
- 管理它们的生命周期（JVM 卸载 ClassLoader 时才能卸载这些类）；
- 维护常量池、元空间（Metaspace）分配。

因此：

> 只有当 `defineClass()` 执行成功，一个类才真正与这个类加载器“绑定”。

> `loadClassBytes()`：只是“从某处拿到 class 字节”，JVM 并不介入；
>
> `defineClass()`：JVM native 层真正创建 `Klass`、注册到 `SystemDictionary`、
>  并把这个类与当前 `ClassLoader` 的 `ClassLoaderData` 绑定在一起；
>  也正是在这里，类才成为 JVM 认识的一个“受控实体”。

## 🧩 一、具体来说

你这段代码：


因此：

- `this` → 当前自定义类加载器对象；

  ```cpp
  loader_data = ClassLoaderData::class_loader_data(loader);
  k->set_class_loader_data(loader_data);
  ```

  这里的 `loader` 就是你传进来的 `this`；

- 最终绑定关系是：

  ```
  ClassLoaderData(this) ───► com.example.MyClass (Klass)
  ```

## ⚙️ 三、一个完整例子

```java
public class MyLoader extends ClassLoader {
    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        if (name.startsWith("test.")) { // 只拦截某个包
            byte[] bytes = loadClassBytes(name);
            Class<?> cls = defineClass(name, bytes, 0, bytes.length);
            System.out.println("Loaded by: " + cls.getClassLoader());
            return cls;
        }
        return super.loadClass(name, resolve);
    }

    private byte[] loadClassBytes(String name) throws IOException {
        String path = name.replace('.', '/') + ".class";
        try (InputStream in = getClass().getClassLoader().getResourceAsStream(path)) {
            return in.readAllBytes();
        }
    }
}

public class Test {
    public static void main(String[] args) throws Exception {
        MyLoader loader = new MyLoader();
        Class<?> cls = loader.loadClass("test.Hello");
        System.out.println(cls.getClassLoader() == loader); // true ✅
    }
}
```

输出：

```
Loaded by: MyLoader@6d06d69c
true
```

即使 `loadClassBytes()` 用的 `getResourceAsStream()` 来自 AppClassLoader，
 `defineClass()` 定义出来的类仍然绑定在 `MyLoader` 上。

## 📦 四、JVM 层对应关系（HotSpot 概念）

JVM 内部存在如下映射结构（来自 `SystemDictionary`）：

```
(key) (ClassLoader*, Symbol*)  →  (value) InstanceKlass*
```

也就是说，JVM 识别类的唯一键是：

```
(类加载器实例, 类全限定名)
```

所以：

- 即使两个不同的加载器定义了相同名字的类，它们也是两个不同的类；
- JVM 会分别注册两份 `(loader1, name)` 和 `(loader2, name)`；
- 两个类之间 `cls1 == cls2` 会返回 `false`。

| 关键点               | 说明                                         |
| -------------------- | -------------------------------------------- |
| `defineClass` 属于谁 | 当前 ClassLoader 实例 (`this`)               |
| 类与加载器的绑定时机 | 在 `defineClass()` 执行期间（native 层）     |
| 绑定对象             | 当前 ClassLoader 对应的 `ClassLoaderData`    |
| 绑定结果             | `cls.getClassLoader() == this` 为真          |
| 来源无关             | 即使字节来自别的加载器，绑定仍是调用者加载器 |

> 在你这段代码中：
>

>

>  并在系统字典中形成唯一映射 `(this, name)` → `Class`。

当类加载的linking阶段的解析阶段将符号引用转直接引用时如何判断需要的符号（类）是否加载？也是走类加载链查找吗？

解析阶段把符号引用变成直接引用时，**不会**自己再“遍历”一遍加载链，而是直接调用 `Jvm::resolve_class()`（HotSpot 源码里的 C++ 函数），这个函数底层就是一句：

```cpp
klassOop k = SystemDictionary::resolve_or_fail(class_name, class_loader, protection_domain, true, CHECK);
```

`SystemDictionary` 是 JVM 内部的全局哈希表，**key = <类名,  initiating loader>**，value = 对应的 `Klass*`。  
因此判断逻辑只有两步：

1. 看表：  
   在 `SystemDictionary` 里按 “类名 + 当前正在解析的类的 initiating loader” 查哈希桶。  
   - 命中 → 直接返回已存在的 `Klass*`，解析完成。  
   - 未命中 → 进入步骤 2。

2. 加载：  
   把“解析”这件事包装成一个 `ldc`/ldc_w/ldc2_w、anewarray、checkcast、putfield……等字节码引发的 `constant_pool_unresolved_klass` 的 `resolve` 调用，**最终落到 Java 层 `ClassLoader.loadClass`**。  
   这一步才真正走“双亲委派”链（或者用户自定义的加载逻辑）。加载成功后再把得到的 `Class` 镜像注册到 `SystemDictionary`，并填回常量池，解析结束。

所以：

- **解析阶段本身不“扫”加载链**，它只查全局哈希表。  
- 只有表里没有时，才触发一次标准加载流程；加载流程才是我们熟悉的“双亲委派”。  
- 因为哈希表以 “类名 + initiating loader” 为键，不同 loader 加载的同名类在表中是两条独立记录，这也保证了解析结果与“运行时包”一致，不会出现把 A 加载器解析到 B 加载器里同名类的情况。

一句话：  
解析 → 先查 `SystemDictionary` 缓存；缓存缺失 → 回调 `ClassLoader.loadClass` 走委派链；加载成功 → 写回缓存，继续链接。
