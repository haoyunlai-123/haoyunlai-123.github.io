若一个类加载器加载了某个接口，另一个无关系的类加载器加载了该接口的实现类，运行时会被jvm认为该类实现了该接口吗？

不会。  
在 JVM 眼里“是否实现了某接口”不是看 Java 源码里的 `implements` 字样，而是看**运行时方法表（itables）**里有没有那条接口记录。而方法表只在**同一个类加载器命名空间**里构造。

- 接口 `I` 由类加载器 `L1` 加载，得到 `L1` 命名空间里的 `I<L1>`。  
- 实现类 `C` 由类加载器 `L2` 加载，它在解析 `implements` 子句时，会去自己命名空间解析符号引用“`I`”。  
  - 如果 `L2` 不能委托给 `L1`（两者无父子关系，也不是同一个），它就会**重新加载**一份接口字节码，生成另一份 `I<L2>`。  
  - 于是 `C` 的常量池里记录的是 `I<L2>`，它的方法表也只包含 `I<L2>` 的方法，与 `L1` 空间里的 `I<L1>` 毫无关系。

结果：  
`obj instanceof I<L1>` → false  
`I<L1> x = (C) obj` → `ClassCastException`  
反射拿接口列表也只会看到 `I<L2>`。

只有**同一个接口类对象**（`I<L1> == I<L2>`）才被认为是同一类型，而这要求两个加载器位于**同一委托链路且至少有一个把请求委托给同一个已加载版本**。否则就是两份互不兼容的接口，JVM 不会认为实现关系成立。

此时是不是方法区里有两个该接口？

是的，方法区（Metaspace）里会出现**两份**接口的 `Class` 实例：

- 一份由 `L1` 加载，类型信息挂在 `L1` 的命名空间；
- 另一份由 `L2` 重新加载，类型信息挂在 `L2` 的命名空间。

虽然字节码内容完全相同，但 JVM 通过  
**“类全限定名 + 定义类加载器实例”**  
作为唯一标识，因此它们被视为两个互不兼容的类型。
