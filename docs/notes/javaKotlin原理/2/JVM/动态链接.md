运行时常量池里有方法地址吗？

没有“裸地址”，只有**带结构封装的间接引用**——HotSpot 里叫 `Method*`（或 `klass*` + `vtable/itable index`）。  
常量池里真正存的是下面两种“句柄”之一，而不是把 x86 指令地址直接写死：

1. 解析前  
   `CONSTANT_Methodref_info` → 索引指向 `CONSTANT_Class_info` + `CONSTANT_NameAndType_info`，全是符号，没有任何地址概念。

2. 解析后  
   字节码常量池对应的 `constantPoolOop` 会把原来的 `constantPoolCacheOop` 项填充为：  
   ```
   [f1] : klassOop  （方法所属的 Klass*）
   [f2] : vtable/itable 下标 或者 methodOop 直接指针
   [flags] : 类型标记（virtual/interface/static/…）
   ```
   运行时解释器或 JIT _stub 拿这两段数据就能唯一找到目标 `Method*` → 再取入口地址。  
   如果方法被 JIT 编译了，入口地址会再指向 `nmethod*` 里的 CodeBlob；未编译则指向解释器入口。  
   也就是说“地址”被包在 `Method->_i2i_entry` / `_from_compiled_entry` 字段里，常量池只保存“找得到 `Method*` 的钥匙”，而不保存随时可能变动的绝对地址。

因此：

- 常量池里没有“函数指针”或“绝对地址”。  
- 解析后保存的是 `Klass* + index` 或 `methodOop` 指针，属于对象句柄级别。  
- 真正的入口地址在 `Method` 结构内部，并且会随着解释/编译/去优化而动态改变，常量池本身不动。
