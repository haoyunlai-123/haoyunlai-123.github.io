------

## 🧩 一、JVM 对象在内存中的真实布局

一个普通 Java 对象在内存里一般分为三部分：

| 部分         | 内容                          | 大小                            |
| ------------ | ----------------------------- | ------------------------------- |
| **对象头**   | Mark Word + Klass Pointer     | 12 或 16 字节（取决于压缩指针） |
| **实例数据** | 成员变量（fields）            | 按声明顺序 & 对齐方式排列       |
| **对齐填充** | 保证整体对象大小是 8 字节倍数 | 可选                            |

> ⚠️ 默认开启 **压缩指针 (CompressedOops)** 时，klass pointer 占 4 字节，
>  所以普通对象头是 **12 字节 (8 + 4)**。

------

## 🧠 二、Teacher 类的字段分析

```java
class Teacher {
    private volatile String name; // 引用类型（4字节，压缩Oops）
    private volatile int age;     // 基本类型 int（4字节）
}
```

按照我们直觉，可能以为：

```
对象头 (12) + name(4) + age(4) = 20
```

所以偏移应是：

- name：12
- age：16

但 **HotSpot 实际布局可能重新排序** ——
 JVM 会根据字段类型、对齐规则、压缩指针、CPU 平台对齐优化重新安放这些 field。

------

## 🧩 三、为什么 age 的 offset 是 12？

关键点在于 **字段重排序 (field reordering)** + **按类型大小对齐**：

> HotSpot 会优先把 **小类型的字段（int、short、boolean）** 靠前放，
>  这样对象结构更紧凑，减少 padding。

所以内存布局实际上是这样的（假设 64-bit + CompressedOops）：

| 区域     | 字段          | 类型       | 偏移      | 大小 | 说明           |
| -------- | ------------- | ---------- | --------- | ---- | -------------- |
| 对象头   | Mark Word     | 8B         | 0~7       | 8    | markword       |
| 对象头   | Klass Pointer | 4B         | 8~11      | 4    | 指向类元数据   |
| 实例字段 | **age**       | int        | **12~15** | 4    | 小字段先排     |
| 实例字段 | **name**      | oop (引用) | **16~19** | 4    | 压缩指针4字节  |
| padding  | ——            | ——         | 20~23     | 4    | 对齐到 8B 倍数 |

所以最终：

- `age` 的偏移是 **12**
- `name` 的偏移是 **16**

> ✅ 这完全符合你的打印结果。

------

## 📊 四、验证：JVM 确实会重排字段

如果你反过来声明：

```java
class Teacher {
    private volatile int age;
    private volatile String name;
}
```

你会发现：

```java
System.out.println(ageOffset);
System.out.println(nameOffset);
```

输出仍是：

```
12
16
```

也就是说，**字段顺序与声明顺序无关**，
 HotSpot 会自动调整字段顺序来 **减少对齐填充**。

------

## 🧱 五、再看对齐原则（对象内部）

1. 对象头一般按 8 字节对齐；
2. 对象内字段按其大小对齐（int 4B 对齐，long/double 8B 对齐）；
3. 整个对象大小必须是 8 字节倍数。

HotSpot 的布局策略：

> 把同类型或相似大小的字段聚在一起，从而减少填充。

------

## 🧩 六、如果你关闭压缩指针会怎样？

JVM 启动参数：

```
-XX:-UseCompressedOops
```

那么引用（oop）占 8 字节，对象头变成 16 字节。
 这时布局会是：

| 区域          | 字段  | 偏移 |
| ------------- | ----- | ---- |
| markword      | 0~7   |      |
| klass pointer | 8~15  |      |
| age           | 16~19 |      |
| padding       | 20~23 |      |
| name (oop)    | 24~31 |      |

输出就会变成：

```
ageOffset = 16
nameOffset = 24
```

------

## ✅ 七、总结结论

| 项目               | 值                 | 说明          |
| ------------------ | ------------------ | ------------- |
| `mark word`        | 8B                 | 对象头        |
| `klass pointer`    | 4B                 | 压缩指针模式  |
| **`age` 偏移 12**  | 4B int，小字段优先 |               |
| **`name` 偏移 16** | 4B oop，压缩指针   |               |
| 总大小             | 24B                | 向 8 字节对齐 |

> **结论**：
>  `Unsafe.objectFieldOffset` 输出的偏移值由 **HotSpot 实际内存布局** 决定，
>  JVM 会重排字段顺序（优先小类型），使对象尽可能紧凑，
>  因此偏移值为 `12` 和 `16` 而不是声明顺序对应的 `16` 和 `20`。
