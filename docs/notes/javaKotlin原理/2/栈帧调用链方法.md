Reflection.getCallerClass();这是什么方法

`Reflection.getCallerClass()` 是 **JDK 内部使用** 的一个反射方法，定义在包 `sun.reflect.Reflection` 中，用于**获取当前正在执行的方法的调用者类（caller class）**。

------

### 🧩 基本作用

它能让你在运行时知道“是谁调用了当前方法”。

例如：

```java
import sun.reflect.Reflection;

public class Demo {
    public static void main(String[] args) {
        A.test();
    }
}

class A {
    static void test() {
        System.out.println(Reflection.getCallerClass());
    }
}
```

输出可能是：

```
class Demo
```

意思是：`Demo.main()` 调用了 `A.test()`，所以 `Reflection.getCallerClass()` 返回调用者 `Demo`。

------

### ⚠️ 注意事项（非常关键）

1. **这个方法是非标准、内部 API**

   - 它来自 `sun.reflect.Reflection`，不是 Java 公共 API；
   - 在 **Java 9+ 中已经被删除**（被标记为“unsafe internal API”），正常代码中**不应该再使用**；
   - JEP 260 把这些内部 API 封装了，普通代码无法直接访问。

2. **安全风险与优化干扰**

   - 它会绕过封装边界，对安全框架和 JIT 优化不友好；
   - 因此被官方弃用。

3. **现代替代方案**
    Java 9+ 提供了新的 API：

   ✅ 使用 `StackWalker`（推荐）：

   ```java
   StackWalker walker = StackWalker.getInstance(StackWalker.Option.RETAIN_CLASS_REFERENCE);
   Class<?> caller = walker.walk(frames ->
       frames.skip(1).findFirst().get().getDeclaringClass()
   );
   System.out.println(caller);
   ```

   ✅ 或者传统的 `Thread.currentThread().getStackTrace()`（性能较差）：

   ```java
   StackTraceElement[] stack = Thread.currentThread().getStackTrace();
   System.out.println(stack[2].getClassName());
   ```

------

### 🧠 小结

| 特性   | `Reflection.getCallerClass()`     | `StackWalker`                   |
| ------ | --------------------------------- | ------------------------------- |
| 来源   | `sun.reflect.Reflection`（内部）  | `java.lang.StackWalker`（标准） |
| 可用性 | JDK 8 及之前                      | JDK 9+                          |
| 安全性 | 不安全、已弃用                    | 安全、推荐                      |
| 性能   | 快（本地实现）                    | 略慢但可接受                    |
| 用途   | 框架内部（如早期的 Log4j、Guice） | 通用替代方案                    |

 我们来逐行解析：

------

### 🧩 1. `StackWalker.getInstance(StackWalker.Option.RETAIN_CLASS_REFERENCE)`

```java
StackWalker walker = StackWalker.getInstance(StackWalker.Option.RETAIN_CLASS_REFERENCE);
```

- `StackWalker` 是 Java 9 引入的新类，用来**高效访问调用栈信息（stack frames）**。

- 默认情况下，`StackWalker` 只暴露每一帧的文本信息（方法名、类名等）。

- 但这里我们传入了：

  ```java
  StackWalker.Option.RETAIN_CLASS_REFERENCE
  ```

  这个选项表示：

  > “不仅要返回类名字符串，还要能拿到真正的 `Class<?>` 对象。”

  所以之后我们才能调用 `frame.getDeclaringClass()` 得到真正的 `Class` 对象。

------

### 🧩 2. `walker.walk(...)`

```java
Class<?> caller = walker.walk(frames -> ... );
```

- `walk()` 方法接受一个 lambda（或者函数式接口 `Function<Stream<StackFrame>, T>`）。
- 它会**提供当前线程的调用栈帧流（`Stream<StackFrame>`）**给你去处理；
- 处理完后返回你指定的结果（这里是一个 `Class<?>`）。

------

### 🧩 3. `frames.skip(1).findFirst().get().getDeclaringClass()`

我们看这一段是如何在调用栈里定位“调用者”的：

1. `frames`：
    是一个流（`Stream<StackFrame>`），代表从当前方法开始的调用栈帧序列，**第 0 个是当前正在执行的帧**。
2. `skip(1)`：
    跳过当前方法自身（即 `walker.walk(...)` 所在的帧）。
3. `findFirst()`：
    取下一个栈帧，也就是**调用当前方法的类**。
4. `get()`：
    从 `Optional` 中取出这个帧。
5. `getDeclaringClass()`：
    拿到这一帧对应的 `Class<?>` 对象，也就是“调用者类”。

------

### 🧩 4. `System.out.println(caller)`

直接打印这个 `Class<?>` 对象。

输出类似：

```
class com.example.Main
```

或者打印类名：

```java
System.out.println(caller.getName());  // com.example.Main
```

------

### 🔍 举个完整例子

```java
public class A {
    public static void main(String[] args) {
        B.test();
    }
}

class B {
    static void test() {
        StackWalker walker = StackWalker.getInstance(StackWalker.Option.RETAIN_CLASS_REFERENCE);
        Class<?> caller = walker.walk(frames ->
            frames.skip(1).findFirst().get().getDeclaringClass()
        );
        System.out.println("调用者类是：" + caller.getName());
    }
}
```

输出：

```
调用者类是：A
```

解释：
 `A.main()` 调用了 `B.test()`，所以调用者是 `A`。

| 步骤                                  | 作用                             |
| ------------------------------------- | -------------------------------- |
| `getInstance(RETAIN_CLASS_REFERENCE)` | 创建能保留类引用的 `StackWalker` |
| `walk(frames -> …)`                   | 获取调用栈流并操作               |
| `skip(1)`                             | 跳过当前方法                     |
| `findFirst()`                         | 找到调用者的栈帧                 |
| `getDeclaringClass()`                 | 获取对应类对象                   |

!VM.isSystemDomainLoader()这是什么方法

非常棒的问题。
 `!VM.isSystemDomainLoader()` 这个看起来像是你在看 **JDK 源码** 或某些底层框架（如 `java.lang.reflect`、`ClassLoader`、`Unsafe` 相关代码）时看到的内部调用。我们来仔细拆解它。

------

## 🧩 一、出处与整体语义

这其实是调用了：

```java
jdk.internal.misc.VM.isSystemDomainLoader(ClassLoader loader)
```

它是 **JDK 内部类 `jdk.internal.misc.VM`** 提供的一个静态方法，用于判断：

> 某个类加载器（`ClassLoader`）是否属于“系统域加载器（System Domain Loader）”。

即，它检查某个类是否是由 **引导类加载器（Bootstrap ClassLoader）或平台类加载器（Platform ClassLoader）** 加载的。

------

## 🧠 二、作用解释

`isSystemDomainLoader()` 返回一个布尔值：

| 返回值  | 含义                                                         |
| ------- | ------------------------------------------------------------ |
| `true`  | 传入的 `ClassLoader` 属于系统域加载器（Bootstrap 或 Platform） |
| `false` | 属于应用类加载器或自定义类加载器                             |

所以：

```java
!VM.isSystemDomainLoader(loader)
```

表示“不是系统域加载器”，即“应用层或自定义类加载器”。

------

## 🔬 三、典型使用场景

这个判断经常出现在 JDK 内部的安全或访问控制逻辑里，比如：

- 反射 (`AccessibleObject`)
- 动态代理 (`Proxy`)
- 模块访问 (`Module`, `Layer`)
- `MethodHandles.Lookup` 等

例如在 `java.lang.reflect.AccessibleObject` 的源码里（JDK 17）：

```java
if (!VM.isSystemDomainLoader(caller.getClassLoader())) {
    // 说明调用方不是系统类
    checkAccess();
}
```

意思是：

> 如果调用者的类加载器不是系统域加载器，就要执行额外的访问检查（防止用户代码越权访问 JDK 内部类）。

------

## ⚙️ 四、系统域加载器是什么？

在现代 Java（JDK 9+ 模块化后）中，类加载器层次 roughly 是：

```
Bootstrap ClassLoader   -> 加载 java.base 等核心模块
Platform ClassLoader    -> 加载 java.sql, java.xml 等标准库模块
App ClassLoader         -> 加载用户类（classpath / modulepath）
Custom ClassLoader      -> 用户自定义
```

其中：

- **System Domain Loader** ≈ Bootstrap + Platform 两个；
- 用户自己加载的类（AppClassLoader、自定义ClassLoader）则属于“非系统域”。

------

## 🧩 五、源码（JDK 17 例）

在 `jdk.internal.misc.VM` 中可看到：

```java
public static boolean isSystemDomainLoader(ClassLoader loader) {
    if (loader == null) return true; // null 代表 BootstrapClassLoader
    return loader == ClassLoaders.platformClassLoader();
}
```

可见：

- `null` ⇒ BootstrapClassLoader ⇒ 系统域
- `platformClassLoader` ⇒ 系统域
- 其他 ⇒ 用户加载器 ⇒ 非系统域

------

## ✅ 六、总结

| 表达式                             | 意义                                                        |
| ---------------------------------- | ----------------------------------------------------------- |
| `VM.isSystemDomainLoader(loader)`  | 判断该 `loader` 是否为系统域加载器（Bootstrap 或 Platform） |
| `!VM.isSystemDomainLoader(loader)` | 判断该 `loader` 是否为 **用户类加载器或自定义加载器**       |
| 所属包                             | `jdk.internal.misc.VM`（内部 API）                          |
| 常见用途                           | 反射、模块访问控制、安全检查                                |
| 注意                               | 内部 API，普通应用代码不应直接调用                          |

------

## 🧩 一、`VM` 是什么类？

`VM` 是 JDK 自身使用的一个“虚拟机辅助类”，
 完整路径是：

```java
jdk.internal.misc.VM
```

它的作用是：

- 作为 Java 层访问底层 HotSpot VM 状态的一层“桥梁”；

- 提供一些虚拟机级别的内部工具方法；

------

## 🧱 二、为什么访问不了？

因为 **JDK 9 模块化（Project Jigsaw）** 之后，
 内部包（如 `jdk.internal.*`, `sun.misc.*`, `sun.reflect.*`）被封装在模块内部，默认不对外暴露。

`VM` 类位于 `jdk.internal.misc` 包，属于模块：

```
module java.base {
    exports java.lang;
    ...
    // 但没有 exports jdk.internal.misc;
}
```

因此：

- 在代码里写：

  ```java
  import jdk.internal.misc.VM;
  ```

  会报错：
   `package jdk.internal.misc is not visible`。

## 🧠 四、为什么 JDK 要封装它？

因为这些类：

- 不是公共 API；

- 不是稳定的；
- 被很多早期框架（如 Spring, Guice, Netty）误用，导致升级崩溃。

所以 JEP 260 把这些全部收回，只保留公开、稳定的标准 API。

## 🧩 2️⃣ 再看你的代码逻辑

你创建 `B` 的方式是：


这一步虽然使用了 `Unsafe.allocateInstance()` 绕过了构造函数，
但 **对象 `b` 的类依然是 `B`，加载器也仍然是 `AppClassLoader`**。
所以，按理说，`B.test()` 中的 `StackWalker` 应该能拿到调用者是 `A`，
`A` 的类加载器就是 `AppClassLoader`，于是


应该返回 `false`。

## 🧩 3️⃣ 关键在于：你取到的并不是 “A 的类加载器”！

这一行：

```java
ClassLoader classLoader = walker.walk(frames ->
    frames.skip(1).findFirst().get().getClass().getClassLoader());
```

其实有一个非常微妙的错误。

### ❌ 你拿到的不是栈帧中方法所属类的加载器，而是 **StackFrame 实现类**的加载器！

---

### 解释：

* `frames.skip(1).findFirst().get()` 返回的是一个 `StackFrame` 对象；
* 这个对象的运行时类通常是：

  ```
  jdk.internal.reflect.ReflectiveFrame
  ```

  或类似的内部实现类；
* 所以你执行：

  ```java
  frame.getClass().getClassLoader()
  ```

  得到的是这个内部类的 `ClassLoader`，而不是调用者类（`A`）的。

---

### 🧩 要拿到真正的调用者类，你应该写：

```java
ClassLoader classLoader = walker.walk(frames ->
    frames.skip(1).findFirst().get().getDeclaringClass().getClassLoader());
```

区别在：

* `getClass()` → 返回的是 `StackFrame` 的实现类；
* `getDeclaringClass()` → 返回的是该栈帧代表的方法所在的类（也就是 `A.class`）；

---

## 🧩 4️⃣ 所以为什么输出了 `hello`

因为：

```java
frame.getClass().getClassLoader()
```

拿到的其实是 JDK 内部类 `jdk.internal.reflect.StackFrameInfo` 或类似的实现类的加载器。
这些类是由 **Bootstrap 或 Platform ClassLoader** 加载的。
所以：

```java
VM.isSystemDomainLoader(classLoader) == true
```

于是输出了 `"hello"`。

---

## ✅ 正确写法（能验证 AppClassLoader）

改成如下即可：

```java
ClassLoader classLoader = walker.walk(frames ->
    frames.skip(1).findFirst().get().getDeclaringClass().getClassLoader());

if (VM.isSystemDomainLoader(classLoader)) {
    System.out.println("system");
} else {
    System.out.println("user");
}
```

输出结果就会是：

```
user
```

因为 `A` 是由 `AppClassLoader` 加载的。

| 表达式                               | 返回什么                         | 加载器            |
| --------------------------------- | ---------------------------- | -------------- |
| `frame.getClass()`                | 栈帧对象实现类（JDK内部）               | Bootstrap      |
| `frame.getDeclaringClass()`       | 该帧的方法所属类（如 A.class）          | AppClassLoader |
| `VM.isSystemDomainLoader(loader)` | Bootstrap / Platform 返回 true | 用户类返回 false    |

---

## ✅ TL;DR

你看到输出 `"hello"` 的原因是：

> 你错误地取了 `StackFrame` 实现类的加载器（由系统加载器加载），而不是调用者类的加载器。

修正成 `getDeclaringClass().getClassLoader()` 就一切符合预期。
