## 🧩 一、核心认知：Java 的一切加载，都是“路径拼接 + 名称映射”

无论是 `.java` → `.class` 的编译，
 还是 `.class` → 对象的运行时加载，
 Java 编译器和 JVM **做的事情本质上都一样：**

> 根据一些“逻辑名”（包名、类名、模块名）
>  去不同的“根路径”下拼接出文件路径，
>  然后读取文件并解析。

这就是你说的「各种路径拼接」思想。

------

## ⚙️ 二、编译时：`javac` 的拼接逻辑

| 元素                   | 意义                                          | 拼接逻辑               |
| ---------------------- | --------------------------------------------- | ---------------------- |
| `user.dir`             | 当前工作目录（执行 `javac` 时的相对路径起点） | 相对路径都从这里起     |
| `-d`                   | 输出目录                                      | 输出 `.class` 的根     |
| `--module-source-path` | 模块源码根                                    | 拼接模块目录 + 包路径  |
| `package` 声明         | 定义包路径                                    | 逻辑名映射成物理路径   |
| 全类名（FQN）          | 包 + 类                                       | 决定输出文件的完整路径 |

💡 例如：

```bash
javac -d out --module-source-path src -m com.example.app
```

编译器内部等价执行：

```
src/com.example.app/com/example/app/Main.java
→ out/com.example.app/com/example/app/Main.class
```

这一步其实就是字符串拼接。

------

## 🔥 三、运行时：`java` 的拼接逻辑

| 元素          | 意义           | 拼接规则                                              |
| ------------- | -------------- | ----------------------------------------------------- |
| `user.dir`    | 当前工作目录   | 相对路径起点                                          |
| `classpath`   | 类搜索根目录   | 旧机制：拼接包路径找 `.class`                         |
| `module-path` | 模块搜索根目录 | 新机制：拼接模块名 + 包路径找 `.class`                |
| 全类名        | 决定相对路径   | `com.example.app.Main` → `com/example/app/Main.class` |

例如：

```bash
java --module-path out -m com.example.app/com.example.app.Main
```

JVM 实际查找：

```
out/com.example.app/com/example/app/Main.class
```

## 🧠 四、所以说，所有这些“看起来复杂的参数”其实都归结于：

> **一个逻辑名（类名/模块名） → 一个物理路径（.java 或 .class 文件）**

一图就能概括：

```
逻辑名                拼接根目录           拼接规则
────────────────────────────────────────────
com.example.app.Main  ← classpath/modulepath + com/example/app/Main.class
com.example.lib.Util  ← module-source-path  + com.example.lib/com/example/lib/Util.java
```

## 🧱 五、IDEA 的“麻烦感”其实是因为它自动在帮你拼

IntelliJ、Eclipse、VSCode 在背后做的事情就是：

- 根据 `.iml` 或 `pom.xml` 推导：
  - 模块名 → 模块路径
  - 包名 → 源码路径
  - 输出目录 → `out/` 或 `target/classes`
- 代替你执行那些 `javac` / `java` 命令

所以当 `.iml`、`module-info.java`、路径结构不一致时，它就“懵逼了”，因为拼接逻辑断了链。

## 📘 七、额外拓展：你可以进一步理解成三层映射模型

| 层级       | 映射关系                         | 示例                                                     |
| ---------- | -------------------------------- | -------------------------------------------------------- |
| **逻辑层** | 模块名、包名、类名               | `com.example.app.Main`                                   |

- 编译器拼 `.java → .class`

- IDE 拼 “文件路径 → 工程模型”

## 💡 为什么这才是“高手级”的理解方式

因为 Java 体系的很多规则（特别是 modulepath / classpath / package）看似不同，
 但背后**全是一个模板**：

```
逻辑名称  →  路径拼接  →  文件读取
```

| 场景         | “拼接思维”怎么用                          | 举例                                        |
| ------------ | ----------------------------------------- | ------------------------------------------- |
| `javac` 编译 | 模块名 + 包名 → `.java` 文件              | 找到 `src/moduleName/com/example/Main.java` |
| `java` 运行  | 类名 + classpath → `.class` 文件          | 找到 `out/com/example/Main.class`           |
| 反射加载     | `Class.forName("xxx")` → classpath 拼接   | JVM 找 `.class` 路径                        |
| ClassLoader  | `getResource("/...")` → 根路径拼接        | 拼出真实文件路径                            |
| 模块系统     | 模块名 + modulepath → module-info + class | `out/moduleName/...`                        |

一旦你脑里能还原这条“路径拼接链”，
 就不需要记任何命令、参数、规则 ——
 全都能**反推出来**。
