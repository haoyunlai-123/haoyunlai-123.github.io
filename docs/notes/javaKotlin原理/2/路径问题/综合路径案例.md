结合user.dir,classpath举个稍复杂的例子

非常棒 🎯——你现在问的这个点已经进入 **Java 模块系统 + 类加载机制 + 工作目录 (user.dir) + classpath/module-path** 的综合层面了。
 这也是很多人搞混的地方：

> 当我们有模块目录、包路径、`user.dir`、`classpath`、`module-path` 时，
>  JVM 到底是怎么找到这些类文件的？

下面我用一个**完全可运行的例子**（在 `D:\test` 下）一步步带你理解整个路径解析逻辑。

------

## 🧩 一、前置知识对齐

| 名称              | 含义                                                         |
| ----------------- | ------------------------------------------------------------ |
| **`user.dir`**    | 当前 JVM 进程的工作目录（`System.getProperty("user.dir")`）。它通常就是你运行 `java` 命令的目录。 |
| **`classpath`**   | 类路径（用于非模块项目），告诉 JVM 到哪里找 `.class` 或 `.jar` 文件。 |
| **`module-path`** | 模块路径（用于 Java 9+ 模块项目），告诉 JVM 到哪里找模块（`module-info.class` 所在目录）。 |
| **包路径**        | 源码或编译后的 `.class` 目录结构，例如 `com/example/app/Main.class`。 |

------

## 🧱 二、我们在 D:\test 下创建如下结构

```
D:\test
│
├─ src
│   ├─ com.example.lib
│   │   ├─ module-info.java
│   │   └─ com\example\lib\Utils.java
│   │
│   └─ com.example.app
│       ├─ module-info.java
│       └─ com\example\app\Main.java
│
└─ run
```

说明：

- 我们在 `src` 下有两个模块：`com.example.lib` 和 `com.example.app`
- 稍后会把编译结果放进 `out` ，再从 `run` 目录运行，让 `user.dir` ≠ 项目根。

------

## ⚙️ 三、代码内容

### 1️⃣ `com.example.lib` 模块

```
D:\test\src\com.example.lib\module-info.java
module com.example.lib {
    exports com.example.lib;
}
D:\test\src\com.example.lib\com\example\lib\Utils.java
package com.example.lib;

public class Utils {
    public static String getPathInfo() {
        return "user.dir = " + System.getProperty("user.dir");
    }
}
```

------

### 2️⃣ `com.example.app` 模块

```
D:\test\src\com.example.app\module-info.java
module com.example.app {
    requires com.example.lib;
}
D:\test\src\com.example.app\com\example\app\Main.java
package com.example.app;
import com.example.lib.Utils;

public class Main {
    public static void main(String[] args) {
        System.out.println("=== 模块路径示例 ===");
        System.out.println(Utils.getPathInfo());
        System.out.println("ClassLoader: " + Main.class.getClassLoader());
        System.out.println("Main.class资源路径: " +
            Main.class.getResource("/com/example/app/Main.class"));
    }
}
```

## 🧰 四、编译模块

```bash
cd D:\test
javac -d out\com.example.lib ^
      src\com.example.lib\module-info.java ^
      src\com.example.lib\com\example\lib\Utils.java

javac --module-path out -d out\com.example.app ^
      src\com.example.app\module-info.java ^
      src\com.example.app\com\example\app\Main.java
```

编译完成后：

```
D:\test\out
├─ com.example.lib
│   ├─ module-info.class
│   └─ com\example\lib\Utils.class
└─ com.example.app
    ├─ module-info.class
    └─ com\example\app\Main.class
```

## 🚀 五、从不同目录运行测试

### ✅ 场景 1：在项目根运行

```

输出示例：

```

### ✅ 场景 2：切换工作目录（user.dir 不同）

```bash
cd D:\test\run

输出：

```
=== 模块路径示例 ===

📍 注意观察：

- 但类加载路径仍然指向 `out/com.example.app`（因为你通过 `--module-path` 明确指定了）。

## 💡 六、换成 classpath（非模块化方式对比）

假设我们不用模块系统，只是普通包结构：

```
D:\test\simple
│
├─ com\example\lib\Utils.java
└─ com\example\app\Main.java
javac -d classes simple\com\example\lib\Utils.java simple\com\example\app\Main.java
```

运行：

```bash
cd D:\test
java -cp classes com.example.app.Main
```

此时：

- `user.dir = D:\test`
- 类路径（classpath）= `classes`
- JVM 会从 classpath 的根去找包路径：`classes/com/example/app/Main.class`

------

## 🔍 七、模块路径 vs 类路径 查找逻辑对比

| 机制                  | 路径参数            | JVM 如何找类                                                 |
| --------------------- | ------------------- | ------------------------------------------------------------ |
| **Classpath（旧）**   | `-cp classes`       | 直接在路径下拼接包路径，如 `classes/com/example/...`         |
| **Module-path（新）** | `--module-path out` | 在每个模块文件夹下找 `module-info.class`；再根据模块名查找包结构 |

> 所以 module-path 是 “模块级目录”，classpath 是 “类级目录”。

------

## ✅ 八、总结关键点

| 名称          | 来源                    | 示例值                                | 作用                           |
| ------------- | ----------------------- | ------------------------------------- | ------------------------------ |
| `user.dir`    | JVM 工作目录            | `D:\test` 或 `D:\test\run`            | 程序中用来解析相对路径         |
| `classpath`   | `-cp` 参数或环境变量    | `classes`                             | 非模块模式下查找 `.class` 文件 |
| `module-path` | `--module-path` 参数    | `out`                                 | 模块模式下查找模块及包         |
| 模块名        | `module-info.java` 声明 | `com.example.app`                     | 逻辑标识                       |
| 包路径        | `package` 声明          | `com.example.app` → `com/example/app` | 文件系统映射                   |

------

## 📘 九、你可以这样一句话总结（面试 or 记忆用）

> 在模块化项目中，
>  `user.dir` 控制程序运行时的工作目录（影响 File 等相对路径）；
>  `--module-path` 控制 JVM 在哪些目录下查找模块；
>  模块名来自 `module-info.java`；
>  包名映射到模块文件夹内的物理路径。
>  模块路径相当于 classpath 的升级版，
>  但以模块为最小单位。

## 🧩 一、为什么感觉比 HashMap 源码还难？

因为两者的“复杂”完全不一样：

| 比较项     | `HashMap` 源码          | 模块系统（JPMS）                        |
| ---------- | ----------------------- | --------------------------------------- |
| 核心逻辑   | 数据结构 + 算法         | 目录结构 + 语言规则 + 编译工具交互      |
| 学习门槛   | 理论复杂但路径清晰      | 规则简单但路径&工具极多                 |
| 主要痛点   | resize / hash / treeify | 路径、类加载器、`module-info`、IDE 配置 |
| 一句话总结 | “算法复杂”              | “生态复杂”                              |

> 🔥 模块系统的问题不是“难懂”，而是“容易出错”：
>  目录对不上、包名不匹配、路径配置错、IDE 不识别。

------

## 🧠 二、把“模块系统”的复杂性，一句话拆开讲

> 模块系统 = 包管理（module-info） + 类加载约束（module-path） + 编译工具链（javac/java）

就三件事：

1. **逻辑上** → 模块名、包名、导出/依赖。
2. **物理上** → 文件夹结构、路径。
3. **工具上** → 命令行参数（或 IDE 自动推断）。

### 举个类比：

如果你熟悉 `HashMap`：

> 模块系统 ≈ `HashMap` 的 `hash() + put() + resize()` 三者必须配合好，
>  少一步都能崩。

同样：

> 模块系统 = `module-info.java` + 路径对齐 + 命令行/IDE 配置三件套，
>  有一个不对应，就会爆。

## 🧱 三、最简单能跑的模块例子（不坑版）

你可以直接新建一个目录结构如下👇

```
D:\test
├─ src
│  ├─ com.example.lib
│  │  ├─ module-info.java
│  │  └─ com\example\lib\Utils.java
│  └─ com.example.app
│     ├─ module-info.java
│     └─ com\example\app\Main.java
└─ out
```

📄 `src/com.example.lib/module-info.java`

```java
module com.example.lib {
    exports com.example.lib;
}
```

📄 `src/com.example.lib/com/example/lib/Utils.java`

```java
package com.example.lib;
public class Utils {
    public static String hello() {
        return "Hello from lib!";
    }
}
```

📄 `src/com.example.app/module-info.java`

```java
module com.example.app {
    requires com.example.lib;
}
```

📄 `src/com.example.app/com/example/app/Main.java`

```java
package com.example.app;
import com.example.lib.Utils;

public class Main {
    public static void main(String[] args) {
        System.out.println(Utils.hello());
    }
}
```

### 编译与运行（超干净版）

```bash
cd D:\test
javac -d out --module-source-path src -m com.example.app
java --module-path out -m com.example.app/com.example.app.Main
```

🎉 输出：

```
Hello from lib!
```

## 🧭 四、为什么路径要这么啰嗦？

其实 Java 是为了兼容 20 年的老代码：

- 旧的 `.class` 查找逻辑依赖 classpath；
- 新的模块引入了 modulepath；
- 模块与包要严格映射；
- IDE（尤其 IntelliJ）在中间夹着要帮你“猜路径”；
- 所以各种规则混在一起后就显得“神经质”。

📘 官方的思路是：

> Java 模块系统是给 **JDK 本身** 和 **大型库（JFX, JDBC, HTTPClient）** 用的，
>  不推荐普通项目手写。

## 🧩 五、程序员的真实使用策略（也是大厂常态）

| 场景          | 推荐做法                                            |
| ------------- | --------------------------------------------------- |
| 普通项目      | 🚫 不用 JPMS 模块化，照常用 package + Maven/Gradle。 |
| 想了解原理    | ✅ 用最小命令行例子练熟 module-info 与 modulepath。  |
| 大型库或 SDK  | ✅ 才考虑真正的模块化（尤其要封装 API）。            |
| IDEA 报错太多 | ✅ 在每个模块文件夹下保留 `.iml`，不手动删。         |

大厂（包括阿里、字节、Google）在内部项目里几乎都 **禁用 module system**，
 因为它和构建工具/框架（Spring、Netty、MyBatis）兼容性太差。
